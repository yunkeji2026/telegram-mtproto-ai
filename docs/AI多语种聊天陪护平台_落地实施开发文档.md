## 1. 本轮思想对齐

### 1.1 我们先统一一个判断

项目不建议继续只按“电商客服翻译平台”推进。电商可以保留为后续行业插件，但第一增长方向建议调整为：

**AI 多语种聊天陪护与交友助手平台。**

核心价值不是“把外语翻译成中文”，而是：

> 帮运营人员跨平台、跨语言、跨账号维护聊天关系，让 AI 帮他看懂、判断、接话、生成自然回复，并在合适场景生成语音。

### 1.2 为什么这样定

我们现有项目已经具备较多陪护类基础：

- 多平台接入：Telegram、Messenger、WhatsApp、LINE 已有不同程度的接入和 RPA 能力。
- 统一收件箱雏形：已经有 `/unified-inbox` 页面和聚合路由。
- 陪护关系阶段：已有初识、升温、暧昧陪伴、稳定陪伴的关系阶段逻辑。
- 情景记忆：系统可以记录用户偏好、事实、上下文片段。
- 亲密度分数：已有基于聊天事件计算关系热度的模块。
- 语音生成：已有 TTS 能力，可生成语音预览或发送语音。
- 人设系统：已有 persona/profile 配置，支持不同账号绑定不同人设。
- 风控基础：已有敏感内容、冷却、节奏、人工审批等基础逻辑。

因此，下一步不是重做底层，而是把这些能力收拢成一个好用、可演示、可试点的产品。

## 2. 已确认的产品决策

以下 8 条作为本项目第一阶段的正式产品决策，后续开发、演示、销售包装都按这个方向推进。

| 决策项 | 已确认方向 | 对开发的影响 |
| --- | --- | --- |
| 第一批客户 | 海外社交运营、交友陪聊、情感陪护、主播私信维护团队 | 页面、话术、人设、指标都围绕“关系维护”和“持续聊天”设计 |
| 电商定位 | 电商作为后续可扩展插件，不作为第一版主线 | 第一版弱化订单、物流、售后，把资源放到聊天体验 |
| 第一批平台 | Messenger + WhatsApp + Telegram + LINE 四个平台一起做 | 统一收件箱和消息模型必须一次支持四个平台 |
| 回复模式 | 必须有开关：全部 AI 自动回复、人工审核回复、多答案选择回复 | 自动化等级要做成产品开关，不只写在配置文件里 |
| 对外包装 | 避免“真人恋爱承诺”，定位为 AI 辅助运营聊天/陪护助手 | 合规边界更稳，也方便企业客户接受 |
| 语音能力 | 分为通用授权声音、克隆声音、语音变声/变语种 | TTS 页面和配置要预留三类能力入口 |
| 商业模式 | 先做私有化/本地试点交付，稳定后再包装 SaaS | 部署、配置、数据安全、授权管理要从第一版考虑 |
| MVP 成功标准 | 一个运营稳定管理 20-50 个活跃对话，AI 回复建议多数可直接采用 | 验收不只看功能是否存在，还看运营效率和回复质量 |

### 2.1 当前不再阻塞开发的默认共识

如果后续没有新的战略变化，以下问题不需要再反复讨论，开发时按默认方案推进：

- 第一版先做“辅助运营人员更会聊天”，不是做完全无人值守机器人。
- 第一版先做私有化试点可用，不追求一开始就做成复杂 SaaS。
- 第一版保留合规边界，不做现实恋爱承诺、不鼓励欺骗身份、不做无授权声音克隆。
- 第一版可以使用 AI 自动回复，但必须有人工审核、低风险自动、全自动三档开关。
- 第一版四个平台都进入统一工作台，但深度能力允许分层：能看消息、能分析、能回复是第一层；语音发送、自动续聊、多账号调度是第二层。

## 3. 产品最终形态

### 3.1 产品一句话

**一个面向多语种聊天交友、情感陪护和私域关系维护的 AI 工作台。**

### 3.2 使用者是谁

- 海外交友类产品运营人员
- 情感陪护/陪聊业务团队
- 主播、达人、网红的私信运营团队
- 海外社交账号矩阵运营团队
- 私域社群、会员陪伴、粉丝关系维护团队
- 后续可扩展到跨境电商客服团队

### 3.3 被服务对象是谁

- 来自不同国家和语言地区的聊天用户
- 粉丝、会员、潜在客户、社交关系对象
- 需要被长期维护和持续互动的人

### 3.4 第一版核心体验

运营人员打开一个工作台，可以看到所有平台的新消息。系统自动做几件事：

1. 把对方原文翻译成中文。
2. 判断对方当前情绪和聊天意图。
3. 显示这个人的关系阶段和历史偏好。
4. 给出 3 个可发送的回复建议。
5. 支持一键改写成温柔、幽默、简短、深入聊天等风格。
6. 支持把回复生成语音。
7. 低风险消息可自动发，高风险消息交给人工确认。

## 4. 现有项目能力盘点

### 4.1 已经具备的能力

| 能力 | 当前状态 | 下一步如何产品化 |
| --- | --- | --- |
| Telegram AI 回复 | 已有较完整逻辑 | 接入统一聊天工作台 |
| Messenger RPA | 已有多账号、审批、陪护模式、语音、风控 | 作为第一批重点渠道 |
| WhatsApp RPA | 已有 pending、语音、主动续聊、状态库 | 作为第一批重点渠道 |
| LINE RPA | 已有账号服务和发送队列 | 作为第二批增强渠道 |
| 统一收件箱 | 已有页面和聚合接口，但只显示基础消息 | 升级为真正聊天工作台 |
| 陪伴关系阶段 | 已实现初识、升温、暧昧、稳定 | 在 UI 中展示，并影响回复建议 |
| 情景记忆 | 已有 SQLite 存储和提示词注入 | 做成聊天对象画像面板 |
| 亲密度分数 | 已有基于聊天事件的分数 | 做成关系热度、流失风险、优先级 |
| 语音生成 | 已有 TTS 管线和预览接口 | 放进回复区做一键生成 |
| 人设配置 | 已有人设和账号绑定基础 | 做成可视化人设选择 |
| 安全和节奏 | 已有冷却、黑名单、审批、安全跳过 | 做成自动化等级设置 |

### 4.2 当前主要短板

- 统一收件箱还只是“最近消息列表”，不是完整聊天工作台。
- 翻译能力分散在不同模块里，还没有统一的翻译服务。
- AI 分析结果还没有以“标签、分数、建议”的形式展示给运营。
- 客户画像和关系阶段已存在，但没有形成好用的右侧面板。
- 语音生成已有接口，但没有成为主流程的一键能力。
- 自动化策略还偏配置和代码，没有变成运营人员能理解的开关。
- 产品演示路径还不够清晰，销售时不容易一眼看出差异化。

### 4.3 现有知识库和人设能力如何整合

当前项目已经有知识库、话术、人设、情景记忆和全局回复规则，这些能力第一版必须复用，而不是另起一套。

| 现有资产 | 当前位置 | 第一版整合方式 |
| --- | --- | --- |
| 陪护/转化知识库 | `domains/conversion/kb/seeds.yaml`、`categories.yaml` | 扩展为“交友开场、冷场接话、情绪安抚、关系升温、边界降温、复聊问候”等分类 |
| 回复策略 | `config/reply_strategies.yaml` | 从业务客服策略改造成聊天策略：秒回问候、短消息轻回、深聊支持、沉默观察 |
| 常用模板 | `config/templates.yaml` | 保留问候、小聊模板，移除或弱化订单/通道模板在陪护场景的默认触发 |
| 人设库 | `config/profiles_runtime.yaml` | 作为第一批演示人设来源：温柔型、活泼型、成熟型、克制型、专业型 |
| 人设绑定 | `config/bindings_runtime.yaml` | 用于账号/会话绑定不同人设，后续在 UI 中可视化管理 |
| 情景记忆 | `EpisodicMemoryStore` | 用作“这个人喜欢什么、讨厌什么、上次聊到哪”的长期记忆 |
| 关系阶段 | `companion_relationship` | 用作初识、升温、暧昧陪伴、稳定陪伴的语气控制 |
| 亲密度分数 | `IntimacyEngine` | 用作热度、流失风险和复聊优先级 |
| 语音管线 | `TTSPipeline` | 用于通用声音、授权克隆声音、变声变语种能力的统一入口 |

### 4.4 知识库优化方向

第一版需要把知识库从“销售转化/客服问答”优化成“聊天陪护运营知识库”。

建议新增知识库分类：

- 开场破冰：适合陌生人、刚加好友、第一次回复。
- 冷场接话：对方只回“嗯”“哈哈”“还好”等短句时如何继续。
- 情绪安抚：对方难过、焦虑、生气、失落时如何接住。
- 关系升温：对方积极时如何自然推进，不显得油腻。
- 关系降温：对方抗拒、反感、要求正经时如何收敛。
- 复聊问候：对方沉默一段时间后如何轻量唤醒。
- 边界与合规：金钱、隐私、成人、危险话题如何拒绝或转人工。
- 主播私信：粉丝互动、感谢支持、邀约观看、维护好感。
- 多语言语气：不同语言下如何保持自然，不直译中文腔。

知识库在产品中的用法：

- 不直接把知识库原文发给用户。
- 知识库作为 AI 回复建议的参考。
- 运营可以在右侧看到“本次建议参考了哪些话术”。
- 命中高风险知识库时，回复必须进入人工审核。
- 后续可把高采纳回复沉淀回知识库，形成团队自己的聊天方法库。

## 5. 第一版 MVP 范围

### 5.1 第一版一定要做

- 多平台聊天列表：Messenger、WhatsApp、Telegram、LINE 四个平台一起进入工作台。
- 完整聊天窗口：看到原文、译文、发送状态和最近上下文。
- 自动翻译：对方消息翻译成中文，我方回复翻译成对方语言。
- AI 聊天分析：情绪、意图、关系阶段、风险等级。
- 3 种回复建议：温柔版、简短版、继续深入版。
- 一键改写：更自然、更亲切、更克制、更幽默。
- 一键语音预览：把回复生成语音，先支持通用授权声音，再接克隆声音和变声变语种。
- 聊天对象档案：语言、平台、关系阶段、偏好、备注。
- 三种回复模式：全部 AI 自动回复、人工审核回复、多答案选择回复。
- 自动化开关：不同账号、不同平台、不同风险等级可单独设置。
- 知识库整合：用现有知识库和话术库生成更贴近业务的人设回复。
- 操作记录：谁生成、谁修改、谁发送。

### 5.2 第一版暂时不做

- 不做 18+ 平台全覆盖。
- 不做复杂电商订单系统。
- 不做大规模群发营销。
- 不做复杂多租户计费。
- 不做过重的 BI 报表。
- 不做未经授权的真人声音克隆；克隆声音必须有授权记录。
- 不做承诺现实恋爱关系的内容。

## 6. 产品页面规划

### 6.1 首页：聊天工作台

推荐布局：

- 左侧：平台、账号、聊天对象列表。
- 中间：聊天记录，原文和译文一起展示。
- 右侧：AI 分析、回复建议、对象档案、关系阶段。
- 底部：输入框、翻译开关、语音按钮、发送按钮。

### 6.2 右侧 AI 助手面板

右侧面板重点显示：

- 当前语言
- 当前情绪
- 当前意图
- 关系阶段
- 亲密度/热度
- 流失风险
- AI 推荐下一步
- 3 条回复建议
- 语音生成按钮

### 6.3 聊天对象档案页

档案页展示：

- 基础信息：昵称、平台、国家、语言。
- 关系信息：阶段、热度、最近活跃、沉默天数。
- 偏好信息：喜欢的话题、禁忌话题、称呼偏好。
- 历史摘要：最近聊过什么、上次的重要信息。
- 运营备注：人工可编辑。

### 6.4 人设与话术页

用于配置：

- 不同账号的人设
- 语气风格
- 禁用话术
- 回复长度
- 亲密度上限
- 语音风格
- 自动回复等级
- 知识库话术分类
- 克隆声音授权状态

### 6.5 数据看板

第一版只做简单数据：

- 今日新消息数
- 已回复数
- AI 建议采纳率
- 平均响应时间
- 关系升温人数
- 流失风险人数
- 语音生成次数
- 不同回复模式使用次数
- 知识库话术采纳次数

## 7. 分阶段开发计划

### 阶段 0：产品定版和演示数据准备

周期：2-3 天

目标：把方向、演示路径、MVP 边界定下来。

开发任务：

- 确认第一批渠道：Messenger、WhatsApp、Telegram、LINE。
- 确认第一批业务场景：交友陪聊、情感陪护、私信维护。
- 准备 20-30 条多语言演示对话。
- 准备 3-5 套人设：温柔、活泼、成熟、克制、幽默。
- 明确自动化等级：全部 AI 自动回复、人工审核回复、多答案选择回复。
- 明确语音能力分层：通用授权声音、克隆声音、语音变声/变语种。
- 明确合规边界和禁用内容。
- 整理现有知识库，补充陪护/交友/主播私信场景话术。

验收标准：

- 有一条 3 分钟内能讲清楚的演示流程。
- 产品负责人确认第一版不再以电商为主线。
- 开发任务可以拆到具体模块。

---

### 阶段 1：统一聊天工作台 2.0

周期：2 周

目标：把现有统一收件箱升级成真正可用的聊天工作台。

现有基础：

- `src/web/routes/unified_inbox_routes.py`
- `src/web/templates/unified_inbox.html`
- Messenger、WhatsApp、LINE、Telegram 各自已有消息来源。

开发任务：

- 新增统一消息格式：平台、账号、对方、原文、译文、时间、方向、状态。
- 四个平台同时纳入统一列表：Messenger、WhatsApp、Telegram、LINE。
- 新增完整聊天记录接口，不只显示最后一条消息。
- 左侧增加平台、账号、未读、风险、关系阶段筛选。
- 中间聊天区展示原文和译文。
- 右侧预留 AI 分析和对象档案区域。
- 发送接口统一封装，不让前端关心底层平台差异。
- 消息加载失败时给出明确提示。

验收标准：

- 能在一个页面看到 Messenger、WhatsApp、Telegram、LINE 的待回复消息。
- 点击一个人能看到最近聊天上下文。
- 能从工作台发送文本回复。
- 页面每 10-15 秒刷新一次，且不会打乱当前选中会话。

---

### 阶段 2：统一翻译服务

周期：1 周

目标：把“翻译”从零散能力变成平台级服务。

开发任务：

- 新增 `TranslationService`。
- 支持语言识别、原文翻译、回复翻译。
- 增加翻译缓存，减少重复成本。
- 保留数字、链接、昵称、平台名。
- 支持“直译”和“聊天语气翻译”两种模式。
- 在聊天窗口展示原文和译文。
- 在发送前预览对方语言版本。

验收标准：

- 英语、日语、西班牙语、阿拉伯语、葡萄牙语等常见语言可正常识别。
- 同一条消息重复翻译能命中缓存。
- 中文回复可以生成对方语言版本。
- 翻译失败时不影响人工查看原文和手动发送。

### 阶段 3：AI 聊天分析与回复建议

周期：2 周

目标：让系统从“翻译工具”升级成“聊天助手”。

开发任务：

- 新增聊天分析模块，输出情绪、意图、关系阶段、风险等级。
- 为每条待回复消息生成 3 个回复建议。
- 支持三种发送工作流：AI 全自动、人工审核、多答案选择。
- 支持回复风格切换：温柔、幽默、简短、深入、安抚。
- 接入现有陪伴关系阶段，让不同阶段生成不同亲密度回复。
- 接入情景记忆，让回复能自然提到对方偏好。
- 接入现有知识库，让回复能参考团队话术、禁忌和场景 SOP。
- 增加风险判断：敏感、投诉、越界、隐私、金钱、辱骂等。
- 高风险回复必须人工确认。

验收标准：

- 每条新消息都能看到清晰标签：情绪、意图、风险。
- 回复建议不出现客服腔，不出现生硬模板。
- 同一用户在不同关系阶段，回复亲密度有明显差异。
- 用户表达“不喜欢太亲密”时，后续回复自动降温。
- 多答案选择模式下，运营可在 3 条建议中选择一条并修改发送。

### 阶段 4：聊天对象画像和关系运营

周期：1 周

目标：让运营人员知道“这个人是谁、聊到什么程度、下一步怎么维护”。

现有基础：

- `ContactStore`
- `EpisodicMemoryStore`
- `companion_relationship`
- `IntimacyEngine`

开发任务：

- 右侧展示聊天对象画像。
- 展示语言、国家、平台、最近活跃。
- 展示关系阶段和热度分数。
- 展示用户偏好、重要记忆、禁忌话题。
- 支持人工备注和标签。
- 支持手动重置关系阶段。
- 支持“需要重点维护”“可能流失”等运营标签。

验收标准：

- 换一个运营人员接手，也能快速理解这个人。
- 系统能显示为什么建议升温、降温或转人工。
- 画像信息不会污染不同平台的不同人，合并要谨慎。

### 阶段 5：语音回复和多媒体增强

周期：1 周

目标：把语音变成明显差异化卖点。

现有基础：

- `src/ai/tts_pipeline.py`
- `/api/voice/tts-test`
- Messenger/WhatsApp 语音相关能力。

开发任务：

- 在回复建议旁加入“生成语音”按钮。
- 支持通用授权声音：温柔女声、成熟女声、年轻男声等。
- 支持克隆声音能力入口：必须有授权、参考音频和使用记录。
- 支持语音变声/变语种能力入口：同一句回复可生成目标语言语音。
- 支持不同语言语音。
- 显示语音时长和试听按钮。
- 设置语音长度上限，避免生成太长。
- 记录语音生成次数和失败原因。
- 后续再开放自动发送语音。

验收标准：

- 任意一条回复建议可一键生成语音预览。
- 语音文件可试听。
- 生成失败时可以回退为文字回复。
- 未授权声音不允许用于克隆。
- 克隆声音、变声、变语种功能在 UI 中分层展示，避免运营误用。

### 阶段 6：自动化等级和运营安全

周期：1 周

目标：让系统能安全地从“辅助”走向“半自动”。

开发任务：

- 增加自动化等级：
  - 只分析，不生成回复。
  - 多答案选择回复。
  - 生成回复，人工确认。
  - 低风险自动发送。
  - 全部 AI 自动回复，仅限白名单账号和白名单场景。
- 增加高风险拦截：
  - 金钱
  - 隐私
  - 投诉
  - 情绪崩溃
  - 违法违规
  - 过度亲密
- 增加发送冷却和频率限制。
- 增加自动发送记录和回滚说明。
- 增加黑名单和停止联系保护。

验收标准：

- 高风险内容不会自动发送。
- 每条自动发送都有记录。
- 运营主管可以随时切回人工确认模式。
- 每个账号、平台、人设都可以单独设置自动化等级。
- 用户明确拒绝联系时，系统停止主动打扰。

## 10. 技术落地原则

为了让项目能稳定落地，开发时遵守以下原则：

1. **先产品化已有能力，不重写底层。**  
   现有 RPA、AI、记忆、语音、关系阶段已经有基础，第一轮重点是统一体验。

2. **先人工确认，再逐步自动化。**  
   陪护和交友聊天容易出现语气、边界和合规问题，所以第一版必须提供“AI 自动、人工审核、多答案选择”三种开关，让客户按风险选择。

3. **翻译、分析、回复建议要分层。**  
   不要把所有能力都塞进一个提示词里。翻译是翻译，分析是分析，回复是回复。

4. **每个回复都要可追溯。**  
   需要知道 AI 为什么这样建议、谁确认发送、是否生成语音。

5. **关系阶段不能盲目升温。**  
   用户冷淡、拒绝、反感时，系统要能降温。

6. **语音必须合规授权。**  
   通用声音可先用；克隆声音和语音变声/变语种必须有授权、日志和使用边界。

7. **电商先做插件，不做主线。**  
   保留后续扩展能力，但第一版卖点集中在聊天关系维护。

8. **知识库要变成团队聊天方法库。**  
   现有知识库先复用，再补充陪护、交友、主播私信、边界降温等场景；高采纳回复后续沉淀回知识库。

## 11. 验收指标

### 产品体验指标

- 运营人员 5 分钟内能理解怎么使用。
- 新消息进入工作台后，10 秒内能看到分析结果。
- AI 回复建议至少 60% 可以直接采用或轻微修改后采用。
- 一条回复从生成到发送不超过 3 次点击。
- 语音生成可在 30 秒内完成。

### 业务效果指标

- 单个运营可同时维护 20-50 个活跃对话。
- 平均响应时间降低 50% 以上。
- 新人培训时间减少 50% 以上。
- 沉默用户复聊成功率可被统计。
- AI 回复建议多数可直接采用或轻微修改后采用。
- 知识库命中的高价值话术能被统计和复用。
- 高风险自动发送为 0。

### 稳定性指标

- 系统连续运行 7 天不需要频繁人工重启。
- RPA 失败有明确错误提示。
- 翻译失败不阻塞人工回复。
- TTS 失败可回退文字。
- 关键操作有日志。

## 16. 最终建议

第一阶段我们要避免两个误区：

- 不要做成普通翻译软件。
- 不要一开始就追求全自动聊天。

正确路线是：

**先做一个运营每天愿意打开的聊天工作台，再把 AI 能力一层层加进去。**

只要第一版能做到“看懂、判断、会回、能发语音、能记住人”，就已经明显超出 HelloWord、易翻译这类传统聊天翻译工具。

## 18. 架构修正与工程底座（v1.1 新增，优先级最高）

### 18.1 先把话说清楚：v1.0「7 阶段已完成」的真实成色

按代码复核，v1.0 交付的是一个**可点击的前端骨架 + 规则桩**，不是文档描述意义上的「AI 工作台」。这不是否定 v1.0——作为演示骨架它达标了——但若按 v1.0 的口径「进入真实账号试跑」，会立刻撞墙。诚实对照如下：

| v1.0 文档说法 | 代码实况（已核实） | 影响 |
| --- | --- | --- |
| 阶段 2「统一翻译服务」完成 | `translation_service.py` 在无 provider 时**直接返回原文** + `provider_unavailable`（L106-117）；而 `config.yaml` 其实**已配 DeepSeek**（L899-905），默认演示路径并未接上 | 演示时「翻译」可能根本没翻 |
| 阶段 3「AI 聊天分析」完成 | `chat_assistant_service.py` 是**纯关键词规则**（`if "难过" in t`），回复建议是**写死的中文模板**（L183-207），`ai_client` 参数存在但从未真正调用 LLM | 「AI 分析」名不副实，无法泛化 |
| 阶段 6「自动化等级」完成 | automation mode 存在**进程内 dict**（`app.state.unified_inbox_automation`），重启即丢 | 不能进生产 |
| 「统一消息模型」完成 | **无持久化**。`thread`/`profile`/`automation` 每个端点都调用 `_collect_all_chats(limit=100)` 重新抓全平台（routes L467/575） | 性能差、无真实会话历史、无法做缓存/分析沉淀 |

**根因**：v1.0 在从「电商」转向「陪护」时，把电商竞品文档里最有价值的**工程底座（持久化数据模型 + 分层管线）一起丢掉了**。第 18 章的任务就是补回来。

### 18.2 补回被跳过的 Phase 0.5：持久化数据底座

这套数据模型与做陪护还是电商无关，是任何「多平台会话工作台」的通用骨架。SQLite 起步（项目现状即 SQLite），多租户/高并发后再迁 PostgreSQL + Redis。所有 `ALTER TABLE` 按 CLAUDE.md 约定集中到对应 `database.py` 的 migration 列表。

| 表 | 关键字段 | 解决的问题 |
| --- | --- | --- |
| `conversations` | 平台、账号、对象、状态、负责人、关系阶段、风险等级、最后消息 | 替代每次重抓；成为会话列表的真实来源 |
| `messages` | 原文、译文、语言、方向、平台 message_id、ts | **写一次、读多次**；真实聊天历史 |
| `message_analysis` | 情绪、意图、风险、置信度、建议缓存 | 分析结果沉淀，不重复调用 LLM |
| `translation_memory` | 原文 hash、译文、引擎、命中次数 | 落实 v1.0 KPI「重复句命中缓存」，跨进程有效 |
| `automation_policy` | 会话/账号/平台/人设级模式 | 替代进程内 dict，落库且可分层 |
| `memory_facts` | 对象、事实、来源、更新时间 | 记忆回写闭环（见 18.4） |

做完这一步，18.1 表里「重抓、重启丢状态、缓存失效」三个问题一次性解决。

### 18.3 把规则桩换成「LLM 主 + 规则兜底」，且单次结构化调用

- **认知更新**：v1.0 原则 #3「翻译、分析、回复要分层、不要塞进一个提示词」在 2024 年成立，但 2026 年有了**结构化输出（JSON schema / function calling）**后，应改为**概念分层、一次调用**——一个请求让模型同时返回 `{语言, 情绪, 意图, 风险, 关系阶段, next_step, 3 条建议}`。把 3 次 LLM 往返压成 1 次，才打得到 v1.0 自定的「10 秒内看到分析」KPI，成本也降约 2/3。
- `ChatAssistantService.analyze()` 真正接上 `ai_client`，规则版**降级为两个角色**：① LLM 不可用时的 fallback；② 高风险硬拦截守卫（自残/金钱/隐私/未成年等关键词必须规则强拦，**不能只信 LLM**）。
- `TranslationService` 默认接上已配置的 DeepSeek，`provider_unavailable` 只在真未配置时出现。

### 18.4 记忆回写闭环（留存第一杠杆）

2026 年 AI 陪伴赛道用户**最大抱怨就是「它不记得我」**。项目已有 `EpisodicMemoryStore` / `IntimacyEngine` / `companion_relationship`，但当前 profile 面板是**只读展示**（routes L200-235），缺「每轮对话后抽取事实回写」的闭环。

实现：每轮发送后异步抽取「对方偏好/事实/禁忌/重要事件」写入 `memory_facts`（**抽取要点后存，删原始 scrollback**，符合 2026 隐私架构最佳实践），下次进对话注入提示词。复用已有 store，投入小、留存收益大。

### 18.5 重排后的优先级

不推翻 v1.0 的 7 阶段，而是**重排 + 补地基 + 把风险拉到台面**：

| 顺序 | 内容 | 为什么 |
| --- | --- | --- |
| **P0.5** | 持久化数据底座 + 翻译记忆（18.2） | 补回地基，解决重抓/丢状态 |
| **P1** | LLM 真正接上（单次结构化）+ 高风险规则守卫（18.3） | 让「AI」名副其实，命中 10s KPI |
| **P2** | 克隆音色 + 跨语种语音前置（第 20 章） | 唯一护城河，前置到首演 |
| **P3** | 记忆回写闭环（18.4） | 留存第一杠杆 |
| **P4** | 防封号横切层 + 自动化分级落库（第 23 章） | 决定能否进真实账号试跑 |
| **P5** | 画像可编辑 + 数据看板 + 安装包/私有化交付（第 21 章） | 同 v1.0 阶段 4/7 |

### 20.1 为什么前置

HelloWord / 易翻译能做翻译，但**做不到「用我克隆的音色，把中文回复直接念成对方语言的语音」**。这是唯一无法被翻译工具抄走的差异点。v1.0 把语音排在阶段 5 是低估了它——应作为**第一卖点前置到首个演示闭环**。

### 20.2 技术已就绪

| --- | --- | --- |
| **Qwen3-TTS（项目已配 `qwen3-tts-vc-2026-01-22`）** | **97ms 流式延迟、3 秒参考音色克隆、跨语种** | 直接对应 v1.0「克隆声音 + 变声/变语种」三类能力 |

## 21. 安装程序与交付形态（v1.1 新增）

### 21.1 现状

项目当前是 **FastAPI 后端 + Android RPA（adb / UI Automator 驱动手机 App）**，启动靠 `python main.py`。这对运营客户门槛太高——竞品 HelloWord 是**桌面客户端 + 手机端 + 独立应用**多形态分发，双击即装。我们必须补上「安装即用」。

### 21.2 桌面安装包推荐方案：Tauri + FastAPI sidecar + PyInstaller

2026 年这套组合是 Python 后端做桌面应用的主流：

1. **PyInstaller / Nuitka** 把整个 Python 栈（解释器 + 依赖 + DLL）打成单个可执行文件。
2. **Tauri** 作为外壳：用系统原生 WebView（比 Electron 轻，无 Chromium 臃肿，安装包可做到 <10MB），把 PyInstaller 产物作为 **sidecar 进程**拉起。
3. 产出 **Windows / macOS / Linux 原生安装包**，用户双击安装，**无需装 Python**。
4. **大文件（本地模型、ASR/TTS 权重）首次运行时按需下载**，保持安装包小。

> 现有 Web 后台（FastAPI + 模板）几乎可零改造塞进这个壳：Tauri 窗口指向 `127.0.0.1` 本地 FastAPI 即可，统一收件箱页面直接复用。

> **⚠️ v1.2 选型修正**：21.2 的 Tauri 方案适用于「只展示我们自己的 UI」。但 v1.2 确定的形态需要**在客户端内嵌入 WhatsApp Web / Telegram Web 等第三方页面、做会话隔离多开、并注入翻译脚本**——这类「嵌入 N 个第三方 SPA + 独立 session + 脚本注入」场景，**Electron 的 BrowserView/partition + preload 模型成熟度远高于 Tauri**（HelloWord/易翻译这类产品本身也是 Electron/CEF 系）。因此**客户端外壳选 Electron**，本地 FastAPI AI 后端作为 sidecar。详见第 24 章。

### 21.3 交付形态分层

| 形态 | 适用客户 | 说明 |
| --- | --- | --- |
| **本地单机版** | 个人/小团队 | 安装包内置后端 + 本地 SQLite；手机用 USB/局域网 adb 连接 |
| **控制端 + 设备端分离** | 多账号矩阵团队 | 桌面端做控制台，Android 设备/云手机做执行端，通过内网/专线连 |
| **私有服务器版** | 重数据安全客户 | Docker Compose 标准化部署（呼应 v1.0 阶段 7），数据不出企业服务器 |

### 21.4 与 v1.0 阶段 7 的关系

v1.0 阶段 7「一键部署说明 + 初始配置向导」在 v1.1 里**具体化为**：Tauri 安装包 + 首启配置向导（填 AI key / 翻译档 / 绑定设备 / 选人设）+ 设备健康检查页。

## 22. 登陆与多开技术（v1.1 新增）

> ⚠️ **本章已被第 24 章取代**：创始团队决定不做手机农场，改为「发程序、用户扫码登陆」的桌面客户端模式。下文「物理机/云手机/Android RPA」仅作为历史方案与可选 legacy 渠道保留，新架构以第 24 章为准。

### 22.1 现状：物理 Android 手机农场

代码实况（`config.yaml`）：多开 = **多台真实 Android 手机**，每台一个 `adb_serial`；**一台手机可同时承载多平台多账号**（例如 serial `IJ8HZLORS485PJWW` 同时跑 `line_ij8` + `msg_ij8` + `wa_ij8`，见 `devices:` L1658-1713）。这套能跑，但扩展性、稳定性、防封号都受限于物理设备。

### 22.2 多开技术谱系（按成本/隔离度/封号风险排序）

| 方案 | 隔离度 | 封号风险 | 成本 | 适用 |
| --- | --- | --- | --- | --- |
| **物理多机（现状）** | 高（真机真环境） | 低-中 | 高（买机 + 维护） | 早期、少量高价值号 |
| **应用分身 / 虚拟化**（Parallel Space、App Cloner、Island、VirtualApp 系） | 中（同机数据隔离，可改 device id / GPS / 通讯录） | 中-高（同机指纹易关联） | 低 | 一机多号、轻量扩展 |
| **云手机**（Redfinger 等 ARM 虚拟化） | 高（独立系统 + 独立 IP，24h 在线） | 低（IP/设备隔离好，**降低 digital linkage**） | 中（按台月租） | **多账号矩阵推荐**：隔离 IP、真机不离线也不掉号 |
| **官方多账号 / 官方 API** | 最高（平台原生支持） | 最低 | 视渠道 | 合规优先客户 |

**建议演进路线**：物理机（保高价值号）→ 云手机（规模化矩阵，主力）→ 能上官方 API 的渠道走官方（见 22.4）。把这一层抽象成 **Device/Channel Adapter**，上层工作台看不到底层是真机、分身还是云手机。

### 22.3 登陆与会话管理

- **会话隔离**：每账号独立 session / 登陆态 / cookie / MTProto session 文件，互不污染。
- **登陆方式**：二维码登陆（WhatsApp/LINE）、短信/验证码、Telegram session 复用；统一做**掉线检测 + 自动重连 + 重新登陆引导**。
- **账号健康度**：每账号维护「在线状态 / RPA 成功率 / 失败截图 / 风控告警 / 最近活跃」，呼应 v1.0 阶段 7 的设备健康面板。

### 22.4 官方 API 双轨登陆

能走官方就别走 RPA（封号风险天差地别，见第 23 章）：

- **Telegram**：MTProto session（项目已有 `telegram_client.py`）/ Bot API。
- **WhatsApp**：Cloud API（注意 24 小时窗口 + 模板消息合规）。
- **LINE**：LINE Official Account（OA）+ Messaging API。
- **Messenger**：受 Meta 政策限制，优先官方页面 API；个人号场景才用 RPA。
- 统一抽象成 **Channel Adapter**：`官方 API 优先，RPA/云手机其次`，前端无感知。

## 23. 防封号体系（v1.1 新增，深度分析）

> 这是被 v1.0 严重低估的**生死线**。整个产品建立在 WhatsApp/Messenger/Telegram 的 RPA 与账号之上；**「全自动 AI 回复 + 多账号」恰恰是封号风险最高的模式，而 v1.0 把它列为目标终态**。本章把风险拉到台面并给工程清单。

### 23.1 2026 平台风控现状（已检索）

- **Meta / WhatsApp**：2026-01-15 起**封禁 WhatsApp 上的第三方 AI 机器人**；完整性检测能识别 **DOM 注入/改写**并标记账号；垃圾/自动/批量消息直接违反 ToS。
- **Telegram**：用 **AI 行为建模**封号——消息间隔过于均匀、UI 操作无人类微延迟会被**时间分析**抓出；多账号最致命的是**环境信号关联（digital linkage）**：同设备/同 IP/同指纹把多个号串成一窝端。
- **共性**：封号不再靠单条规则，而靠**行为模式 + 环境关联**的综合判定。

### 23.2 三层防护体系

**第 1 层 · 行为人性化（对抗时间分析）**
- 随机化消息间隔，禁止固定 interval。
- 模拟打字延迟、阅读停顿、已读再回的时间差。
- 单账号每日发送量/主动发起量限流 + 冷却（复用项目已有 cooldown/限流基础）。
- 活跃时段拟人化（不在对方深夜批量发）。
- 避免完全相同的回复文本批量复用（AI 生成天然有助于此）。

**第 2 层 · 账号/设备/网络隔离（对抗 digital linkage）**
- 每账号独立 IP（住宅代理 / 云手机自带独立 IP），**严禁多号共用一个出口 IP**。
- 每账号独立设备指纹（真机 / 云手机 / 虚拟化改 device id）。
- 账号与设备/IP/指纹的**绑定关系固定**，不频繁漂移（漂移本身也是信号）。
- 新号**养号期**：先低频真人化使用，再逐步接入自动化。

**第 3 层 · 策略/分级（对抗批量与高风险）**
- 自动化分级与 v1.0 阶段 6 对齐，但**默认档下调**：默认「人工审核」，而非「全自动」。
- 高风险内容（金钱/隐私/投诉/未成年/敏感）强制人工，**绝不自动发**。
- **「跨多账号全自动」默认关闭**，仅对白名单账号 + 白名单场景开放，且需主管二次确认。

### 23.3 双轨制：官方 API 优先

| 渠道 | 优先轨 | 次轨 |
| --- | --- | --- |
| Telegram | MTProto / Bot API（合规、可审计） | RPA |
| WhatsApp | Cloud API（模板 + 24h 窗口合规） | 云手机 RPA（个人号场景） |
| LINE | OA Messaging API | RPA |
| Messenger | 官方页面 API | RPA（个人号，最高风险，限场景） |

能上官方的渠道走官方，RPA 仅覆盖官方不支持的个人号灰色地带，且加全套隔离 + 人性化。

### 23.4 账号健康度监控与熔断

- 实时指标：RPA 成功率、被限速信号、异常登出、对方无回率突变、平台告警弹窗截图。
- **自动熔断**：单账号风险指标超阈值时，自动降级（停自动发 → 转人工 → 暂停该号），避免一个号把同设备/同 IP 的号带崩。
- 账号画像：养号天数、历史封禁记录、风险评分，用于决定该号能开到哪一档自动化。

### 23.5 产品与合规红线（写进合同与 UI）

- 产品内**明确把「全自动」标为高风险档**，给客户知情同意，不当默认终态卖。
- 不承诺「绝对不封号」——任何第三方自动化都有平台政策风险；产品的价值是**把风险降到可控并提供熔断**。
- 用户明确拒绝联系（`停止联系` 意图）时，系统停止主动打扰（v1.0 已有逻辑，保留强化）。

### 23.6 对应开发任务（接入 P4）

1. 行为人性化层：间隔/延迟/限流随机化模块（横切所有 RPA runner）。
2. 账号-设备-IP 绑定与隔离配置（扩展现有 `accounts:` / `devices:` 配置）。
3. 账号健康度采集 + 风险评分 + 自动熔断。
4. 自动化默认档下调 + 「跨号全自动」白名单门禁 + 主管二次确认。
5. 官方 API Adapter 逐渐替换高风险 RPA 路径（能替则替）。
6. 云手机接入预研（Redfinger 类 ARM 虚拟化）作为规模化矩阵方案。

## 24. 交付架构重大调整（v1.2 新增，权威架构）

### 24.0 创始团队定下的硬约束

- **不做手机农场**（不买/不托管 Android 设备，不做云手机矩阵）。
- 产品是**一个发给用户的程序**；**用户用自己的账号、自己扫码登陆**（WhatsApp Web / Telegram Desktop / LINE 那样）。
- 据此重做第 18.x 的三大核心判断，并定义不可被翻译工具复制的**护城河**。

这条约束实际上**大幅降低了产品的封号风险和运营成本**（不再是「我们替用户跑号」，而是「用户在自己机器上用自己的号，我们做增强」），是好事。

### 24.1 易翻译 / HelloWord 的工作机制拆解（已检索 + 推断）

这类产品（HelloWord、易翻译、ChatKnow、OneChat 等）本质不是「无头浏览器翻译」，而是**桌面聚合客户端**，机制如下：

1. **外壳**：Electron / CEF 桌面客户端（Windows 为主，部分 Mac），内嵌多个第三方网页版聊天客户端。
2. **多开 = 会话隔离的多个 WebView**：每个账号一个独立 session 分区（独立 cookie / localStorage / IndexedDB），互不串号。N 个分区 = N 个独立登陆。这就是它们宣传的「应用无限多开」的真相——**不是改手机，是浏览器级会话隔离**。
3. **登陆**：在内嵌的 WhatsApp Web / Telegram Web / LINE 里**用户自己扫码/验证码登陆自己的号**。产品不持有账号密码。
4. **翻译注入**：用 **MutationObserver** 监听新消息 DOM 节点 → 调翻译 API → 在原文下方**插入译文浮层**；拦截**输入框**，把中文译成对方语言后再发。
5. **翻译计费**：多翻译引擎（Google/DeepL/腾讯/百度/有道/小牛）**按字符量计费**，这是它们的核心商业模式。
6. **基础 SCRM**：好友统计、客户档案、快捷话术、简单自动回复。

**它们的天花板**：停留在「翻译 + 多开 + 简单自动回复」，**不懂情绪、不记得人、不会判断关系、没有语音人格**。这正是我们的切入口。

### 24.2 我们的客户端架构（Electron 壳 + 隔离多开 + 翻译注入 + 本地 AI 后端）

```
┌─────────────────── Electron 桌面客户端（发给用户，双击安装） ───────────────────┐
│  左：账号/会话列表        中：内嵌官方网页客户端(隔离 session)      右：AI 助手面板  │
│                          ┌───────────────────────────────┐                      │
│  partition: wa_user_a →  │ WhatsApp Web (用户自己扫码登陆)  │ ← preload 注入脚本   │
│  partition: tg_user_a →  │ Telegram Web                   │   · MutationObserver  │
│  partition: line_a   →  │ LINE / Messenger               │   · 译文浮层          │
│                          └───────────────────────────────┘   · 输入框拦截        │
│                                      ↕ IPC / HTTP                                 │
└──────────────────────────────────────┼──────────────────────────────────────────┘
                                        ↓ 127.0.0.1 (sidecar)
        ┌──────────── 本地 FastAPI AI 后端（复用现有代码） ────────────┐
        │ TranslationService · ChatAssistantService · 记忆/画像        │
        │ 人设/知识库 · TTS(qwen3-tts-vc) · 持久化(SQLite, 第18章)      │
        └──────────────────────────────────────────────────────────────┘
```

要点：
- **外壳 Electron**（理由见 21.2 v1.2 修正）：用 `BrowserView` + `session.fromPartition('persist:账号id')` 实现会话隔离多开；用 `preload` 脚本做 DOM 监听/译文浮层/输入框拦截。
- **本地 FastAPI AI 后端作为 sidecar**：现有翻译、分析、记忆、人设、知识库、TTS、持久化**全部复用**，只是从「Web 后台」变成「客户端内的本地服务」。聊天数据默认**留在用户本机**（数据主权卖点）。
- **Telegram 双轨**：除内嵌 Telegram Web 外，可叠加官方 **MTProto**（项目已有 `telegram_client.py`），做更稳的一等公民集成。

### 24.3 三大核心判断在新模型下重做

**判断 ①：数据底座（持久化）——仍然 100% 成立，且更必要。**
桌面客户端必须有本地存储：会话、消息、译文缓存、记忆、人设、自动化策略。第 18.2 的数据模型原样适用，**落在用户本机 SQLite**。这也是「数据不出本机」合规卖点的技术基础。→ **维持 P0.5 最高优先。**

**判断 ②：防封号——风险被结构性降低，重心转移。**

| 维度 | 旧模型（手机农场 RPA） | 新模型（用户扫码 + 内嵌官方网页端） |
| --- | --- | --- |
| 账号归属 | 我们替用户养号 | **用户自己的号，自己登陆**（知情、自愿） |
| 客户端 | 逆向/RPA 驱动手机 App | **官方网页客户端**（WhatsApp Web/Telegram Web） |
| 主要风险 | 设备/IP 关联、RPA 行为、第三方封禁 | **DOM 注入被完整性检测**（主要是 WhatsApp）+ **消息速率/批量行为** |
| 风险等级 | 高 | **中-低**（基线大幅下降） |

新模型下的防封号重心：
- **WhatsApp 注入要克制**：WhatsApp 完整性检测会识别 DOM 改写。**译文浮层尽量「旁路化」**——优先用独立侧栏/悬浮窗展示译文，减少对 WA 自身 DOM 的侵入式改写；MutationObserver 只读监听 + 在我们自己的 UI 层渲染译文，比直接往 WA 消息节点里塞 DOM 安全。
- **速率/行为人性化**（第 23.2 第 1 层仍适用）：随机间隔、打字延迟、限流；**默认人工审核档**，「全自动」默认关。
- **Telegram 走官方 API** 几乎无封号风险（开放协议、允许多客户端）。
- **每账号独立 session 分区**天然隔离，不共享登陆态。
- 产品话术：**「用你自己的账号、官方网页端登陆，比第三方机器人安全」**本身就是卖点；但仍**不承诺绝对不封**，提供熔断与降级。

**判断 ③：多开——从「设备多开」变成「会话分区多开」，成本骤降。**
不再需要 adb/云手机/物理机。多开 = Electron 里 N 个 `persist:` 分区。一台普通电脑即可多开多账号多平台。第 22 章的物理机/云手机内容降级为**可选 legacy 渠道**（仅当某客户坚持用手机端个人号时才用）。

### 24.4 护城河（不可被翻译工具复制的差异化）

HelloWord/易翻译能抄走「翻译 + 多开」，抄不走下面这套**AI 关系层**：

1. **关系记忆层（最硬的护城河）**：每个聊天对象的长期语义记忆 + 关系阶段 + 亲密度，自动注入回复建议。「它记得这个人是谁、聊到哪、喜欢什么」——这是陪伴赛道用户**第一痛点**，翻译工具完全没有。复用 `EpisodicMemoryStore`/`IntimacyEngine`/`companion_relationship` + 第 18.4 回写闭环。
2. **克隆音色 + 跨语种语音**：把中文回复用**用户授权的克隆音色**念成**对方语言**的语音（`qwen3-tts-vc` 已配，97ms 流式、3s 克隆）。翻译工具发不了「我的声音说外语」。
3. **会聊天的 AI（不只翻译，还告诉你下一句怎么说）**：情绪/意图/关系阶段 → 3 风格回复建议 + 一键改写（温柔/幽默/简短/深入/安抚）。
4. **数据主权 + 私有化**：本地优先，聊天数据留在用户本机，可选本地 LLM（Ollama）。对重隐私客户是决定性卖点。
5. **结构性低封号架构**：用户自有号 + 官方网页端 + 辅助而非托管 + 旁路化注入 + 人性化节奏。把「更安全」做成可宣传的产品特性。

一句话护城河：**它们让你「看懂」外语，我们让你「记得人、会接话、用自己的声音、还更不容易被封」。**

### 24.6 落地步骤（接 P0.5 之后插入）

1. **Electron 壳 PoC**：单窗口内嵌一个 WhatsApp Web + 一个 Telegram Web，验证 `persist:` 会话隔离与扫码登陆。
2. **本地 sidecar 接线**：Electron 启动时拉起本地 FastAPI，preload 脚本通过 `127.0.0.1` 调翻译/分析。
3. **旁路化译文浮层**：MutationObserver 只读监听消息 → 在我们自己的侧栏/悬浮层渲染译文（先不侵入式改写 WA DOM）。
4. **输入框增强**：发送前中文→对方语言翻译预览 + 3 条 AI 建议 + 一键语音。
5. **多账号分区管理 UI**：账号列表、扫码登陆、掉线重连、健康状态。
6. **打包**：electron-builder 出 Windows/Mac 安装包，sidecar 用 PyInstaller 打进去，大模型/TTS 权重首启下载。

> 注意：这是从「Android RPA 后台」到「桌面聚合客户端」的形态切换，是多周工程。**AI 后端不动、只换渠道/外壳层**是关键策略，避免推翻已验证的能力。建议先做第 1 步 Electron PoC 验证可行性，再逐步迁移。
