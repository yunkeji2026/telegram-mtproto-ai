# Telegram MTProto AI 后台监控 - 开发方案

## 1. 目标

- 为 `telegram-mtproto-ai` 提供**可查看与监控**的后台能力。
- 后端提供 **REST API**，前端由其他 AI/团队独立开发，通过接口对接。
- 监控与主进程同机部署，不改变现有业务逻辑，通过「指标采集 + HTTP 服务」实现。

## 2. 架构概览

│  │ (消息队列等)   │  │ (意图/技能)   │  │   (API 调用)          │  │
│  └──────┬───────┘  └──────┬───────┘  └──────────┬───────────┘  │

│  └─────────────────────────────────────────────────────────────┤

## 3. 技术选型

| 模块           | 选型说明 |
|----------------|----------|
| API 框架       | **FastAPI**：异步友好，与现有 asyncio 主循环兼容，可同一进程内用 `uvicorn` 挂载。 |
| 指标存储       | **进程内单例**：内存字典 + 线程安全/异步安全更新，满足单实例监控。 |
| 日志拉取       | **尾读日志文件** + 可选 **SSE 流**：配置中的 `logging.file`，不依赖额外中间件。 |
| 进程/运行状态   | 由主进程在启动时注册「运行中」及 Telegram 连接状态，API 只读。 |

## 4. 功能范围（后端提供）

- **运行状态**：进程是否在跑、Telegram 是否已连接、运行时长。
- **核心指标**：消息收发数、AI 调用次数、响应耗时（含平均/P99）、错误次数、当前队列长度。
- **近期日志**：按行尾读 `config.logging.file`，支持 `?tail=N`；可选 SSE 推送最新日志行。
- **配置摘要**：仅展示与监控/排障相关的非敏感配置（如 session 名、模型名、已启用技能），不暴露 API Key 等。

前端可在此基础上做：仪表盘、图表、告警规则（仅前端定时轮询即可，后端仅提供数据）。

## 5. 部署与运行

- 监控 API 与主程序**同一进程**：在 `main.py` 中，在 `assistant.initialize()` 之后、`assistant.start()` 之前或之中，根据 `config.monitoring.enabled` 启动 FastAPI（如用 `uvicorn` 在 9090 端口）。
- 若需**仅本机访问**：绑定 `127.0.0.1:9090`；若需内网其他机器访问前端：绑定 `0.0.0.0:9090`，并在防火墙/安全组做限制。
- 依赖：`fastapi`、`uvicorn`，可加入 `requirements.txt`。

## 6. 与现有代码的对接点

- **TelegramClient**：在收到消息、发送回复、队列 put 时，调用 `MetricsStore` 记录：收消息数、发回复数、队列长度、单次响应时间。
- **claude-4.6-oups-highClient**：在每次调用模型前后记录：API 调用次数、单次耗时、失败次数。
- **SkillManager**：可选按意图/技能统计次数，便于前端按意图维度展示。
- **AIChatAssistant (main)**：启动时设置「运行中」和启动时间；关闭时清除；将 Telegram 的「是否已连接」定期或按事件写入 MetricsStore，供 `/api/health` 使用。

以上对接均通过「单例 MetricsStore」或由 main 注入到各组件，尽量少改现有类接口（仅增加 1～2 行上报调用）。

## 7. 安全与性能

- 不在 API 中返回任何密钥、Token、手机号；配置摘要仅返回必要且脱敏的信息。
- API 不做认证时，**仅监听本机或内网**，避免暴露到公网。
- 指标为内存结构，无持久化；进程重启后从 0 开始，符合「当前实例监控」语义。
- 日志尾读与 SSE 均限制条数与频率，避免大文件或高频 IO 影响主业务。

## 8. 文档与对接

- **开发方案**：本文档。
- **技术对接方案**：见 `docs/MONITORING_API_SPEC.md`，包含 Base URL、CORS、各接口的请求/响应约定，供前端或其它 AI 直接对接实现。
- **前端对接**：主程序启动且 `config.monitoring.enabled` 为 true 时，监控 API 监听 `http://127.0.0.1:9090`（端口见 `config.monitoring.metrics_port`）。前端直接请求 `GET /api/health`、`GET /api/metrics`、`GET /api/logs?tail=100`、`GET /api/config/summary` 即可。
