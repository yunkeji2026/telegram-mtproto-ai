# 综合优化方案 — 外部建议采纳分析与实施计划

本文档对**其他 AI 的 8 条建议**做采纳分析，并结**测试对话日志**中尚未闭环的项，形成统一优化方案（仅方案与步骤，无代码）。

---

## 一、外部建议采纳分析

### 1. 会话上下文窗口的动态管理（滑动窗口 + 摘要 + 关键信息锚定）

| 维度 | 结论 | 说明 |
|------|------|------|
| **采纳程度** | **部分采纳** | 当前群聊以「单轮/短轮」为主，且额度类已走规则不占上下文；长对话多轮集中在私聊或少数群，可先做「关键信息锚定」，再做「窗口+摘要」。 |
| **关键信息锚定** | **采纳** | 订单号、通道名、额度数值等若随历史被截断，会直接影响订单/额度回复准确性。在每次组 AI prompt 前，从上下文或独立存储中取出「本会话已出现的订单号、当前问询的通道、额度结论」等，拼到 system 或 user 消息顶部，成本低、收益明确。 |
| **滑动窗口 + 摘要压缩** | **二期采纳** | 需明确：当前 claude-4.6-oups-high 调用是否已传入「最近 N 条历史」、N 与 token 上限关系；若尚未传历史，先做「有限条数历史」再考虑摘要。摘要需额外调用模型或轻量模型，建议在观测到「因截断导致答非所问」后再上，避免过度工程。 |

**优化方案要点（不写代码）**：  
- 在组装发往 claude-4.6-oups-high 的上下文时，增加「关键信息块」：从当前会话中抽取并去重的订单号、通道名（EP/JC）、本轮或上轮已给出的额度结论，以固定格式（如「【本会话关键信息】订单号：xxx；通道：EP/JC；已回复额度：…」）放在 prompt 顶部或 system 末尾。  
- 若现有逻辑已传「最近 K 条消息」，先设定 K 与 token 上限的对应关系（如 4k 窗口留 2k 给历史），并打日志在接近阈值时告警；二期再引入「超过阈值则对最早若干条做摘要并替换」的流程。

### 2. 多语言回复的准确性与术语一致性

| 维度 | 结论 | 说明 |
|------|------|------|
| **采纳程度** | **采纳** | 测试日志已出现 Pay y in、代代付等错别字，多语言场景下术语误译风险更大，与现有「专有名词不得错别字」一致，可做增强。 |
| **术语表注入** | **采纳** | 在 system_prompt 中明确「以下术语保留英文或使用指定译法」，并配 YAML 术语表（中英及可选阿/土/葡等）；生成回复后可做**后处理**：对表中术语在回复中进行规范化替换（避免模型自译）。 |
| **语言检测与强制输出语言** | **部分采纳** | 在请求中指定输出语言（如 user_lang）可做；非支持语言回退到英语需先定义「支持语言列表」及检测方式（关键词或轻量检测库），避免误判。 |

**优化方案要点**：  
- 新增或扩展现有配置：`config/terminology.yaml`（或并入现有 YAML），格式可为「原文 → 必须使用的表述」或「原文 → 多语言映射」。  
- 在 system_prompt 中增加「术语规范」段落：列出 Pay in、Pay out、EasyPaisa、JazzCash、代收、代付、EP、JC 等及指定译法；并写明「回复中涉及以上术语时，必须使用上述表述，不得自创译法」。  
- 回复后处理：在发送前对回复文本做一次术语替换（按术语表），仅替换完整词匹配，避免误伤数字或其它句子。  
- 若使用 claude-4.6-oups-high 的 language 类参数，在调用时传入「用户消息检测到的语言」或默认英语，减少模型自由发挥语言。

### 3. 技能工作流的插件化与热加载

| 维度 | 结论 | 说明 |
|------|------|------|
| **采纳程度** | **二期/部分采纳** | 当前技能已按「意图→技能」映射，新增技能需改代码与配置；若业务稳定、新技能不频繁，可维持现状，在「新增技能」时再抽象为注册制。 |
| **技能注册制** | **可选演进** | 统一接口（意图匹配、处理函数、冷却配置）+ 配置文件声明 + 动态加载，有利于扩展；实现时需考虑与现有意图识别、冷却的兼容，避免破坏现有额度/订单优先级。 |
| **冷却规则精细化** | **采纳** | 当前冷却为全局/按用户/按内容，若在技能配置中为每个技能单独设置冷却（如订单 3s、闲聊 10s），并支持按用户等级差异化，可提升灵活性；需与现有 `skills.cooldown` 合并设计，避免两套逻辑。 |

**优化方案要点**：  
- 短期：在现有 `skills.cooldown` 基础上，支持**按意图/技能名**覆盖冷却（如 `cooldown.by_intent: { order_query: 3, small_talk: 10 }`），主流程在检查冷却时先查 by_intent 再回退到全局。  
- 二期：若需要按用户等级（VIP/普通）差异化，在用户上下文或外部接口中提供「用户等级」，冷却检查时按等级取不同配置。  
- 插件化与热加载：作为架构演进项，在确有「频繁新增技能」需求时再设计统一 Skill 接口与配置驱动加载，当前可只做配置化冷却。

### 4. 错误处理与降级兜底策略

| 维度 | 结论 | 说明 |
|------|------|------|
| **采纳程度** | **采纳** | 与「求助其他 AI」中的可用性诉求一致，且实现成本低，优先做。 |
| **多级降级** | **采纳** | claude-4.6-oups-high 超时：重试 1 次 + 指数退避；仍失败则返回预设提示并打日志。订单识图失败：引导用户手动输入订单号。 |
| **熔断机制** | **部分采纳** | 按错误率切换纯规则模式可有效防止雪崩；需定义「错误率窗口」（如最近 1 分钟）与阈值（如 30%），以及「恢复」条件（如 5 分钟内无新错误或错误率回落）。 |

**优化方案要点**：  
- 在调用 claude-4.6-oups-high 的封装处：捕获超时/网络/5xx，首次失败后等待 1～2 秒再重试一次；再次失败则返回固定文案（如「网络繁忙，请稍后再试」），并记录错误类型与 message_id 到日志。  
- 订单相关技能：若依赖识图/OCR 且识别失败，回复中明确「暂未识别到订单信息，请直接发送订单号或再发一张清晰截图」。  
- 熔断：在内存或轻量存储中维护「最近 N 次调用」的成功/失败计数；当失败率超过阈值时，将「是否允许调用 claude-4.6-oups-high」置为否，仅走规则与模板回复，并打 warning 日志；定时或按成功请求逐步恢复。

### 5. 监控、日志与可观测性建设

| 维度 | 结论 | 说明 |
|------|------|------|
| **采纳程度** | **采纳** | 与测试对话日志分析中的「监控埋点」一致，且为排错与业务分析所必需。 |
| **埋点与指标** | **采纳** | 统计：每秒/每分钟请求数、claude-4.6-oups-high 调用耗时（P50/P95）、各技能命中次数、冷却命中次数、分条发送条数；可选：额度规则命中、意图修正次数。 |
| **结构化日志** | **部分采纳** | 在关键路径（触达、意图、技能执行、AI 调用、发送）打日志时，统一增加字段：request_id 或 message_id、user_id、intent、skill、result（ok/fail）、duration_ms；若当前为文本日志，可先采用「固定前缀 + JSON 片段」便于后续导入 ELK/Loki。 |
| **告警** | **采纳** | 当 API 错误率突增或平均响应时间 > 5s 时，通过 Telegram 或邮件告警；可先做简单定时轮询指标再发告警，无需先上复杂告警平台。 |

**优化方案要点**：  
- 为每条消息生成或复用 request_id（如 message_id + chat_id），在整条链路的日志中携带该 id。  
- 在现有监控指标（若有）上增加：按技能/意图的计数、claude-4.6-oups-high 调用次数与耗时、冷却触发次数、分条发送条数；若已有 Prometheus 等，可暴露为 counter/histogram。  
- 告警：定义「错误率 > 30%（最近 5 分钟）」或「平均响应时间 > 5s」时发送一条 Telegram 消息给运维群或指定账号；实现可为定时任务读内存/文件中的统计再判断。

### 6. 安全性加固与账号保护

| 维度 | 结论 | 说明 |
|------|------|------|
| **采纳程度** | **部分采纳** | 多账号轮询与图片脱敏需评估运维与合规成本；消息频率控制与最小必要安全建议优先。 |
| **多账号轮询** | **二期** | 需多会话、负载分配与状态一致性，复杂度高；建议在单账号出现限频或封禁风险时再评估。 |
| **图片脱敏** | **可选/合规驱动** | 若业务或合规要求「不将含身份证/人脸的图片原样送第三方 OCR」，再引入本地脱敏（如 OpenCV 模糊）；否则可仅做「图片不落盘或短期即删」的策略。 |
| **消息收发频率控制** | **采纳** | 严格遵守 Telegram 限制，对群发或连续回复增加随机延迟，模拟真人；与现有「分条发送 delay」可统一为配置项。 |

**优化方案要点**：  
- 在发送侧：除现有分条 delay 外，可配置「全局最小发送间隔」与「随机抖动」，避免短时间大量发出。  
- 密钥与会话：API 密钥与会话文件不提交版本库；生产环境通过环境变量或独立密钥服务读取；会话文件权限限制为当前进程用户。  
- 日志脱敏：对写入日志的 message 内容做截断或脱敏（如只保留前 20 字 + 「…」），避免完整订单号/金额写入明文日志。

### 7. 用户体验细节打磨

| 维度 | 结论 | 说明 |
|------|------|------|
| **采纳程度** | **部分采纳** | 情绪感知与智能分条与现有「情绪增强」「分条发送」可叠加增强；Inline 按钮依赖 Telegram 能力与产品需求。 |
| **情绪感知回复** | **采纳** | 在传入 AI 的 prompt 中增加一句「用户当前情绪：着急/不满/中性」等（可从现有情绪分析或关键词粗判），使模型调整语气；与现有情绪增强器可共用分析结果。 |
| **智能分条** | **采纳** | 当前已按自然段与句号分条；可增强为：避免在数字、专有名词（如 EP、JC、Pay in）中间截断；分条后加「1/3」类序号为可选，需权衡与简洁风格。 |
| **Inline 按钮** | **可选** | 若产品需要「查看物流」「联系客服」等按钮，再在发送消息时附加 reply_markup；需与现有「言简意赅」风格统一。 |

**优化方案要点**：  
- 在调用 claude-4.6-oups-high 时，将上下文分析中的情绪或简单关键词结论写入 system/user 提示（如「用户似乎较着急，请简洁肯定回复」），不改变现有情绪增强逻辑，仅增加输入信息。  
- 分条逻辑：在按句号/段拆分时，若某段包含「EP」「JC」「Pay in」「100」等，避免在该词中间切开；可优先在标点或换行处切，且单条长度仍受 max_chars 限制。  
- Inline 按钮：作为产品可选功能，在需求明确后再定义按钮文案与回调逻辑（如仅跳转链接或触发固定回复）。

## 二、测试对话日志相关项的并入

以下为**测试对话日志分析**中已实施与未闭环项，统一并入本综合方案。

### 已实施（仅列项，不重复方案）

- 额度未指定通道时默认返回 EP+JC。  
- prompt 不重复贴长段、专有名词无错别字。  
- 「介绍一下」+ 上文额度/通道 → 意图修正为 channel_info 并直接给额度。  
- 情绪增强对额度/通道类回复跳过。

### 未闭环项 → 纳入综合方案

| 来源 | 项 | 并入本方案的位置 |
|------|----|------------------|
| 测试对话 | L2 置信度不足导致「测试20」「我也不知道」等仅靠兜底才回 | **触发与意图**：补 L1 关键词（如「介绍」「介绍一下」入 channel_related 或 general_business），或微调 L2 阈值；与「4. 技能」的冷却精细化独立。 |
| 测试对话 | 重复回复（「都什么通道」「详细介绍一下」两段几乎相同） | **上下文与重复**：与「1. 会话上下文」结合——在传入 AI 的历史中携带「上一条机器人回复摘要」或至少「上一条是否为通道/额度说明」标志，并在 prompt 中再次强调「若上条已是通道/额度说明则简短确认」；可选在 channel_info 内对上条回复做摘要判断，若为额度说明则直接返回简短确认不调 AI。 |
| 测试对话 | 监控与排查（意图修正、额度分支命中） | 已并入 **5. 监控与可观测性**：埋点含意图修正次数、额度分支命中、分条发送条数等。 |

## 三、综合优化方案实施计划（按优先级，无代码）

### 第一优先级（高收益、低成本）

1. **错误处理与降级**  
   - claude-4.6-oups-high 调用：超时/失败重试 1 次（指数退避），再失败返回固定提示并打日志。  
   - 订单识图失败：回复引导用户发订单号或清晰截图。  
   - 可选：简单熔断（最近 N 次失败率超阈值则暂停调 AI，仅规则回复）。

2. **关键信息锚定**  
   - 在组装发往 claude-4.6-oups-high 的上下文时，从会话中抽取订单号、通道名、已给额度结论，以固定格式拼到 prompt 顶部，确保不被截断丢失。

3. **多语言术语一致**  
   - 新增或扩展术语表配置（YAML），在 system_prompt 中规定 Pay in、Pay out、EP、JC、代收、代付 等必须表述。  
   - 回复后处理：按术语表对回复做规范化替换（完整词匹配）。

4. **监控与可观测**  
   - 为每条消息串联 request_id/message_id，在关键步骤打日志携带该 id。  
   - 增加指标：各技能/意图命中、claude-4.6-oups-high 调用次数与耗时、冷却命中、分条发送条数；可选额度规则命中、意图修正次数。  
   - 简单告警：错误率突增或平均响应 > 5s 时通过 Telegram 通知。

### 第二优先级（体验与稳定性）

5. **情绪感知传入 prompt**  
   - 将现有情绪分析结果或「着急/不满」等粗判写入传给 claude-4.6-oups-high 的提示，引导语气。

6. **分条发送增强**  
   - 拆分时避免在数字、EP/JC、Pay in 等专有名词中间截断；优先在句号/段边界切。

7. **触发与意图微调**  
   - 将「介绍」「介绍一下」等补入 L1 关键词（channel_related 或 general_business），或微调 L2 置信度阈值，减少仅靠 AI 上下文/追问才回的情况。

8. **冷却精细化**  
   - 在配置中支持按意图/技能覆盖冷却（如 order_query 3s、small_talk 10s），与现有全局/按用户/按内容兼容。

### 第三优先级（二期或可选）

9. **会话窗口与摘要**  
   - 在观测到「长对话截断导致答非所问」后，再引入历史条数限制 + 超阈值时对早期消息做摘要压缩；先明确当前传入 claude-4.6-oups-high 的历史条数与 token 占用。

10. **多语言模板与语言强制**  
    - 额度等规则模板支持多语言版本（如英文/阿语），并在检测到用户语言后选用对应模板；claude-4.6-oups-high 请求中指定输出语言。

11. **安全与频率**  
    - 发送侧全局最小间隔 + 随机抖动；日志脱敏（消息内容截断）；密钥与会话不落库、权限最小化。

12. **技能插件化与热加载**  
    - 在确有「频繁新增技能」需求时再设计统一 Skill 接口与配置驱动加载；当前可只做配置化冷却。

| 3 | **多语言术语一致** | 配置 `ai.terminology`（错误/变体→正确表述）；回复发送前按术语表做完整短语替换（key 按长度降序避免短串破坏长串）。system_prompt 中已有「通道/额度回复」术语约束。 | `config/config.yaml`（`ai.terminology`）、`src/client/telegram_client.py`（`_apply_terminology`，发送前对 `reply_final` 应用） |

| 5 | **订单识图失败引导** | 订单查询技能在无订单号且 AI 无回复时，返回固定引导：「暂未看到订单信息，请发订单号或清晰付款截图，我帮您查。」覆盖识图失败场景。 | `src/skills/skill_manager.py`（`OrderQuerySkill.execute` 后备文案） |

**在方案基础上的再优化**  
- 关键信息锚定未做「订单号/通道名抽取列表」的独立存储，仅用「上条是否已给额度/识图是否存在」两条布尔型提示，减少实现复杂度且满足「不重复贴长段」与「有识图则按识图答」的需求。  
- 术语后处理按 key 长度降序替换，避免「代付」先于「代代付」替换导致残留。  
- request_id 同时带入 `chat_id`/`chat_title` 到 `user_context`，便于技能（如额度分支）直接使用，无需再传一层。
