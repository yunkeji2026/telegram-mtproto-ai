# 客服额度回复规则 — 整合与实施方案（仅方案，无代码）

本文档说明如何将《客服额度回复规则最终方案》融入现有 Telegram 客服系统的回复与触发规则，并回应方案中的修改需求。**先给出方案与结论，不涉及具体代码实现。**

---

## 一、现状简要对照

| 项目 | 现有实现 | 方案要求 |
|------|----------|----------|
| **触发** | L1 已有「限额」「EP」「JZ」等；意图里 channel_info 关键词为「通道、渠道、channel、有哪些通道、通道状态」 | 增加「额度」「能过多少」「最大多少」「多少额度」「JC」「ja」「jz」「EasyPaisa」「JazzCash」等；精确/包含匹配、不区分大小写 |
| **通道额度** | ChannelInfoSkill 写死 EP/JC 均为 100–100,000，无客户分层 | 普通 EP 100–20,000、特殊 EP 100–100,000、黑名单定制；JC 普通/特殊均为 100–100,000 |
| **客户分层** | 无 | 按**群名**区分：普通 / 特殊（高额度 EP）/ 黑名单（定制回复） |
| **群名** | 未传入 Skill | 需在 context 中传入 **chat_title**（群组标题）供群名识别 |
| **回复格式** | 自由文本 + 简单列表 | 固定模板：「当前额度如下：{通道}：{最低}–{最高}」+ 补充说明；未指定通道时引导选 EP/JC |
| **配置** | 额度与通道写死在代码 | 规则可配置（通道代号、额度区间、特殊群名单、黑名单群及定制文案） |

---

## 二、整合思路总览

1. **触发与意图**：在现有 L1 + 意图体系上**补充额度相关触发词**，并让「纯额度问询」能命中通道/额度逻辑。  
2. **客户分层**：用**群名（chat_title）**查表，区分普通 / 特殊 / 黑名单。  
3. **回复内容**：在现有 **ChannelInfoSkill** 内增加「额度子分支」：先判断是否在问额度 → 解析 EP/JC/未指定 → 按群名查客户类型 → 用**配置化模板**生成回复；非额度类通道问题仍走现有 AI/通用通道信息。  
4. **配置与热更新**：额度与群名单**独立配置**（如 `config/quota_rules.yaml`），支持**热更新**（改配置不重启），并回答「规则存哪、怎么同步、要不要重启」。

---

## 三、具体整合方案

### 3.1 触发层（L1 + 意图）

- **trigger_rules.yaml（L1）**  
  - 在 `high_frequency_keywords.categories.channel_related` 及 `all_keywords` 中**补充**：  
    - 直接触发：`EP`、`JC`、`ja`、`jz`、`EasyPaisa`、`JazzCash`（已有 EP、JZ 的保留并补全 JC/ja/jz 等）。  
    - 间接触发：`额度`、`能过多少`、`最大多少`、`多少额度`、`额度多少`、以及组合如 `EP额度`、`JC额度` 等（包含匹配、不区分大小写）。  
  - 不改变现有 L1 结构，只做关键词扩展，保证「问额度」的消息能进入回复流水线。

- **config.yaml（意图）**  
  - 在 `intent.keywords.channel_info` 中**增加**：`额度`、`限额`、`EP`、`JC`、`EasyPaisa`、`JazzCash` 等，使「只问额度」的消息也被识别为通道/额度类，走 ChannelInfoSkill。  
  - 不新增单独意图（如 quota_query），避免意图膨胀；在 **ChannelInfoSkill 内部**再区分「是否纯额度问询」，是则走额度规则引擎，否则走现有「通道信息」AI/模板。

- **匹配优先级（在技能内实现）**  
  - 在额度子分支内按方案约定顺序即可：  
    1. 精确匹配通道代号（EP/ep/EasyPaisa、JC/jc/ja/jz/JazzCash）  
    2. 包含通道代号的组合词（如 EP额度、JC额度）  
    3. 仅通用额度词（未指定通道）→ 回复「请问您需要查询哪个通道的额度？目前支持 EP、JC」

---

### 3.2 群名识别与 context 传递

- **问题**：当前 `process_message(context)` 只有 `chat_id`、`context_analysis`、`image_ocr_text`、`recent_bot_messages`，**没有群组标题**，无法做「按群名识别特殊/黑名单」。  

- **做法**：  
  - 在 `_process_message_async` 里组装传入 `skill_manager.process_message` 的 `context` 时，**增加 `chat_title`**：  
    - 群组：`message.chat.title`（无则用空串或 `"Unknown"`）；  
    - 私聊：空串或固定如 `"私聊"`。  
  - 这样 ChannelInfoSkill（额度子分支）即可用 `context.get('chat_title')` 去查「特殊客户群名列表」「黑名单群名列表」，决定用哪一套额度或定制回复。

---

### 3.3 额度规则配置（存储与结构）

- **存储位置**：建议**独立文件** `config/quota_rules.yaml`，与 `trigger_rules.yaml`、`config.yaml` 并列，便于运营只改额度与群名单，不动触发与主配置。  
- **结构**：采用《客服额度回复规则最终方案》5.1 的 YAML 思路，例如：  
  - **channels**：`ep` / `jc` 的 `names`（别名列表）、`default_range`（普通客户）、`special_range`（仅 EP 有，特殊客户）。  
  - **special_groups**：群名列表，匹配到的群视为「特殊客户」（EP 用 special_range）。  
  - **blacklist_groups**：群名 → 每通道的定制回复或额度（如 `ep: "当前EP渠道受限…"`，`jc: "50-5000"` 或一段话）。  
- **与代码关系**：  
  - 额度子分支**只读**该配置（或从 config_manager 统一加载），不写死额度与文案；  
  - 新增通道时只需在配置中增加一项，无需改意图或技能注册。

---

### 3.4 ChannelInfoSkill 内「额度子分支」逻辑（流程）

1. **是否额度问询**  
   - 用配置的「额度/限额/能过多少/EP/JC…」等关键词（或复用 L1/意图里的词）对当前 `text` 做一次轻量判断；  
   - 若**不是**额度问询（例如只问「通道状态」「有哪些通道」），则走**现有逻辑**：AI 生成 + 失败时用现有 channel_info 兜底。

2. **若为额度问询**  
   - **解析问的是哪个通道**：精确或包含匹配 EP / JC（含别名），得到「EP / JC / 未指定」；  
   - **查客户类型**：用 `context.get('chat_title')` 查 `special_groups`、`blacklist_groups` → 普通 / 特殊 / 黑名单；  
   - **选模板与参数**：  
     - 黑名单：用 blacklist_groups 中该群对该通道的定制回复（或固定模板+配置的额度）；  
     - 特殊：EP 用 special_range，JC 同普通；  
     - 普通：EP 用 ep default_range（100–20,000），JC 用 jc default_range（100–100,000）；  
     - 未指定通道：返回「请问您需要查询哪个通道的额度？目前支持的通道有：EP, JC」。  
   - **拼装回复**：按方案 3.1/3.2 的固定模板填空（当前额度如下：{通道}：{最低}–{最高}；如有特殊情况…）；  
   - **不再调用 AI**，保证口径统一、可审计。

3. **优先级**  
   - 额度子分支在 ChannelInfoSkill 内**优先**于「通用通道信息 + AI」：先判断额度问询，是则直接走规则+配置，否则再走现有 AI/模板。

---

### 3.5 模板与方案的一致性

- 完全采用《客服额度回复规则最终方案》**三、回复模板**：  
  - 标准格式：「当前额度如下：{通道名称}：{最低额度} – {最高额度}」+「如有特殊情况需补充说明：当前通道额度根据实时情况会有调整，请以实际提交为准。」  
  - 普通/特殊/黑名单示例按方案 3.2 的表格来；  
  - 未指定通道按方案「通用回复」：引导选择 EP/JC。  
- 模板文案可放在 `quota_rules.yaml` 或 `config.yaml` 的 `templates.quota_*` 下，由额度子分支读取，便于运营改话术。

---

## 四、方案中「修改需求」的回应

### 4.1 识别机制（方案 4.1）

- **黑名单 / 特殊客户**：均通过**群名识别**；群名列表由运营提供，写在 `quota_rules.yaml` 的 `special_groups`、`blacklist_groups`。  
- **普通客户**：未出现在上述两个列表中的群（以及私聊）均视为普通客户。

### 4.2 数据同步方式（方案 4.2）

- **规则存储**：`config/quota_rules.yaml`（或项目内约定的配置文件路径）。  
- **格式**：YAML；结构见 3.3。  
- **同步方式**：  
  - **已实现热更新**：`ConfigManager.get_quota_rules()` 按文件 mtime 判断是否重载，修改 `quota_rules.yaml` 后**无需重启进程**，下次命中额度问询时自动使用新规则。  
  - 后续若需要运营界面，可再增加 API 或数据库，最终仍建议**落盘到 YAML**，便于版本管理与回滚。

### 4.3 系统集成（方案 4.3）

- **修改规则后是否需要重启**：建议实现**热更新**，则**不需要重启**；若未实现热更新，则需要重启。  
- **重启对象**：当前客服系统即由 `main.py` 启动的 Python 进程（无单独服务名时，可说明为「Telegram 客服主进程」）。  
- **其他通道**：架构上 `channels` 为配置列表/字典，后续新增通道只需在 `quota_rules.yaml` 的 `channels` 与 `blacklist_groups` 中增加对应配置即可，无需改代码逻辑（仅需在「未指定通道」的引导语中补上新通道名）。

### 4.4 更新机制（方案 5.2）

- **建议**：支持热更新；配置文件变更后自动或手动触发重载；若后续有运营后台，可提供 API 更新群名列表并写回配置文件。  

---

## 五、与现有回复/触发流程的衔接

- **四层触发**：不改变 L1→L2→L3→L4 的顺序与逻辑，仅扩展 L1 与意图关键词，使「额度」类消息更稳定地进入回复流程。  
- **Skill 入口**：仍由意图识别为 `channel_info`，进入 ChannelInfoSkill；在 ChannelInfoSkill 内部**先做额度分支判断**，再决定走「额度规则引擎」还是「原有通道信息 + AI」。  
- **情绪增强 / 上下文**：额度子分支若返回的是固定模板文案，仍可和现在一样经过情绪增强器（若需要可对额度回复做白名单，跳过语气修改，避免数字被改）；上下文记录不变。

---

## 六、测试建议（对应方案六）

- **功能**：  
  - 普通客户问 EP/JC 额度 → 分别得到 100–20,000 与 100–100,000 及标准话术；  
  - 特殊客户群问 EP → 100–100,000；  
  - 黑名单群问 EP/JC → 定制回复或方案示例中的额度；  
  - 只问「额度」未指定通道 → 引导选择 EP/JC；  
  - 大小写（EP/ep/Ep、JC/jc）均能识别。  
- **边界**：  
  - 群名不在任何列表 → 普通；  
  - 私聊（无群名）→ 普通；  
  - 不相关消息不触发额度逻辑，仍走原有 channel_info 或其它意图。

---

## 七、实施顺序建议

1. **配置**：新增 `config/quota_rules.yaml`，按方案 5.1 写好 channels、special_groups、blacklist_groups（黑名单可先空或示例 1 个群）。  
2. **context**：在 `_process_message_async` 中向 `process_message` 的 context 增加 `chat_title`。  
3. **触发与意图**：在 trigger_rules 与 config 的 channel_info 意图中补充额度相关关键词。  
4. **ChannelInfoSkill**：实现额度子分支（是否额度问询 → 解析通道 → 群名查表 → 选模板 → 拼装回复）；其余逻辑保持不变。  
5. **热更新**（可选）：为 quota_rules 增加加载入口与重载逻辑（文件监听或接口），并注明在文档中。  
6. **测试**：按第六节与方案 6.1/6.2 做回归与边界测试。

---

## 八、小结

- **触发**：L1 + 意图补充「额度/限额/能过多少/EP/JC/EasyPaisa/JazzCash」等词，不新增意图。  
- **分层**：用 **chat_title** 查配置的 special_groups / blacklist_groups，区分普通/特殊/黑名单。  
- **回复**：在 **ChannelInfoSkill** 内增加「额度子分支」，按通道 + 客户类型从 **quota_rules.yaml** 取数和模板，输出方案规定的固定格式。  
- **配置**：独立 `quota_rules.yaml`，支持热更新则无需重启，并回答方案中「规则存哪、怎么同步、要不要重启、其他通道」等确认问题。  

按上述方案实施后，即可在现有回复规则体系中完整落地《客服额度回复规则最终方案》，并满足其中技术实现与修改需求说明。
