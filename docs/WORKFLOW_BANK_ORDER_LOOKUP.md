# 工作流：识图 → 银行/账单唯一性识别 → 查单依据

## 目标

识图后先判断**这是哪个银行或哪种账单**，并提取**唯一性识别依据**；在与客户聊天时，把该依据作为**查单依据**，用于后续查单、对账、回复。

---

## 1. 流程概览

```
用户发图（付款凭证/账单/订单截图）
    ↓
识图（Vision / OCR）→ 得到图中文字与结构
    ↓
识别：银行或账单类型 + 唯一识别依据
    ↓
与客户对话：用「唯一识别依据」作为查单依据（查单、确认、回复）
```

- **先知道是什么**：银行/支付渠道或账单类型（如 easypaisa、某银行转账、某平台订单）。
- **再定唯一依据**：该渠道下能唯一标识这笔交易/订单的字段（见下表）。
- **聊天时用这个依据**：客户问「查单」「看到订单了吗」时，用已识别的唯一依据去查或回复，不编造。

---

## 2. 各银行/支付渠道的唯一帐号识别性

以下为常见渠道及其**唯一性识别依据**（识图时优先提取，并作为查单依据）。

| 渠道类型 | 示例（银行/产品名） | 唯一识别依据 | 说明 |
|----------|----------------------|--------------|------|
| 巴基斯坦钱包 | EasyPaisa、JazzCash | **Transaction ID**（交易号）、或 单号/Reference | 凭证上通常有 ID#xxxxx、Transaction ID |
| 银行转账/回单 | 各商业银行 | **交易流水号/参考号**、**转账时间+金额+付款方** 组合 | 不同银行字段名不一，识图时提取「唯一编号」或「日期+金额+对方户名」 |
| 平台订单 | 电商/内部订单系统 | **订单号**（Order No / 订单号） | 如 #20231215-8742、纯数字长单号 |
| 支付链接/收据 | 第三方支付 | **Payment ID / Reference / 商户订单号** | 收据或成功页上的唯一 ID |
| 发票/账单 | 各类账单 | **发票号/账单号**、**开票日期+金额** | 无单号时用日期+金额+对方作为组合依据 |

**识图输出要求**（见下节配置）：

- 必须包含：**① 银行/账单类型** ② **唯一识别依据**（具体字段名+取值）。
- 可选：金额、币种、时间、付款方/收款方，用于与客户核对。

---

## 3. 与客户聊天时的查单依据用法

1. **用户发图后**
   - 识图结果中若已包含「银行/账单类型」和「唯一识别依据」，在回复中可简要确认：「您发的是 [渠道名] 的凭证对吧？交易号/单号是 xxx，我会按这个帮您查。」
   - 若图中无法确定唯一依据，回复请用户补充单号或说明「需要您提供交易号/订单号才能查」。

2. **用户问「查单」「看到订单了吗」**
   - 若本轮或上文已通过识图得到**唯一识别依据**，则用该依据回答：可说明「按您刚才发的凭证，单号是 xxx，查询结果是…」或把该依据带入后续查单接口。
   - 若没有唯一依据（未发图或图里没有），则回复请用户发凭证或订单号，**不得编造**单号或订单内容。

3. **查单接口/后台**
   - 若有内部查单 API，请求参数应使用识图得到的**唯一识别依据**（如 Transaction ID、订单号、参考号等），保证「识图 → 查单依据」一致。

---

## 4. 配置与实现要点

- **Vision 提示语**（`config/config.yaml` → `vision.prompt`）：要求模型在描述图片时**先判断银行/账单类型，再提取唯一识别依据**；默认逻辑见 `src/vision_client.py` 中 `default_prompt`。
- **系统提示词**（`config/config.yaml` → `ai.system_prompt`）：约定回复时仅根据识图结果中的真实信息作答，**查单依据 = 识图得到的唯一识别依据**，不编造；与客户对话时明确说出渠道与单号/交易号。
- **上下文**：识图结果（Vision/OCR）会以「用户刚发的图片/截图内容（Vision/OCR）」传入 claude-4.6-oups-high 的对话上下文，AI 从中读取「银行/账单类型」和「唯一识别依据」用于回复与查单。
