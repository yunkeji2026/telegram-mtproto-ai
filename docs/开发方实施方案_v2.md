## 二、实际运行发现的问题与优化建议

> 以下为原指南中未涉及的、从生产环境运行中发现的关键问题和优化方向。

### 2.1 已修复的生产问题（供开发方了解）

| 问题 | 根因 | 已实施的修复 |
|---|---|---|
| Bot 处理 gxp_notify_bot 的回复消息导致命令循环 | 缺少 Bot 账号过滤 | 添加 `is_bot` 检查 + `no_reply_sender_usernames` 配置 |
| 重启后处理历史消息导致循环 | 无启动时间过滤 | 添加 `_boot_timestamp` 对比消息时间 |
| Bot 处理自身发送的消息 | 无自身 ID 检查 | 添加 `from_user.id == self.user_info.id` + `outgoing` 检查 |
| 数字选项回复 (1/2/3) 被冷却拦截 | 全局 5s 冷却 | GXP 数字选项豁免冷却 |
| "查订单+单号" 错误走 what 流程 | `is_ambiguous_what` 逻辑缺陷 | 有单号时强制走 intent 流程 |
| 12 位数字被误判为 UTR | `_extract_utr` 范围过宽 | 收紧为仅当文本含 "utr" 关键词时才提取 |
| 日志停止写入但进程存活 | Windows 文件编码/缓冲区 | 添加 UTF-8 编码配置 |

### 2.2 新增架构优化建议（开发指南中未涉及）

#### A. 消息防重放机制（建议 P0 加入）

**问题**：Pyrogram 用户账号重启后可能重新接收已读消息，导致重复处理。  
**现状**：已用 `_boot_timestamp` 时间戳做基础过滤。  
**建议增强**：
- 维护已处理消息 ID 的滑动窗口（最近 1000 条），避免任何重复处理
- 持久化到本地文件，重启后不丢失

```
新增文件：src/utils/message_dedup.py
类：MessageDeduplicator
方法：is_duplicate(message_id) -> bool, mark_processed(message_id)
存储：SQLite 或 JSON 文件（按日轮转）
```

#### B. GXP 命令结果追踪（建议 P1 加入）

**问题**：Bot 代发命令后只说"请查看上方机器人回复"，不主动追踪 gxp_notify_bot 的回复结果。  
**建议**：
- 代发命令后启动短时监听（5-10s），等待 gxp_notify_bot 的回复
- 收到结果后主动转述给用户（如："查询结果：订单 XXX 状态为处理中"）
- 超时无回复则提示"机器人暂未回复，请稍后重试"

```
新增方法：GxpCommandSkill._wait_for_gxp_response(chat_id, timeout=8)
依赖：telegram_client 提供 _collect_bot_messages(chat_id, username, since_ts, timeout)
```

#### C. 配置热更新覆盖范围扩大

**现状**：仅 `quota_rules.yaml` 支持 mtime 热更新。  
**建议**：将以下也加入热更新：
- `trigger_rules.yaml`（调整触发关键词不用重启）
- `templates.yaml`（已标注支持热更新，需确认 ConfigManager 实际实现）
- `config.yaml` 的部分安全项（如 `no_reply_sender_usernames`、`cooldown` 参数）

#### D. telegram_client.py 拆分（P3 但影响大）

**现状**：1690+ 行，包含消息处理、触发决策、发送逻辑、OCR、语音等。  
**建议拆分方案**：

```
src/client/
├── telegram_client.py          # 主入口，仅做消息分发（~300 行）
├── message_preprocessor.py     # 文本归一化、语音转写、图片 OCR
├── reply_decision.py           # 四层触发 + 回复链 + 追问判断
├── message_sender.py           # 分条发送、延迟、术语替换、日志脱敏
└── group_context.py            # 群消息历史拉取、Bot 消息收集
```

## 三、开发任务详细设计

### 阶段一（2 天）：核心配置管理功能

#### 任务 1：话术模板管理

**目标**：通过对话命令实时更新 Bot 的回复话术。

**入口**：`EnhancedQuotaConfigSkill._handle_template_management(text)`

**命令规格**：

| 命令 | 示例 | 说明 |
|---|---|---|
| `更新话术 <模板名> <内容>` | `更新话术 greeting 嗨，我是客服小C～` | 替换指定模板的第一条 |
| `添加话术 <模板名> <内容>` | `添加话术 greeting 你好呀，需要帮忙吗？` | 在模板列表末尾追加 |
| `删除话术 <模板名> <序号>` | `删除话术 greeting 2` | 删除第 N 条 |
| `查看话术 <模板名>` | `查看话术 greeting` | 显示该模板所有条目 |
| `列出话术` | `列出话术` | 显示所有模板名称及条目数 |

**实现要点**：
1. 读取 `config/templates.yaml`（通过 `ConfigManager.get_dynamic_templates_config()`）
2. 模板名必须在已有 key 中存在（`greeting`, `order_query`, `gxp_ask_intent` 等）
3. 内容不能为空，长度限制 10-500 字符
4. 写入后调用 `invalidate_templates_cache()` 使缓存失效
5. 返回操作结果（含更新前/后内容预览）

**测试用例**：
```
输入: "更新话术 greeting 嗨！有什么需要帮忙的～"
预期: "✅ 已更新话术 greeting 第 1 条\n旧: 在的～我是 Camille...\n新: 嗨！有什么需要帮忙的～"

输入: "查看话术 greeting"
预期: "📋 greeting 话术模板（共 3 条）\n1. 在的～我是 Camille...\n2. 嗨，在呢！...\n3. 你好呀..."

输入: "列出话术"
预期: "📋 话术模板列表\n- greeting (3 条)\n- order_query (2 条)\n- gxp_ask_intent (1 条)\n..."

输入: "更新话术 不存在的模板 内容"
预期: "❌ 模板 '不存在的模板' 不存在。可用模板：greeting, order_query, ..."
```

#### 任务 2：汇率/通道配置管理

**目标**：通过对话命令实时修改通道费率、额度、状态。

**入口**：`EnhancedQuotaConfigSkill._handle_exchange_rate_management(text)`

**命令规格**：

| 命令 | 示例 | 说明 |
|---|---|---|
| `更新汇率 <通道> <费率>` | `更新汇率 EP 0.45%` | 修改通道费率 |
| `更新额度 <通道> <范围>` | `更新额度 EP 200-50000` | 修改通道额度范围 |
| `启用通道 <通道>` | `启用通道 JC` | 设置通道状态为"正常" |
| `禁用通道 <通道>` | `禁用通道 JC` | 设置通道状态为"暂停" |
| `查看汇率 <通道>` | `查看汇率 EP` | 显示指定通道完整配置 |
| `列出汇率` / `列出通道` | `列出通道` | 显示所有通道状态摘要 |

**实现要点**：
1. 读取 `config/exchange_rates.yaml`（通过 `ConfigManager.get_exchange_rates_config()`）
2. 通道名支持别名匹配（EP/EasyPaisa/ep 均指向 `ep` 配置块）
3. 费率格式验证：数字或百分比（如 `0.5%`、`7.15`）
4. 额度格式验证：`数字-数字`（如 `100-20000`）
5. 写入后调用 `invalidate_exchange_rates_cache()`
6. 自动记录 `last_updated` 时间戳

**测试用例**：
```
输入: "更新汇率 EP 0.45%"
预期: "✅ EP通道 费率已更新\n旧: 0.5%\n新: 0.45%"

输入: "禁用通道 JC"
预期: "✅ JC通道 状态已更新为 暂停"

输入: "列出通道"
预期: "📊 通道状态\n- EP通道: 正常 | 费率 0.45% | 额度 100-20,000\n- JC通道: 正常 | 费率 0.6% | 额度 100-100,000"

输入: "更新汇率 XX 0.5%"
预期: "❌ 通道 'XX' 不存在。可用通道：EP, JC"
```

#### 任务 4：自然语言管理命令解析

**新建文件**：`src/utils/nlp_admin_parser.py`

**核心思路**：先用正则 + 关键词匹配，置信度不够时调用 AI API 辅助理解。

**支持的自然语言场景**：

| 用户说 | 解析结果 |
|---|---|
| "把问候语改得亲切一点" | `{action: "update_template", target: "greeting", style: "亲切"}` → 调用 AI 生成新话术 |
| "JC 汇率调整到 7.2" | `{action: "update_rate", channel: "jc", value: "7.2"}` |
| "看看当前的话术配置" | `{action: "view_config", scope: "templates"}` |
| "EP 通道暂停一下" | `{action: "disable_channel", channel: "ep"}` |

**实现策略**：
1. **第一层**：正则匹配标准命令格式（`更新话术 XX YY`）→ 直接执行
2. **第二层**：关键词 + 规则推断意图（"亲切一点" → 修改话术风格）→ AI 生成内容
3. **第三层**：AI API 语义理解（完全自由文本）→ 提取意图和参数
4. **回退**：无法理解时返回帮助信息

## 四、架构优化实施方案（阶段四，P3）

### 4.1 统一配置接口

**现状**：混用 `config.get('telegram', {})` 和 `config.get_telegram_config()`。  
**目标**：所有模块通过 `ConfigManager` 的 getter 方法访问配置。

```python
# 统一为：
config.get_telegram_config()    # Telegram 相关
config.get_ai_config()          # AI 模型相关
config.get_trigger_config()     # 触发规则
config.get_skills_config()      # 技能配置
config.get_templates()          # 话术模板
config.get_exchange_rates()     # 汇率通道
config.get_quota_rules()        # 额度规则
```

### 4.2 意图识别优化

**现状**：意图匹配依赖配置中 key 的出现顺序，先匹配先命中。  
**建议**：
- 改为「优先级 + 匹配」模型：每个意图有显式 priority 值
- 匹配多个时取 priority 最高的
- 新增 `default` 意图，避免短句被误判为 small_talk

### 4.3 冷却文档明确化

**补充说明**（加入运维文档）：
- `by_intent` 冷却是"距该用户上次**任意回复**的间隔"，不是按意图分别计时
- GXP 数字选项回复（1-5）已豁免所有冷却
- 订单号跟进已豁免用户冷却（仍走全局和内容冷却）

## 十、P4 阶段实施记录（已完成）

### P4-1 GXP 命令结果超时提醒
- **文件**: `src/client/telegram_client.py`
- **实现**: 发送 GXP 命令后自动启动 `asyncio` 延迟协程，60s 内未收到 `gxp_notify_bot` 回复则主动发超时提醒
- **优化**: 用协程代替定时器/线程，零额外资源消耗；通过 `original_ts` 精确匹配避免误报

### P4-2 配置变更通知
- **文件**: `src/utils/config_manager.py`, `src/client/telegram_client.py`, `config/config.yaml`
- **实现**: `ConfigManager` 新增 `on_reload()` 回调注册机制；`TelegramClient` 在启动时注册回调，热重载成功后向 `admin_chat_id` 发送通知
- **优化**: 回调模式避免 ConfigManager 反向依赖 TelegramClient，保持架构清洁

### P4-3 单元测试
- **文件**: `tests/test_config_manager.py`, `tests/test_enhanced_quota_config.py`, `tests/test_audit_store.py`
- **覆盖**: ConfigManager（写入/缓存/热重载/保护字段/回调）、EnhancedQuotaConfigSkill（权限/模板CRUD/通道管理/审计/回滚）、AuditStore（SQLite 基本操作/JSONL 迁移/防重复迁移）
- **结果**: 36 项测试全部通过

### P4-4 审计日志迁移 SQLite
- **文件**: `src/utils/audit_store.py`（新建）, `src/skills/skill_manager.py`
- **实现**: 独立 `AuditStore` 类，SQLite 后端，支持按操作/用户/时间筛选；首次启动自动迁移现有 JSONL 数据并备份
- **优化**: 职责分离（不嵌入 SkillManager）；`check_same_thread=False` 适配多协程；JSONL 迁移后自动重命名为 `.bak` 防止重复导入

- **实现**: `PriorityMessageQueue` 基于 `asyncio.PriorityQueue` 实现。4 级优先级：CRITICAL(0) > HIGH(10) > NORMAL(50) > LOW(90)。GXP/配置命令为 HIGH，闲聊为 LOW。
- **背压控制**: 队列利用率超阈值（默认 80%）时自动丢弃 LOW 优先级消息；队列满时高优先级消息可踢出旧消息
- **优化（超原方案）**: 不引入 Redis/RabbitMQ（零外部依赖）；意图→优先级自动映射（`INTENT_PRIORITY` 字典）；统计接口暴露丢弃率/背压次数

### 测试结果
- **97 项单元测试全部通过**（原 76 + P8 新增 21 项）
- 新增测试覆盖: TokenBucket+RateLimiter（6项）、QualityTracker（5项）、ConfigMigrator（3项）、ChannelHealth（3项）、PriorityQueue（4项）

## 十七、S2 阶段实施记录（已完成）— 智能分析 + Auto-Pilot

### S2-1 质量评分与智能诊断
- **文件**: `src/utils/strategy_advisor.py`（新建）
- **实现**: `compute_quality_score()` 综合评分 0-100（响应速度 20% + 一次解决率 35% + 同意图追问惩罚 30% + API 效率 15%）。`analyze()` 检测异常生成 critical/warn/info 级别告警和建议
- **优化**: 纯被动指标，不需要用户反馈或内联键盘

### S2-2 Auto-Pilot 自动策略降级
- **文件**: `src/utils/strategy_advisor.py`, `src/skills/skill_manager.py`, `config/config.yaml`
- **实现**: 消息驱动（每 100 条检查一次），`generate_auto_actions()` 检测持续低效策略（评分 < 30 + 样本 ≥ 10 + 与最优差距 > 15 分），自动重映射意图到更优策略
- **安全**: 仅重映射意图不禁用策略；所有操作记审计日志；默认关闭需手动启用

### S2-3 会话级追踪
- **文件**: `src/utils/strategy_tracker.py`
- **实现**: `_resolve_session_id` 动态分配会话 ID — 30 分钟内同一 user+chat 的消息复用会话。`session_summary()` / `session_stats()` 提供会话级聚合（会话数、平均轮次、解决率）
- **优化**: 增量 schema 迁移，session_id 列不影响旧数据

### S2-4 参数微调建议 + 一键应用
- **文件**: `src/utils/strategy_advisor.py`, `src/web/admin.py`
- **实现**: `suggest_param_adjustments()` 规则引擎根据指标异常生成具体参数调整建议（如 context_rounds +2、max_tokens -128）。`/api/apply-param-suggestion` POST 一键应用到 YAML
- **优化**: 规则引擎比贝叶斯优化更实用（数据量小时更稳定），建议可预览后再应用

### S2-5 数据导出与清理
- **文件**: `src/web/admin.py`
- **实现**: `/api/export-strategy-events?fmt=csv|json` 导出全量策略追踪数据。`/api/data-purge` 清理过期数据（可配置保留天数）并 VACUUM 回收空间

### S2-6 多模型 A/B 对比
- **文件**: `src/ai/claude-4.6-oups-high_client.py`, `src/skills/skill_manager.py`, `src/utils/strategy_tracker.py`, `src/web/admin.py`, `src/web/templates/strategy_analytics.html`
- **实现**: 策略新增 `model` 字段覆盖全局模型。`generate_reply()` 使用 `use_model` 替代 `self.model`。追踪层记录 `model_id` 列。仪表盘新增"模型效果对比"面板（模型卡片 + 模型×策略交叉表）
- **优化**: 零独立系统开销 — 完全复用已有 A/B 分流、追踪、质量评分管道。不配 model 的策略透明使用全局模型

### S2-7 用户分群分析
- **文件**: `src/utils/strategy_tracker.py`, `src/web/admin.py`, `src/web/templates/strategy_analytics.html`
- **实现**: `user_segment_analysis()` 按活跃度分群 heavy(≥10)/moderate(3-9)/light(1-2)。每个分群统计用户数、事件数、加权追问率、Top 5 偏好策略
