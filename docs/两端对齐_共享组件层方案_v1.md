# 程序端 / 网页端 对齐 — 共享组件层方案 v1

> 决策结论(2026-06-09):**策略 B(共享组件层 + 渐进功能对齐)** / **双向各取所长** / **双主题共享同一 tokens**。
> 本文为可执行开发文档,落地代码以本文为准;历史分析文档以代码为准。

## 0. 背景:两端是"同一后端大脑的两个头"

| 维度 | 程序端(Electron) | 网页端(`/workspace`) |
|---|---|---|
| 本质 | 内嵌真·官方网页(Telegram/WhatsApp Web)+ 注入脚本抓 DOM + 右侧 `#copilot` | 自研坐席工作台,从后端 `InboxStore` 读消息 |
| 消息源 | 浏览器实时 DOM(`tg-inject.js`) | 后端库(RPA/协议适配器灌入,已清洗) |
| 后端 | **同一个 FastAPI**(经 `main.js` IPC 调 `/api/...`) | 同一个 FastAPI(直接渲染) |

两端共享同一后端、同一 `chat_key`(P1 同步桥已落库)。功能对齐 ≈ 前端接线 + UI 统一,后端基本复用。

## 1. 前置探查结论 — `chat_key ↔ conversation_id`(已验证)

`conversation_id` 是**确定性公式**(`src/inbox/normalizer.py::conv_id`):

```
conversation_id = f"{platform}:{account_id}:{chat_key}"
```

- 是 `conversations` 表主键;`chat_key` 为单独列,无 `(platform,account_id,chat_key)` 唯一索引;查找只按 `conversation_id`。
- 桌面三要素齐全:`platform`(webview id)、`account_id`(`config.json::platforms[].account_id` 或 `${platform}-desktop`)、`chat_key`(inject 的 peer id)。
- **无需新后端桥**:桌面本地拼 `conversation_id` 即可调所有 conv 级端点。
- **权威兜底**:`POST /api/desktop/ingest` 已回传 `conversation_id`,可用于校准,避免 `account_id` 写入/读取不一致导致指向空会话(唯一坑点)。

所有 conv 级端点(均 `src/web/routes/unified_inbox_routes.py`,路径参数 `conversation_id`):
关系阶段 `/api/workspace/conv/{cid}/relationship-stage`、下一步动作 `/next-actions`、剧本 `/script-suggestions`、注解 `/notes`、摘要 `/summarize`。

## 2. 架构:一份 tokens / 一份组件 / 一层数据适配

```
shared/copilot/                 # 单一事实来源(repo 根)
├─ tokens.css                   # 主题无关比例尺(圆角/间距/字号/阴影)
├─ theme-dark.css               # 暗色(桌面)语义色
├─ theme-light.css              # 亮色(网页)语义色
├─ client/copilot-client.js     # 传输抽象 + 两端适配器 + conversationId()
└─ components/                  # 框架无关 Web Components(custom elements)
   └─ cp-rel-stage.js           # 首个:关系阶段(P1 落地)
```

- **网页**:FastAPI 把 `shared/copilot` 挂到 `/copilot/*`(独立前缀,避开 `/static` 匹配顺序),模板用 `/copilot/...` 引用。
- **桌面**:`npm prestart` 跑 `copy-shared.js` 把 `shared/copilot` → `desktop/renderer/shared/copilot`(满足 CSP `'self'` 本地加载),`index.html` 本地引用。

### 2.1 数据适配层 `CopilotClient`
组件只依赖统一接口,不关心传输:

| 方法 | 桌面实现(IPC→main.js) | 网页实现(fetch) |
|---|---|---|
| `getRelStage({conversationId})` | `window.shell.relStage` | `GET /api/workspace/conv/{cid}/relationship-stage` |
| (后续)`getNBA / getScript / getNotes / addNote / getDrafts / resolveDraft ...` | 对应 `desktop:*` IPC | 对应 `/api/workspace/conv/{cid}/*` |

`createCopilotClient()`:`window.shell` 存在 → 桌面适配器,否则 → 网页适配器。
`conversationId(platform, accountId, chatKey)` → 复刻后端公式。

### 2.2 双主题
`tokens.css` 定义比例尺;`theme-dark/light.css` 在 `:root[data-cp-theme="dark|light"]` 下覆盖语义色变量(`--cp-bg/surface/border/text/text-dim/accent/ok/warn/danger`)。CSS 自定义属性穿透 Shadow DOM,组件内 `var(--cp-*)` 即随宿主主题切换。桌面 `<html data-cp-theme="dark">`,网页 `light`。

## 3. 功能对齐清单(双向)

**进桌面(网页有→桌面补)**:关系阶段 ✅(P1)、建议操作 NBA、剧本话题、内部注解@、待审草稿队列。
**进网页(桌面有→网页补)**:人设选择+钉绑、回复语言、发送前风险护栏(`/api/desktop/guard-check` 拟通用化为 `/api/copilot/guard-check` 并保留旧别名)。
**统一视觉**:AI 分析、客户档案。

## 5. 风险与对策
- **`account_id` 不一致** → conversation_id 指向空会话:桌面统一用 `accountIdFor()`(与 ingest 同源),并以 ingest 回传 id 兜底。
- **CSP**:共享资源本地/同源加载,组件用经典 `<script>` 挂 `window.CopilotShared`,不引外链。
- **双消息源**:分析/草拟以"库 + DOM 合并去重"为准(已有 `ensureFullThread`/`loadFullThread`)。
- **防漂移**:右栏业务一律走 `shared/copilot`,禁止在 `renderer.js` / 模板各写一份(本仓库 `project_tasklist_drift.md` 教训)。
