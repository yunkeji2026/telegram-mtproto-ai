# 客服额度与回复 — 日志分析与优化方案

基于《客服额度回复规则最终方案》与已实施的整合优化，对当前 **app.log** 聊天日志做分析，归纳仍需优化的问题，并给出**仅方案级**的优化建议（不涉及具体代码实现）。

---

## 一、已实施的优化回顾

- **额度规则**：`config/quota_rules.yaml`（通道 EP/JC、普通/特殊/黑名单、模板）、热更新。
- **触发与意图**：L1 与 channel_info 意图已补充额度/限额/EP/JC/EasyPaisa/JazzCash 等关键词。
- **群名**：context 中已传入 `chat_title`，供特殊/黑名单识别。
- **ChannelInfoSkill**：额度子分支优先于 AI，命中则按配置返回固定模板，不调用大模型。

---

## 二、日志中暴露的问题

### 2.1 额度问询未走规则，AI 报错数

**现象**：用户明确问「ep现在限额多少」「jz的额度多少」「ep的额度呢？」「ep普通客户限额多少，特殊客户限额多少？jp的限额多少？」时，回复均为 **AI 生成**，且**数字错误**：

- 方案规定：**EP 普通** 100–20,000、**EP 特殊** 100–100,000、**JC** 100–100,000。
- 日志实际回复：**Rs. 500 至 Rs. 50,000**（EP 与 JZ 均被 AI 编造为同一错误区间）。

说明在这些请求上**额度子分支未生效**，仍走了「通道信息 + AI」并产生幻觉。

**可能原因归纳**：

1. **额度规则未加载**：`get_quota_rules()` 因路径或权限问题返回空，`_quota_reply` 直接返回 None，回退到 AI。
2. **意图未进 channel_info**：若意图被识别为 order_query 等（例如上下文主题为 order），则不会进入 ChannelInfoSkill，额度分支无从执行。
3. **额度分支判定过严**：例如仅识别「额度」关键词而未识别「限额」「多少」等，或对「jz」「ep」大小写/别名处理不一致，导致未判为额度问询。

---

### 2.2 额度问询被上下文「带偏」成订单回复

**现象**：用户发「ep现在限额多少」时，回复却是：「这边看到您之前查询的EP通道状态，目前系统显示为『查询中』…付款方是 Camille.，时间是22:52…」（订单/截图相关），与额度完全无关。

说明在**同一群内**，因存在较强订单上下文，AI 被历史对话带偏，把「额度问询」当成订单跟进在处理。

**根因归纳**：  
- 要么未走额度规则（同上），  
- 要么在走 AI 时，传入的**上下文过长/过强**（含近期订单、截图、交易号），模型优先延续「订单」话题而非「额度」话题。

---

### 2.3 误触发：对「讨论额度功能」的回复

**现象**：用户发「他知识库里没有，他没地方判断限额信息」—— 这是在**讨论** bot 能力不足，并非在**询问**限额。L1 因含「限额」触发回复，AI 又扯回订单（交易号 705806645700），造成答非所问且易引发困惑。

说明：**含有「额度/限额」的元讨论、吐槽、内部沟通** 不应被当作「额度咨询」处理，或至少不应再绑定到订单上下文。

---

### 2.4 机器人通知消息触发回复并报错

**现象**：`gxp_notify_bot` 发送的「巴基斯坦通道最新通知」（含「成功」等词）触发 L1，进入回复流程，随后报错：`'utf-16-le' codec can't decode bytes...`。

说明两件事：  
1. **发送者过滤缺失**：来自其他 bot（如通知 bot）的消息本不应触发客服回复。  
2. **编码/解码**：对部分消息（如含特殊字符或多语言的通知）存在解码异常，需在入口做防护或归一化。

---

### 2.5 @本账号 + 示例文案被当成订单回复

**现象**：用户 @wookfaith 并粘贴「回复示例：当前额度如下：EP：100 – 100,000…」时，bot 回复的仍是「已根据您提供的截图信息（交易号 705806645700）…」，完全忽略消息中的额度示例，继续沿用订单上下文。

说明：  
- 当消息**同时包含** @本账号、额度示例文案 和 强订单历史时，若未先走**额度规则**（或未把「当前额度如下」等视为明确额度话术），就会继续被订单语境主导。  
- 多行、带格式的「示例」消息也可能影响意图或文本清洗（如截断、去换行），导致额度分支未命中。

---

### 2.6 订单类回复存在编造

**现象**：用户问「看到发的订单了吗」且**无图或识图失败**时，回复出现「春季新款连衣裙、两双袜子、上海浦东、下午3点42分」等**完全虚构**内容，违反 SOP「严禁编造」。

说明：在**无识图/无机器人通知凭证**时，订单类回复未严格约束为「暂未看到订单信息，请发截图或订单号」，而是任由模型编造细节。

---

### 2.7 情绪增强器未生效

**现象**：日志多次出现「情绪增强器已禁用（配置: emoticons.enabled: false）」。  
因此「有温度」的称呼与语气优化（如少用「您好！客服Camille为您服务」、多用「在的～」等）未生效，部分回复仍偏正式。

说明：当前运行环境中文案与方案/代码不一致，需确认配置与部署是否统一（如 emoticons.enabled）。

## 三、优化方案（按优先级）

### 3.1 P0：确保额度问询必走规则、且数字正确

- **目标**：凡符合「问 EP/JC 额度/限额」的消息，一律走 `quota_rules.yaml` 模板，返回方案规定数字（普通 EP 100–20,000、特殊 EP 100–100,000、JC 100–100,000），**不调用 AI 生成额度数字**。
- **建议措施**：  
  1. **规则加载可观测**：在额度子分支入口打日志（如「额度规则条数」或「未加载到规则」），便于确认线上是否成功加载 `quota_rules.yaml`；若加载为空，应告警或降级到明确提示而非用 AI 编数字。  
  2. **意图与技能路由**：对含有「额度/限额/能过多少」+「EP/JC/jz/ja」等组合的消息，**优先**路由到 channel_info，并在 ChannelInfoSkill 内**最先**执行额度子分支；若文本明显为额度问询，可考虑**跳过或大幅压缩**订单/识图等上下文再调 AI，避免「额度问询仍走 AI 且被订单上下文带偏」。  
  3. **额度问询识别放宽**：在额度子分支内，将「限额」「额度」「多少」「能过多少」与通道代号（含 jz、ja、ep 等）的匹配条件写清并略放宽（如不依赖意图结果，仅看当前句），确保「ep现在限额多少」「jz的额度多少」「ep的额度呢」等都能被判为额度问询并直接返回模板。  
  4. **多问一答**：用户一次问「ep普通/特殊限额、jp限额」时，应**按方案分条回复**（EP 普通、EP 特殊、JC 各一行），而不是只答 EP 或只答一条；可在额度分支内识别「普通/特殊」「ep/jc/jp」并拼接多段模板。

### 3.2 P0：发送者过滤 — 不回复其他 Bot

- **目标**：来自 **bot 账号**（如 `gxp_notify_bot`）或配置中「仅拉取、不回复」的账号的消息，**不进入**四层触发与回复流程，避免误回与编码等问题。
- **建议措施**：在群消息入口（或 `_should_reply_to_group_message` 最前）判断 `message.from_user.is_bot == True` 或 `from_user.username` 在「不回复名单」内，则直接返回不回复，不打 L1/L2。

### 3.3 P1：额度相关「元讨论」不触发或降权

- **目标**：像「他知识库里没有，他没地方判断限额信息」这类**讨论限额功能/知识库**的句子，不当作「用户正在问限额」处理，避免误触发后还扯到订单。
- **建议措施**：  
  - 在 L2 或额度分支前增加**负面规则**：若整句明显是「讨论/吐槽/内部沟通」（如含「知识库」「他没地方」「不会」等），则 either 不触发回复，或 L2 降权；  
  - 或：额度子分支仅当句子为**直接问句**（如「…额度多少」「…限额多少」）且不含「知识库/不会/没有」等元描述时才返回模板，否则交给通用逻辑并尽量不绑定订单上下文。

### 3.4 P1：订单类回复严禁编造

- **目标**：无识图/无机器人通知凭证时，**不得**编造订单详情（商品、金额、地址、时间等），只做「暂未看到订单信息，请发截图或订单号」类引导。
- **建议措施**：在 system_prompt 与订单类技能中再次强调「无凭证即不描述具体订单内容」；若有「订单状态」专用提示，可加入负面示例（如禁止出现具体商品名、地址、虚构时间），并在回复前做轻量校验（若上下文中无识图/无 bot 订单消息，则禁止输出具体订单描述）。

### 3.5 P1：@本账号 + 含额度示例的消息

- **目标**：用户 @ 本账号并粘贴「回复示例：当前额度如下：EP：100 – 100,000…」时，应识别为「在给 bot 示范额度话术」或「在问额度」，回复应以**额度规则**为主（或简短确认），而不是继续沿用订单 705806645700。
- **建议措施**：  
  - 在额度子分支中，对「当前额度如下」「EP：100」「JC：100」等**方案内标准句式**做一次匹配；若当前消息包含这些，视为额度相关，优先返回标准额度回复或「已记录示例」的简短确认，且**不**把长段示例与订单历史一起喂给 AI 做订单回复。  
  - 或：对「@本账号 + 多行 + 含额度关键词」的消息，在传入 AI 前缩短/裁剪订单历史，避免订单语境覆盖额度语境。

### 3.6 P2：编码与异常

- **目标**：避免因某条消息的解码异常导致整次处理失败（如 utf-16-le 报错）。
- **建议措施**：在消息文本进入队列前做**统一解码/清洗**（如替换无法解码的序列、或捕获异常后以安全子串或占位符继续），并记录原始 message 信息便于排查；结合 3.2 的 bot 过滤，可减少此类消息进入处理链。

### 3.7 P2：情绪增强与配置统一

- **目标**：与「有温度、少官方腔」方案一致，正式环境应开启情绪增强。
- **建议措施**：确认运行所用配置中 `emoticons.enabled` 为 true，与文档和代码默认一致；若有多种部署（如测试/生产），在部署说明中写明该开关及影响。

### 3.8 P2：特殊/黑名单与群名

- **目标**：方案要求的「普通 / 特殊 / 黑名单」按群名区分，且特殊群 EP 100–100,000、黑名单定制回复，需在真实群名配置下验证。
- **建议措施**：在测试环境用**真实群标题**（与 `chat_title` 完全一致）配置 `special_groups`、`blacklist_groups`，跑一遍方案 6.1 的测试用例，并在日志中打印当前 `chat_title` 与命中结果（普通/特殊/黑名单），便于运营核对。

## 五、建议实施顺序

1. **P0**：发送者过滤（bot 不回复）+ 额度规则加载与路由（确保额度问询必走规则、数字来自 quota_rules、多问一答）。  
2. **P1**：额度元讨论降权/不触发、订单无凭证不编造、@本账号+额度示例优先走额度回复。  
3. **P2**：编码防护、情绪增强配置统一、特殊/黑名单群名测试与可观测。

完成 P0 后，建议用方案 6.1/6.2 的用例在测试群回归（含「ep现在限额多少」「jz的额度多少」「ep普通客户/特殊客户限额多少」「只问额度」等），并抓取日志确认：额度子分支命中、返回数字为 100–20,000 / 100–100,000、无 AI 编造额度。

### 6.2 在方案基础上的再优化

- **多问一答与多行格式**：方案要求「EP 普通、EP 特殊、JC 各一行」；实现上增加了「多行时单标题 + 多行额度 + 单段说明」，避免每行重复整段「如有特殊情况…」，并支持配置 `templates.line_only`（默认 `{channel_name}：{range}`）。
- **jp 视为 jc**：在解析「问到的通道」时，将「jp」视为 JC 通道，与「jz/ja」一致，减少漏答。
- **空规则可观测**：不仅「加载成功」打 info，在用户消息明显像额度问询但规则为空时打 warning，便于排查未加载原因。

2. **P2：情绪增强与配置统一**  
   - 确认运行环境 `emoticons.enabled: true`，与文档一致；在部署说明中写明该开关及影响，避免日志中出现「情绪增强器已禁用」。

3. **P2：特殊/黑名单与群名验证**  
   - 用真实群标题配置 `special_groups`、`blacklist_groups`，在测试环境跑方案 6.1 用例，并在额度分支内打 debug 日志（如当前 `chat_title`、命中普通/特殊/黑名单），便于运营核对。

4. **回归与监控**  
   - 用「ep现在限额多少」「jz的额度多少」「ep普通客户/特殊客户限额多少？jp的限额多少？」「他知识库里没有限额信息」等用例回归，确认：额度问询走规则且数字正确、元讨论不返回额度模板、@+额度示例走 channel_info；并观察日志中「额度规则已加载」「额度规则未加载或为空」等输出，便于线上排错。

   - 用方案 6.1/6.2 用例在测试群回归（额度问询、元讨论、@+额度示例、多问一答），并观察日志：额度规则已加载、额度分支 chat_title/special/blacklist、情绪增强状态、无 utf-16-le 等编码报错。

2. **可选增强**  
   - 若仍有编码问题：在写入/读取文件或调用外部 API 的边界处显式指定 `encoding='utf-8', errors='replace'`，并记录原始 message 的 repr 或 id 便于排查。  
   - 特殊/黑名单：用真实群标题在测试环境跑通后，可在部署说明中注明「请将 special_groups/blacklist_groups 与群标题完全一致」。
