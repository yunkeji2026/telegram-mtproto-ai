# 研发经费申请：智能客服订单查询与多模态对话系统

**申请项目名称**：基于大语言模型的 Telegram 智能客服订单查询与多轮对话系统  
**申请日期**：2026 年 3 月  
**版本**：v1.0  

---

## 一、项目概述与定位

本项目旨在构建一套**企业级、多语言、多模态**的智能客服系统，以 Telegram 为触达渠道，面向订单查询、通道状态、支付与回调等核心业务场景，实现 7×24 小时自动化、高拟人化的客户服务。系统采用**业界领先的大语言模型**（支持**本地部署大模型**并**时刻进行自检**；开发阶段可使用 Claude Opus 4.6、GPT-5.4 等一流模型）作为理解与生成引擎，结合五层触发机制、多轮上下文与视觉理解能力，在保证**不漏回、低误回**的前提下，显著提升客服响应效率与用户体验。

**核心价值**：将「查单、通道咨询、多语言支持」等高频、标准化能力自动化，释放人力处理复杂投诉与异常；同时为后续**自建/私有化部署大模型**预留架构与安全设计，满足数据不出域与合规要求。

## 三、研发经费主要用途说明

### 3.1 费用构成概览

研发与运行成本**主要集中在 AI 大模型 Token 消耗**上，其余为开发少量云/本地算力（如语音识别、OCR 本地化）。

### 3.2 Token 消耗说明（核心支出）

- **模型选型**：当前采用**业界一流大模型**（如 Claude Opus 4.6 等；可扩展至 GPT-5.4 多模态并存）；**本地部署大模型时可时刻进行自检**，保证回复质量、多语言能力与安全合规表现。单次请求即产生相应 Token 燃烧，是经费的主要消耗点。
- **单次对话消耗**（估算）：
  - **主对话**：系统提示词（SOP + 多语言 + 订单规则）约 1.5k–2k token，用户消息 + 上下文（含识图摘要、机器人通知等）约 0.5k–3k token，单轮回复上限 768 token；**单次主对话合计约 2k–6k token**。
  - **四层触发中的 L2 语义判断**：单次约 100–150 token（短提示 + 用户消息 + 简短判断输出）。
  - **AI 上下文回复判断**（是否与工作相关、是否回复）：单次约 150–250 token。
  - **识图/多模态**：若使用云端视觉模型（如 GLM-4V），单图约 500–2k token 量级（依分辨率与描述长度而定）。
- **综合估算**：  
  - 纯文字单轮咨询：约 **2k–4k token/次**。  
  - 含识图查单：约 **4k–10k token/次**。  
  - 高并发或长会话多轮：按轮次累加，日均/月均可根据业务量线性估算。

上述 Token 消耗直接对应云上大模型 API 费用（按输入/输出 token 计费），是当前阶段**研发与试运行的主要经费去向**。采用一流模型（如 Claude Opus 4.6）在理解准确度、多语言一致性与安全策略上具有明显优势，有利于减少误回、漏回带来的客诉与二次人力成本。

### 3.3 其他费用（简要）

- **开发与维护**：架构设计、四层触发与回复逻辑、多语言与 SOP 配置、日志与监控。
- **可选本地化**：语音识别（Whisper）、OCR（EasyOCR）等若采用本地部署，需少量算力或本地 GPU 资源，可替代部分云端调用以降低成本或满足数据本地化。

## 四、已实现功能与系统优势（当前成果）

### 4.1 已实现功能清单

| 模块 | 功能说明 |
|------|----------|
| **多轮对话与回复判定** | 回复链（@ 或回复我们必回）、四层触发（L1 关键词/图片+文字 → L2 语义 → L3 上下文 → L4）、追问上下文（最近 K 条内有我们则按追问处理）、会话窗口 + L2 兜底，有效平衡「不漏回」与「低误回」。 |
| **多语言同语回复** | 支持中/英/阿/土/葡/西/法/德/意/俄等，客户用什么语言发就用同语言回，关键词与触发规则已覆盖多语言查单/订单/通道用语。 |
| **订单查询与识图** | 图片+文字触发；Vision（如 GLM-4V）+ OCR 兜底；支持「群内最近一张图」作为凭证；可拉取指定机器人通知作为订单/回调上下文；严格按识图或通知内容回复，不编造单号/金额。 |
| **客服人设与 SOP** | 测试期固定人设 Camille、真人客服语气；订单状态类话术（有凭证则确认收到+处理中、不重复索要订单号）；情绪与语气增强（减少官方腔、增加温度）。 |
| **意图与技能** | 问候、订单查询、价格/状态/通道咨询、投诉、闲聊、测试等意图识别与技能路由；模板与 AI 结合，避免机械重复。 |
| **语音与视觉** | 语音消息转文字（Whisper 本地/云端可选）；图像理解+OCR 作为查单依据。 |
| **可观测与运维** | 日志、监控指标（处理量、API 调用、响应时间、错误率等）、健康检查；配置与规则集中管理，便于审计与迭代。 |
| **用户画像与数据策略** | **用户画像**：按会话与行为做轻量画像，用于偏好与语言倾向识别，提升回复贴合度。**数据本地化存储**：对话与识图等数据仅落盘于部署环境，**7 天自动删除**，满足合规与隐私；**保留用户喜好**（如常用语言、咨询类型）在留存期内用于个性化回复与统计，过期即清理。 |

### 4.2 系统优势（亮点归纳）

- **代码与配置安全**：  
  - 敏感信息（API Key、会话）通过配置文件与环境变量管理，**无硬编码密钥**，便于脱敏与权限控制。  
  - 配置与规则（触发条件、回复逻辑、SOP）**集中化、可审计**，满足内部安全与合规检查。  
  - 消息处理链路清晰，关键决策（是否回复、触发层级）可打标、可追溯，便于事后分析与风控。  

- **自研 API 与系统一体化**：  
  - **独有自研 API 技术接口**与业务系统**无缝结合**，从消息接入、触发决策、AI 调度到回复下发全链路自研可控，无第三方黑盒依赖，便于扩展与安全审计。  
  - **高并发与低延迟**：架构设计支持**十万级并发**接入，**响应毫秒级**（首包/决策链路），满足大流量与 7×24 稳定运行。  

- **业务侧优势**：  
  - **高拟人化**：语气与情绪增强、多语言一致、少官方腔，结合**用户画像与偏好保留**提升回复贴合度。  
  - **高可用设计**：异步队列、超时与降级、重试机制，单点故障不扩散。  
  - **可扩展**：技能与意图可插拔；AI 提供商与模型可切换（可扩展至 Claude Opus 4.6 等）；**本地部署大模型时支持时刻自检**，为后续自建模型预留接口。  

- **数据与合规**：  
  - **数据本地化存储**，**7 天自动删除**，仅保留必要的**用户喜好**等轻量信息用于体验优化，过期即清理，符合隐私与合规要求。  

- **成本与效果平衡**：  
  - 通过四层触发与 L2 兜底，减少不必要的长上下文调用；会话窗口与追问逻辑精准触发，在控制 Token 消耗的同时提高有效回复率。

## 五、规划中的功能与演进方向

- **多模态与多轮增强**：更细粒度的订单状态回传、多图/多凭证串联、语音回复（TTS）等。  
- **数据分析与策略优化**：漏回/误回率统计、各触发层级贡献度、多语言分布；基于数据调整阈值与关键词。  
- **高优先级与分级策略**：按群/用户/业务线配置不同回复策略与兜底强度。  
- **风控与安全**：防刷、防滥用、敏感信息过滤、审计日志持久化与脱敏策略。  

**重点方向：自建/私有化部署大模型与安全性**

- **自建大模型或私有化部署**：  
  - 在现有「可切换 AI 提供商与模型」的架构基础上，规划对接**自建或私有化部署的大模型**（如内网 API、本地推理服务），实现**业务数据不出域**；**本地部署大模型时刻进行自检**，保障服务可用性与输出质量。  
  - 经费可覆盖：私有化模型推理资源（GPU/推理服务）、内网 API 网关、与现有 Telegram 客服管线的集成与测试。  

- **安全与合规**：  
  - **数据安全**：对话内容、识图结果、订单相关上下文仅在内网或授权环境中处理；对接私有化模型后，不经过第三方公有云大模型，降低泄露与合规风险。  
  - **审计与留痕**：关键操作与回复决策可记录、可回溯，满足内部审计与行业合规要求。  
  - **访问与密钥管理**：API 密钥与配置分层管理、最小权限、定期轮换，与现有「无硬编码、可脱敏」设计一致并增强。  

上述规划与当前 Token 经费申请形成衔接：**现阶段**以一流公有云大模型（如 Claude Opus 4.6 等）快速验证效果与打磨 SOP；**下一阶段**将部分或全部流量切换至**本地部署大模型**，**时刻进行自检**，在保证体验的前提下实现数据本地化与长期成本可控。
