# 智能客服订单查询与对话系统 — 功能总结与使用说明

本文档汇总当前系统**全部功能**及**使用与配置说明**，便于运营、部署与后续需求对齐。

---

## 一、项目定位

- **系统名称**：基于 Telegram MTProto + 本地部署大模型的智能客服订单查询与对话系统（Camille 客服）
- **核心能力**：以 Telegram 用户身份登录，在私聊与群聊中自动处理**订单查询、通道/额度咨询、多语言客服对话**，并结合**识图、语音转写、额度规则、情绪增强**提供统一回复。
- **技术栈**：Pyrogram（MTProto）、大模型 Chat API、YAML 配置、异步消息队列、四层触发 + Skill 工作流。

## 二、功能清单

### 2.1 消息接收与触发

| 功能 | 说明 | 配置/入口 |
|------|------|-----------|
| **私聊/群聊** | 支持私聊与群组消息；可配置是否处理群组（`process_groups`） | `config.yaml` → `telegram.process_private` / `process_groups` |
| **发送者过滤** | 不回复其他 Bot：`from_user.is_bot` 或 `no_reply_sender_usernames`（如 gxp_notify_bot）直接跳过 | `telegram.no_reply_sender_usernames`、`telegram_client.py` 群消息入口 |
| **四层触发（L1–L4）** | L1 关键词/图片+文字瞬间触发；L2 追问/回复我们；L3 会话窗口内兜底；L4 AI 判断是否工作相关 | `config/trigger_rules.yaml`、`config.yaml` → `trigger` |
| **群回复模式** | `mention_or_keyword`：被 @ 或命中关键词才在群里回复；支持多语言关键词与 @ 名单 | `telegram.group_reply.mode`、`keywords`、`mention_usernames` |
| **回复链逻辑** | 回复我们必回、追问检测（最近 K 条内我们发过则可能回）、会话窗口内 L2 兜底、AI 上下文判断 | `telegram.reply_logic` |

### 2.2 多模态输入

| 功能 | 说明 | 配置 |
|------|------|------|
| **文本** | 正文与 caption 统一归一化（P2 编码防护），再进入技能与 AI | `_normalize_message_text()` |
| **语音消息** | 下载后调用 Whisper 本地转录，结果以「[语音转录] xxx」形式参与后续流程 | `voice_recognition`（provider、whisper、language、timeout） |
| **图片消息** | 以 **Vision（智谱 GLM-4V）为主、OCR 兜底** 解析图中内容，用于订单凭证识别与查单依据 | `vision`（provider、model、prompt）、`image_recognition`（OCR） |
| **群内最近图** | 当前消息无图但为订单相关问句时，拉取群内最近一张图做 OCR，作为查单依据 | `context.bot_sources`、`_get_recent_chat_image_ocr` |
| **编码防护** | 对 message 文本、语音/识图结果、入队前 text 做 UTF-8 安全归一化，避免 utf-16-le 等异常导致整次失败 | `telegram_client._normalize_message_text` |

### 2.3.1 代发 gxp 命令（gxp_command）

用户在本群说「查代收 单号」「回调代收 单号」「查汇率」「余额」「代收成功率」「utr 查询」「添加成员 @xxx 验证码」「切换语言」等时，Camille 会**在群里代发**对应命令给 gxp_notify_bot（如 `/cxds 12345`、`/hl`、`/cxye`），然后回复用户「已帮您发起请求，请查看上方机器人回复。」  

- **配置**：`config.yaml` → `telegram.gxp_commands.enabled`（默认 true）；意图关键词见 `intent.keywords.gxp_command`。  
- **命令对应**：/qhyy 切换语言；/tjcy 添加操作成员；/mnds 代收模拟回调；/cxds 查询代收；/htds 回调代收；/cxdf 查询提现；/htdf 回调提现；/utr utr 查询/补单；/hl 汇率；/cxye 余额；/cgl 代收成功率。  
- **仅发单号**：用户只发 6~24 位单号（或「单号 12345678」）时，先问「您需要：查询代收/回调代收/查询提现/回调提现？」；用户下条回复需求或数字 1/2/3/4 时，用缓存的单号代发命令。待用单号按会话隔离、可配置过期时间（默认 300 秒）。  
- **帮用户确认需求**：用户只说「查一下」「帮我查」等未说明要查什么时，客服会问「您要查的是：1 汇率 2 余额 3 代收订单状态 4 提现订单状态 5 代收成功率？请直接说一项或发数字。」用户回复 1/2/5 即代发 /hl、/cxye、/cgl；回复 3/4 则引导发单号。  
- **参数**：单号从消息或上下文待用单号获取；数字简写 1→查代收、2→回调代收、3→查提现、4→回调提现（仅单号流程）；@用户名、验证码、UTR 按规则解析；缺参数时提示用户补全。

### 2.4 额度规则（Channel 信息子分支）

| 功能 | 说明 | 配置 |
|------|------|------|
| **额度子分支** | 在 ChannelInfoSkill 内优先执行；命中则按配置返回固定文案，**不调 AI** | `config/quota_rules.yaml`、`skill_manager._quota_reply` |
| **通道与数字** | EP 普通 100–20,000、EP 特殊 100–100,000、JC 100–100,000；支持 jz/ja/jp 等别名 | `quota_rules.yaml` → `channels` |
| **按群名区分** | 特殊客户群用 `special_groups` 名单，EP 用 special_range；黑名单群用 `blacklist_groups` 定制 ep/jc 文案（当前可为空） | `special_groups`、`blacklist_groups` |
| **额度说明补充** | 可选 `quota_note_supplement`：用尽恢复、每日限额变化等说明，追加在所有额度相关回复末尾；**额度或规则变更时只改此处即可**，无需改代码 | `quota_rules.yaml` → `quota_note_supplement` |
| **运营自改配置** | 主管确认：黑名单/特殊群后续会常变，需运营自己改配置。**方式一**：直接编辑 `config/quota_rules.yaml`（热更新，无需重启）。**方式二**：与机器人对话发送命令（需在 `telegram.quota_config_commands` 中开启并填写 `allowed_user_ids`），见下表。 | `quota_rules.yaml`、`telegram.quota_config_commands` |
| **引导语** | 未指定通道时引导语同时覆盖「额度/限额」「代收代付限额」，与商户常问一致 | `templates.ask_channel` |
| **多问一答** | 一次问「ep 普通/特殊 + jp/jc」时，分条回复 EP（普通）、EP（特殊）、JC，多行时单标题+单段说明 | `_quota_reply` 内 asked_ep/asked_jc/want_both_tiers |
| **元讨论不触发** | 消息含「知识库/他没地方/不会」等且含「额度/限额」时不返回额度模板，避免误回 | `_quota_reply` 开头 meta_words 判断 |
| **规则可观测** | 加载成功打 info（通道数）；规则为空且像额度问询时打 warning；额度分支打 debug（chat_title、special、blacklist） | `config_manager.get_quota_rules`、`_quota_reply` |
| **路径容错** | 主路径无 `quota_rules.yaml` 时回退到 `.../config/quota_rules.yaml` | `config_manager.get_quota_rules` |

### 2.5 订单查询与 SOP

| 功能 | 说明 |
|------|------|
| **有凭证** | 上下文中已有识图/机器人订单信息（交易号、金额、时间等）时，回复「已根据您发的凭证（交易号 xxx）确认到订单…处理中，请耐心等待系统回调」，不再索要订单号。 |
| **无凭证** | 无识图/无机器人订单上下文时，**禁止**输出具体订单细节（金额、商品、地址、时间等），仅允许「暂未看到订单信息，请发订单号或付款截图方便核查」。 |
| **多语言** | 客户用什么语言发，就用同一语言回复（阿拉伯语、土耳其语、乌尔都语、葡语、英/西/法/德/意/俄等）。**已支持**：术语表、额度英文模板、prompt 输出语言粗判（配置见 `ai.terminology`、`quota_rules.templates.en`）；后续可按反馈增加更多语种或固定话术库。 |
| **人设** | Camille 客服；避免机械「稍等」「已反馈」；语气带温度、少官方腔（见 `ai.system_prompt` 与 `docs/SOP_PERSONA_AND_FLOW.md`）。 |

## 三、使用说明

### 3.1 部署与运行

1. **环境**：Python 3.8+，依赖见 `requirements.txt`。
2. **配置**：复制 `config/config.example.yaml` 为 `config/config.yaml`，填写：
   - `telegram.api_id` / `api_hash` / `phone_number` / `session_name`
   - `ai.api_key`（大模型 API 密钥）
   - 群回复 `mention_usernames`、`no_reply_sender_usernames`（按需）
   - `vision` / `voice_recognition` 等（若使用识图/语音）
3. **首次运行**：`python main.py`，按提示完成手机验证与两步验证（若有）。
4. **会话**：会话文件保存在项目目录（如 `session_name`），勿泄露。

### 3.2 关键配置项速查

| 目的 | 配置位置 | 说明 |
|------|----------|------|
| 不回复某 Bot | `telegram.no_reply_sender_usernames` | 列表，如 `["gxp_notify_bot"]` |
| 群内何时回复 | `telegram.group_reply.mode` | `mention_or_keyword` / `mention_only` / `keyword_only` / `always` |
| 被 @ 名单 | `telegram.group_reply.mention_usernames` | 如 `["@ai_zkw","Camille"]` |
| 额度数字与群名单 | `config/quota_rules.yaml` | 改 `channels`、`special_groups`、`blacklist_groups`、`templates` |
| 情绪增强开关 | `emoticons.enabled` | 建议 `true`；启动日志会显示当前状态 |
| 对话命令改额度配置 | `telegram.quota_config_commands.enabled`、`allowed_user_ids` | 设为 `true` 并填 Telegram 用户 ID 后，该用户发「列出特殊群」「添加特殊群 群名」「删除黑名单 群名」等即可改 `quota_rules.yaml`，热更新生效 |
| 日志详细程度 | `logging.level` | `DEBUG` 可看到额度分支 chat_title/special/blacklist |

### 3.3 运营注意

- **额度**：EP/JC 数字与特殊/黑名单以 `quota_rules.yaml` 为准；修改后热更新生效，无需重启。**额度或规则变更**（如用尽恢复、每日限额说明）时，更新配置中的 `quota_note_supplement` 即可，AI/用户通过回复内容自然获得最新口径。
- **群名**：`special_groups`、`blacklist_groups` 需与群标题**完全一致**（含空格与大小写）。当前可无黑名单（`blacklist_groups` 为空），后续若需则按配置格式添加。
- **订单**：无截图/无机器人订单上下文时，AI 仅会引导发凭证，不会编造订单细节（已写入 system_prompt）。

### 3.4 日志与排错

- **额度未走规则**：查日志是否有「额度规则已加载」「额度规则未加载或为空」；若为空，检查 `quota_rules.yaml` 路径与权限。
- **群名未命中特殊/黑名单**：将 `logging.level` 设为 `DEBUG`，查看「额度分支: chat_title=…, special=…, blacklist=…」与配置是否一致。
- **编码报错**：消息文本与识图/语音结果已做归一化；若仍有异常，可查原始 message 的 id 或 repr 便于定位。
- **情绪增强未生效**：启动时看「情绪增强: 已启用/已禁用」；若禁用，将 `emoticons.enabled` 设为 `true` 并重启。

## 四、管理面板 UI 模式（简洁 / 完整）

管理面板新增双模式切换功能，解决客服人员反馈的「后台太复杂」问题。

### 4.1 架构设计

- **CSS 隐藏方案**：通过 `<html data-ui-mode="simple|full">` 配合全局 CSS `[data-ui-mode="simple"] .pro-only{display:none!important}` 实现，无需在每个模板中写大量 `{% if %}` 判断。
- **Cookie 持久化**：用户偏好存储在 `ui_mode` Cookie 中，服务端通过 `resolve_ui_mode()` 函数解析。
- **角色默认值**：`ROLE_DEFAULT_UI_MODE` 字典定义各角色默认模式（viewer 默认 simple，master/admin 默认 full）。

### 4.2 涉及文件

| 文件 | 改动 |
|------|------|
| `web_user_store.py` | `UI_MODE_SIMPLE/FULL`、`SIMPLE_MODE_CORE_PAGES`、`SIMPLE_MODE_MORE_PAGES`、`resolve_ui_mode()`、`is_page_visible_in_simple()` |
| `admin.py` | Jinja2 上下文注入 `ui_mode`；`/set_ui_mode` 端点；简洁模式首页重定向 |
| `base.html` | `data-ui-mode` 属性、全局 `.pro-only` CSS、侧边栏双模式、模式切换按钮、命令面板模式命令 |
| 所有页面模板 | 条件标题、`.pro-only` 类标记、简洁模式引导横幅、友好空状态 |

### 4.3 已适配页面清单

全部 17 个内容模板（不含 login/setup）均已完成简洁模式适配：

- **深度适配**（pro-only 标记 + 引导横幅 + 标题切换）：cases, knowledge, learner, help, dashboard, channels, analytics, audit, settings, strategies, templates
- **轻度适配**（高级页面提示横幅）：users, logs, diff, import, strategy_analytics
- **无需适配**：login（独立页面）, setup（独立页面）
