# 四层触发 · 代查漏触发分析与优化方案

## 一、日志中「未触发」且与代查/业务相关的话术

从 `logs/app.log` 中整理出的、**未触发回复**且**语义上应走代查或业务咨询**的消息如下。

| 用户原话 | 理解语意 | 当前结果 | 期望 |
|----------|----------|----------|------|
| 什么时候回 调出来 | 问「什么时候回调出来」 | 跳过未触发 | L1 触发 → 进代查/业务 |
| Order inquiry | 订单查询（英文） | 跳过未触发 | L1 触发 → 进代查 |
| 你能做什么 / 你能做什么。 | 问能力/服务范围 | 跳过未触发 | L1 或 L2 触发 → 介绍/代查入口 |
| 回答我。 | 催回复 | 跳过 | 可不触发或 L2 低置信度 |
| @wookfaith 你叫什么，聊聊天呗 | @ 本账号 + 闲聊 | 跳过未触发 | 若需「@即触发」则 L1 提及需含本 bot |
| 我来测试你的能力 | 测试/能力 | 跳过 | 可选：L2 给中等置信度 |
| 怎么不回复了 | 催回复 | 跳过 | 可不触发 |

**与代查强相关的漏触发**（优先改）：  
**「什么时候回 调出来」「Order inquiry」「你能做什么」** —— 应通过 L1 或 L2 稳定触发，再进入技能层做「代查/问意向」等。

---

## 二、当前四层逻辑简要

- **L1**：规则触发（关键词、订单号格式、@ 提及、图片+文字）。  
  - 关键词来自 `config/trigger_rules.yaml` → `l1_rule_trigger.high_frequency_keywords.all_keywords`。  
  - 已对文本做**去空格**：`normalized = re.sub(r'\s+', '', message_text)`，再在 `normalized` 上做包含匹配。  
- **L2**：语义置信度；当前为规则兜底（业务词命中 ≈0.95，问候 ≈0.65，默认 0.5）。  
- **L3**：上下文过滤（冷却、闲聊序列等）。  
- **L4**：人工兜底/沉默。

因此：  
- 「什么时候回 调出来」→ 去空为「什么时候回调出来」→ 应命中 L1「回调」「什么时候」；若仍漏，可能是历史版本或传参差异，**建议 L1 显式增加「调出来」「回调出来」等变体**，并统一做文本规范化。  
- 「Order inquiry」→ 去空为「orderinquiry」→ 理论上可命中「order」「inquiry」；若漏，**建议 L1 增加整词 "orderinquiry" 或 "order inquiry" 的归一形式**，并保证英文大小写一致（统一 lower）。  
- 「你能做什么」→ 当前 L1 无「能做」「做什么」→ **建议 L1 增加「你能做什么」「能做什么」「做什么」「可以做什么」「服务什么」等**，或 L2 对这类「问能力」给高置信度。

## 三、优化方案（理解语意、更好触发）

### 3.1 L1：补充「代查/追问/能力」相关关键词（trigger_rules.yaml）

在 `l1_rule_trigger.high_frequency_keywords.all_keywords` 中**新增**（与现有不重复即可）：

- **回调相关**：`调出来`、`回调出来`、`什么时候回调`、`多久回调`、`回 调`（若希望未归一也命中可加，一般靠归一即可）。  
- **查单/代查相关**：`查单`、`查订单`、`查看订单`、`帮我查`、`查一下`、`查到了吗`、`查新订单`。  
- **问能力/服务**：`你能做什么`、`能做什么`、`有什么功能`、`可以做什么`、`服务什么`、`有什么服务`。  
- **英文查单**：`order inquiry`、`orderinquiry`（或保证现有 `order` + `inquiry` 在归一后都能命中）。

这样「什么时候回 调出来」「Order inquiry」「你能做什么」等会在 L1 就被识别为业务/代查相关并触发。

### 3.2 L1：文本规范化增强（four_layer_trigger.py）

- 保持现有：`normalized = re.sub(r'\s+', '', message_text)`。  
- 建议增加：  
  - 全角空格 → 半角并去空：如 `message_text.replace('\u3000', ' ')` 再 `re.sub(r'\s+', '', ...)`。  
  - 英文统一小写再匹配：对 `normalized` 做 `lower()`（当前若 `case_sensitive=False` 已做，可确认 L1 关键词匹配处一致）。  

避免「回　调」「Order  Inquiry」等因空格/全角/大小写导致漏触发。

### 3.3 L2：代查/问能力 语义置信度增强（_get_ai_confidence）

在 `_get_ai_confidence` 的规则兜底中，对「代查/追问/问能力」类短语**单独给高置信度**（如 0.95），作为 L1 未命中时的兜底：

- 短语示例：`你能做什么`、`能做什么`、`可以做什么`、`有什么功能`、`服务什么`、`什么时候回调`、`调出来`、`回调出来`、`查单`、`查订单`、`order inquiry`、`查新订单`。  
- 实现方式：在现有 `business_keywords` 列表后增加一段「短语列表」；若 `text_lower` 或 `normalized` 包含其中任一条，则 `return 0.95`。

这样即使 L1 漏掉个别说法，L2 也能按「语意」补触，减少代查漏触发。

### 3.4 L1 提及触发：支持「@本 bot 即触发」

若希望「@wookfaith 你叫什么，聊聊天呗」这类「@ 本账号」的消息一律先触发，再交给 L3/L2 判闲聊：

- 在 L1 的 `mention_trigger` 中，不仅配置固定 `usernames`（如 客服、Camille），还支持**当前 bot 的 @username**（由 telegram 层或 config 传入，如 `@wookfaith`）。  
- 或：在 Telegram 层判断「消息是否 @ 了当前账号」，若是则视为「已触发」，再交给四层做 L2/L3 决策（是否真回复、是否代查等）。

这样「理解语意」不变，但触发更符合「@ 即算一次有效触发」的预期。

### 3.5 不建议触发的保持现状

- 「回答我。」「怎么不回复了」：纯催回复，可不加 L1 词，L2 维持默认或略低置信度即可。  
- 「你猜我想让你做什么」：模糊，可不做特别触发，避免误触。

## 四、实施优先级建议

| 优先级 | 项 | 说明 |
|--------|----|------|
| P0 | L1 增加「代查/追问/能力」关键词（3.1） | 直接解决「什么时候回 调出来」「Order inquiry」「你能做什么」漏触发 |
| P1 | L2 代查/问能力短语置信度（3.3） | 语意兜底，减少类似说法的漏触发 |
| P2 | L1 文本规范化（3.2） | 全角空格、大小写统一，避免边界漏触 |
| P3 | L1 或 Telegram 层 @ 本 bot 即触发（3.4） | 按产品需求决定是否「@ 即触发」 |

## 五、小结

- **问题**：若干与代查/业务明确相关的话（如「什么时候回 调出来」「Order inquiry」「你能做什么」）在日志中未触发。  
- **原则**：在**理解每句用户语意**的基础上，通过 L1 关键词与 L2 语义兜底，让「代查/问能力/订单·回调相关」更稳定触发，同时不扩大无关话术的误触。  
- **落地**：在 `trigger_rules.yaml` 的 L1 中补充上述关键词与短语；在 `four_layer_trigger.py` 的 L2 规则置信度中增加「代查/问能力」短语高置信度；可选做规范化与 @ 本 bot 支持。  

按上述顺序改完后，可用日志中的原话（尤其是「什么时候回 调出来」「Order inquiry」「你能做什么」）回归测试，确认均能触发并进入代查或相应技能流程。

## 六、斜杠命令无法激活（日志结论与修复）

**现象**：用户直接发 `/cxye`、`/cxds 1234567890`、`/cgl`、`/cxdf` 等时，日志出现「四层触发决策: 不回复 - L2置信度不足: 0.5」，只能靠 AI 上下文才进处理，且易漏。

**原因**：L1 规则使用的 `trigger_rules.yaml` 里 `all_keywords` 未包含这些斜杠命令名，消息中只有 `/cxye` 等时没有任何 L1 关键词命中，L2 规则里也没有对应词，置信度落在默认 0.5，被判定为不回复。

**修复**：
- 在 `trigger_rules.yaml` 的 L1 `all_keywords` 中增加：`/qhyy`、`/tjcy`、`/mnds`、`/cxds`、`/htds`、`/cxdf`、`/htdf`、`/utr`、`/hl`、`/cxye`、`/cgl`，使用户发任意一条上述命令时 L1 即触发。
- 在 `four_layer_trigger.py` 的 `_get_ai_confidence` 的 `business_keywords` 中增加上述斜杠命令，作为 L2 兜底，避免个别格式仍漏触。

| 优先级 | 内容 | 实施说明 |

| **P1** | L2 代查/问能力短语置信度 | 在 `_get_ai_confidence` 中先匹配「代查/问能力」短语列表（同上+order inquiry），命中即返回 0.95；并对文本做去空格归一便于短语匹配。 |
| **P2** | L1 文本规范化 | L1 关键词匹配前：`message_text` 先做全角空格→半角（`\u3000`→空格），再去空格、小写，避免「回　调」「Order  Inquiry」漏触。 |
| **P3** | @本 bot 即触发 | `should_reply` 增加可选参数 `bot_username`；`_check_l1_rule_trigger` 增加「@本账号」检测（消息含 `@bot_username` 即 L1 触发）。Telegram 层调用时传入 `self.user_info.username`。 |

**在方案基础上的再优化：**

- L2 短语匹配时同时用「去空格归一」与「原文小写」两种方式，减少漏判。
- @本 bot 使用 `strip().lstrip('@')` 兼容传入带 @ 或不带 @ 的 username。
- L1 将「@本账号」放在首条（0），优先于图片/关键词/订单号/配置提及，逻辑更清晰。

### 8.3 结论

- **所有日志里「1」「2」「3」的案例**（含「跳过未触发」和「仅靠 AI 上下文/追问才进」）在修改后都会：
  - 被 **L2 纯数字 1～5 高置信度** 在四层直接判为「回复」，稳定进入技能层并执行代发；
  - 若用户发的是 `1。`、`2，` 等带尾标点，**追问放宽**会继续用「选项数字 1～5」分支触发，作为补充。
- **「4」原本就触发**的案例，修改后仍由 L2（`"4"` → 0.95）或追问覆盖，行为一致。
- 建议回归：在同一群内先发单号/「查订单」→ 机器人问「1 汇率 2 余额…」或「1 查代收 2 回调…」→ 用户分别回复 **1、2、3、4**，确认均能触发并代发对应命令。
