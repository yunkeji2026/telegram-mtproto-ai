## 二、修改建议（方案层面，不涉及代码）

### 1. 对「黑名单/特殊群 — 需要」的落地建议

- **配置方式**  
  - 继续沿用当前方式：黑名单与特殊群名单写在 **同一份配置文件**（如 `quota_rules.yaml`）的 `blacklist_groups`、`special_groups` 中，运营通过**编辑 YAML 文件**即可增删群、改话术，无需改代码。  
  - 在《功能总结与使用说明》或运维文档中**明确写出**：  
    - 黑名单/特殊群名单的**文件路径与段名**；  
    - **热更新**是否支持（若支持，说明「改完保存即生效、无需重启」）；  
    - 若需重启，写清「修改后需重启 XX 服务」。

- **运营操作指引**  
  - 提供**简短操作说明**（可放在 `quota_rules.yaml` 顶部注释或单独一页）：  
    - 如何新增/删除一个黑名单群（复制示例格式、改群名与 ep/jc 话术）；  
    - 如何新增/删除一个特殊客户群（在 `special_groups` 列表中增删一行，群名与 Telegram 群标题完全一致）。  
  - 说明群名必须与 Telegram 群标题**完全一致**（含空格、大小写、符号），避免运营改错导致不生效。

- **是否做「运营自助界面」**  
  - 当前方案：**仅配置化**，运营改 YAML 即可满足「自己改配置」；主管已接受「黑名单用户很少可人工通知」，故**不必**在本阶段做 Web/后台自助界面。  
  - 若后续运营明确要求「不想碰 YAML、要在网页上勾选群名」，再单独排期做「黑名单/特殊群管理界面」与配置同步逻辑。

### 2. 与现有实施的衔接

- 当前程序已实现：  
  - 特殊客户群、黑名单群均从 `quota_rules.yaml` 读取；  
  - 支持热更新（按文件 mtime 重载）；  
  - 黑名单当前可为空（`blacklist_groups: {}`），后续按格式添加即可。  

- **建议**：  
  - 在《功能总结与使用说明》或《下一步需求_回复对比与程序优化方案》中**补充一句**：「主管确认：黑名单/特殊群后续会常变，需要运营自己改配置；当前通过编辑 `quota_rules.yaml` 即可完成，已支持热更新，无需重启。」  
  - 若有内部运维/运营手册，增加「黑名单与特殊群配置变更 SOP」：谁有权改、改完谁验证、是否需通知开发。

### 3. 可选：配置校验与排错

- 若运营反馈「改完不生效」，可考虑（后续迭代）：  
  - 在加载 `quota_rules.yaml` 时打 **info 日志**：如「黑名单群数：N」「特殊客户群数：M」，便于核对是否加载到最新；  
  - 或在文档中说明：将 `logging.level` 设为 DEBUG 可看到「额度分支: chat_title=…, special=…, blacklist=…」，用于核对群名是否匹配。  

以上为**可选项**，不强制本次实现。
