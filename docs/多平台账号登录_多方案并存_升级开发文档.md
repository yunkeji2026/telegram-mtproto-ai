## 1. 背景与目标

竞品（易翻译 / HelloWords / 云译 YunYi GPT）的核心卖点之一是「**无限账号登录**」：用户可在
一台机器上接入大量跨平台账号（WhatsApp / Telegram / LINE / Messenger / Instagram / Signal…），
并在统一收件箱里收发 + 翻译 + 话术。

本项目当前以**真机 / 模拟器 RPA**为主（一号≈一台设备），账号数受物理设备数限制，
无法达到「无限」。本次升级目标：

1. **多登录方式并存**：同一平台可同时支持「协议多开 / 网页隔离 / 真机 RPA」多种接入方式。
2. **用户可选**：在「新增账号」时由用户选择用哪种方式接入；每个账号记住自己的方式。
3. **统一聚合不变**：无论账号来自哪种方式，统一收件箱、翻译、话术、关系阶段等上层逻辑无感知。
4. **可渐进**：框架先行，provider 逐个补齐，未就绪的方式在 UI 上灰显「不可用」。

## 3. 竞品实现原理（结论速览）

| 路线 | 做法 | 适用平台 | 优点 | 缺点 |
|---|---|---|---|---|
| A 协议多开 | 协议库 + 一号一鉴权文件，无浏览器 | TG(tdlib/pyrogram)、WA(Baileys/whatsmeow)、IG(instagrapi)、Signal(signal-cli) | 单机海量、极省资源 | 逆向/ToS 风险、封号 |
| B 网页隔离 | 内嵌 Chromium，一号一隔离 profile + 网页二维码 | WA Web、TG Web、messenger.com、instagram.com | 兼容好、像真人 | 资源重、需指纹+代理 |
| C 真机/模拟器 | ADB+投屏，二维码在设备屏 | LINE（首选）、各平台 | 最难封 | 账号数=设备数，做不到无限 |

**「无限」真正的护城河 = 防关联**：一号一指纹（Canvas/WebGL/UA/字体/时区）+ 一号一代理 IP +
会话持久化 + 行为模拟（打字延迟/在线时段/心跳）。否则多开必被批量封。

## 4. 目标架构

### 4.1 分层

```
            统一收件箱 / 翻译 / 话术 / 关系阶段（上层，无需感知 mode）
                                  ▲ 标准化消息事件（normalizer）
            ChannelAdapter 注册表（按平台聚合「多种 worker」的会话/状态/发送）
                                  ▲
        账号池 Orchestrator（增删账号、拉起/保活 worker、分配代理+指纹）
        ┌───────────────┬───────────────┬───────────────┐
   protocol-worker   web-worker      device-worker      …（水平扩展=「无限」）
   (pyrogram/Baileys) (指纹浏览器)    (真机/模拟器 RPA)
```

### 4.2 登录方式（mode）枚举

| mode | 含义 | 首选平台 |
|---|---|---|
| `protocol` | 协议多开（TG: pyrogram session；WA: Baileys/whatsmeow 微服务） | Telegram、WhatsApp |
| `web` | 网页隔离（指纹浏览器 profile + 平台网页二维码） | Messenger、IG、WA |
| `device` | 真机/模拟器 RPA（现状） | LINE、各平台兜底 |

### 4.3 数据模型

```
# Provider 注册（现状是单 key=platform，目标扩为 (platform, mode)）
register_login_provider(platform, mode, fn)
  fn(request, platform, mode, account_id) -> {qr_url?, qr_image?, instruction?, login_token?}

# 能力探测
list_modes(platform) -> [{ mode, available: bool, recommended: bool, reason: str }]
  available 由依赖/配置探测：pyrogram 是否装、WA 微服务是否在、指纹浏览器是否接好、设备是否在线

# 账号注册表（需新增持久化）
account = {
  platform, account_id, mode,          # 运行时据 mode 决定用哪种 worker 拉起
  label, proxy_id?, fingerprint_id?,   # 防关联
  status, created_at, last_online_at
}
```

### 4.4 现状 → 目标的接口演进

- `platform_login.register_login_provider(platform, provider)`
  → `register_login_provider(platform, mode, provider)`，内部 key 改为 `f"{platform}:{mode}"`。
- `POST /api/platforms/{platform}/login/start`
  → body 增加 `mode`（缺省取该平台 default mode）。
- 新增 `GET /api/platforms/{platform}/modes` 返回 `list_modes` 结果，供前端渲染方式选择。
- 新增账号注册表（落库），承载 `mode/proxy/fingerprint`，供 Orchestrator 重启后正确拉起。

## 9. 风险与取舍

- **合规/封号**：协议多开与逆向库（Baileys/whatsmeow/instagrapi）违反各平台 ToS，存在封号风险；
  需业务侧确认风险承受度，并强制配套「一号一指纹一代理 + 养号」。
- **复杂度**：每多一种 mode 即多一套 worker 运维。建议每平台默认只开 1 种推荐方式，其余按需开启。
- **资源**：web 方式（指纹浏览器）单账号内存占用高；protocol 方式最省。单机目标账号规模决定方案配比。
- **多语言栈**：WA protocol 主流实现是 Node/Go，需决定是否引入微服务（与现有 Python 主进程通过本地 API/队列对接）。

### M5 编排器 API + 启用

| 方法 | 路径 | 说明 |
|---|---|---|
| GET | `/api/accounts/orchestrator` | 编排器状态（运行循环/各账号 worker 状态/分布计数） |

### M6① 收发桥接 API + 启用

| 方法 | 路径 | 说明 |
|---|---|---|
| POST | `/api/internal/protocol/ingest` | 内部入站桥：外部 worker（Baileys Node）把收到的消息 push 进统一收件箱（body: platform/account_id/chat_key/name/text/ts/msg_id/direction） |

### M6③ 联调自检 + 操作者 checklist

```bash
python scripts/protocol_doctor.py              # 只读自检（配置/依赖/Node 可达性/编排器/sink）
python scripts/protocol_doctor.py --json       # JSON（供脚本/CI 解析；overall_ready 决定退出码）
python scripts/protocol_doctor.py --smoke \
  --server-url http://127.0.0.1:8000 --token <admin_token>   # 入站桥烟囱测试（无需真号）
```

`GET /api/accounts/protocol/readiness` 返回同一份报告，供前端展示。

**真号联调步骤（建议顺序）**：

1. `python scripts/protocol_doctor.py` → 修齐所有 ❌（开 flag、装 pyrogram、填 api_id/api_hash、起 Node）。
2. `--smoke` 烟囱测试通过 → 证明 push→落库→收件箱 thread 链路 OK（不依赖真账号）。
3. Telegram：`scripts/create_pyrogram_session.py` 生成测试号 session → 收件箱「扫码新增」→ 验证收/发/回填。
4. WhatsApp：起 Node（配 `PY_INGEST_URL`/`PY_API_TOKEN`）→ 收件箱「协议多开」扫码 → 给测试号发消息验证入站、坐席回复验证出站。
5. 开 `orchestrator_enabled: true` → 重启主进程，确认多账号自动复活；杀 Node/断网验证退避重连。

> 媒体落地默认开。Telegram 直接下载到 `src/web/static/protocol_media/telegram/`（已 gitignore）。
> WhatsApp 需设 Node 端 `WA_MEDIA_DIR`（单机指向 `src/web/static/protocol_media/whatsapp` 绝对路径）；
> 未设则 WA 仅取文本/caption。下载失败一律降级为占位，不阻断消息落库。
