## 一、系统概述

本系统是一套基于 **Telegram 用户账号（MTProto 协议）+ AI 大模型** 的智能客服对话系统。以真实用户身份（Camille / @wookfaith）登录 Telegram，在群组中自动识别用户意图并回复，同时提供 Web 管理面板和 RESTful API 进行远程运维。

### 核心依赖（均为远程 API，无需本地额外服务）

| 依赖 | 用途 |
|---|---|
| Telegram MTProto API | 收发群组消息（用户账号登录） |
| claude-4.6-oups-high Chat API | AI 对话生成、语义理解 |
| 智谱 GLM-4V-Flash | 图片识别（付款截图 OCR） |

### 技术栈

| 组件 | 技术 |
|---|---|
| Telegram 客户端 | Pyrogram（MTProto） |
| AI 对话 | claude-4.6-oups-high V3（OpenAI 兼容 API） |
| 图片识别 | 智谱 GLM-4V-Flash + Tesseract OCR 兜底 |
| 后端框架 | FastAPI + Uvicorn |
| 模板引擎 | Jinja2 |
| 数据库 | SQLite（审计日志 `audit.db` + 用户上下文/事件 `bot.db` + 策略追踪 `strategy_events.db`） |
| 异步框架 | asyncio + aiohttp |
| 配置管理 | PyYAML（热重载） |
| 测试 | Pytest（97+ 项单元测试） |

## 三、功能清单

### 3.1 核心业务功能

#### 3.1.1 GXP 命令代发

Bot 在群里代用户向 `gxp_notify_bot` 发送查询命令，支持三种触发方式：

**直接斜杠命令（11 个）：**

| 命令 | 说明 | 示例 |
|---|---|---|
| `/hl` | 查询系统汇率 | `/hl` |
| `/cxye` | 余额查询 | `/cxye` |
| `/cgl` | 查询代收成功率 | `/cgl` |
| `/qhyy` | 切换语言 | `/qhyy` |
| `/cxds 单号` | 查询代收订单 | `/cxds 234534535345` |
| `/htds 单号` | 回调交易订单 | `/htds 234534535345` |
| `/cxdf 单号` | 查询提现订单 | `/cxdf 456547567567` |
| `/htdf 单号` | 回调提现订单 | `/htdf 456547567567` |
| `/mnds 单号` | 代收订单模拟回调 | `/mnds 234534535345` |
| `/utr UTR号` | UTR 查询/补单 | `/utr 123456789012` |
| `/tjcy @用户 验证码` | 添加操作成员 | `/tjcy @user1 8888` |

**中文自然语言识别：**

| 用户输入 | Bot 代发的命令 |
|---|---|
| 查汇率 / 系统汇率 | `/hl` |
| 查余额 / 余额查询 | `/cxye` |
| 查成功率 / 代收成功率 | `/cgl` |
| 查代收 + 单号 | `/cxds 单号` |
| 回调代收 + 单号 | `/htds 单号` |
| 查提现 + 单号 | `/cxdf 单号` |
| 回调提现 + 单号 | `/htdf 单号` |
| 模拟回调 + 单号 | `/mnds 单号` |
| utr查询 + 号码 | `/utr 号码` |

**交互式问询流程：**

- 发纯单号 → Bot 弹出意图菜单（1 查代收 / 2 回调代收 / 3 查提现 / 4 回调提现）
- 发 "查一下" → Bot 弹出功能菜单（1 汇率 / 2 余额 / 3 代收 / 4 提现 / 5 成功率）
- 发 "查订单 + 单号" → Bot 弹出意图菜单

#### 3.1.2 四层触发决策

Bot 不会回复群里所有消息，通过四层判断决定是否回复：

| 层级 | 说明 | 触发条件 |
|---|---|---|
| L1 规则触发 | 关键词 / 订单号 / @本账号 / 图片 | 含"订单""查""通道"等关键词，或 @wookfaith，或 8 位以上纯数字 |
| L2 语义触发 | AI 判断置信度 | 数字选项回复给 0.95 高置信度 |
| L3 上下文过滤 | 冷却 / 闲聊过滤 | 同一内容 30s 内不重复回复 |
| L4 追问/会话窗口 | 回复链 / 追问上下文 | 用户回复 Bot 消息，或 Bot 回复后 45 分钟内追问 |

#### 3.1.3 图片识别（Vision / OCR）

用户发送付款截图，Bot 通过 GLM-4V-Flash 识别图片内容（银行类型、交易号、金额等），结合 AI 给出订单状态回复。支持 EasyPaisa / JazzCash 等多种支付渠道截图。

#### 3.1.4 通道/额度查询

用户问通道状态、额度信息时，Bot 根据配置中的额度规则回复。触发词包括：通道、额度、限额、EP、JC、介绍一下。

#### 3.1.5 多语言支持

Bot 根据用户消息语言自动回复对应语言：中文、英文、阿拉伯语、土耳其语、葡萄牙语、法语、德语、俄语等。

#### 3.1.6 回复分条发送

长回复自动拆成多条短消息（每条 ≤120 字符），模拟真人打字节奏，随机延迟 0.35s ± 0.15s。

#### 3.1.7 情绪增强

Bot 在回复中适当添加表情符号，增加亲和力（可配置开关、最大数量、最小消息长度）。

#### 3.2.2 配置快照与回滚

每次配置变更自动保存快照到 `config/snapshots/` 目录，支持任意版本回滚。

#### 3.2.3 审计日志

所有配置变更操作记录到 SQLite 数据库（`audit.db`），包含操作类型、目标、变更前后值、操作者、时间戳。支持关键词模糊搜索。

#### 3.2.4 配置导入/导出

通过 Web UI 可一键导出全部配置为 ZIP 包，或导入 ZIP 包覆盖配置。

### 3.3 运维管理功能（P6-P8）

#### 3.3.1 国际化（i18n）

**Bot 消息 i18n：** Bot 的系统消息（GXP 超时、查询结果、热重载通知等）支持中/英文，基于每个群的语言偏好自动切换。

**Web UI i18n：** 管理面板支持中/英文切换，语言偏好通过 Cookie 持久化。

#### 3.3.2 定时任务调度器

纯 `asyncio` 实现的轻量级调度器，可定时向指定群发送 GXP 命令（如每小时自动查汇率）。

#### 3.3.3 用户行为分析

`EventTracker` 记录用户命令事件、响应时间等数据到 SQLite。Web UI 提供数据分析页面，包含命令使用频率图表、每小时消息趋势、活跃用户排行、响应时间分布。

#### 3.3.4 通道告警

自动解析 `/cgl` 命令返回的成功率数据，低于阈值时向管理群/管理员发送告警通知。

#### 3.3.5 多 Bot 协同

支持按群分配不同 Bot 实例，同一程序运行多个 session 时互不干扰。

#### 3.3.6 分级权限（RBAC）

三级角色体系：

| 角色 | 权限 |
|---|---|
| **super_admin** | 全部权限（含回滚、批量修改、导入导出） |
| **admin** | 话术/通道增删改、审计查看 |
| **operator** | 仅查看配置和审计 |

#### 3.3.7 对话上下文持久化

用户对话上下文自动持久化到 SQLite（`bot.db`），重启后恢复。LRU 缓存 + TTL 自动清理过期数据。

#### 3.3.8 配置版本对比

Web UI 提供快照对比页面，可选择两个快照（或一个快照与当前配置）对比差异，以 unified diff 格式高亮显示。

#### 3.3.9 安全消息模板引擎

`str.format_map` + `SafeFormatDict` 实现安全变量插值，缺失变量不报错而是保留原样占位符。

#### 3.3.10 实时日志查看

环形缓冲区 + SSE（Server-Sent Events）实时推流日志到 Web UI，支持暂停/继续、清空、连接状态指示。

#### 3.3.11 插件系统

从 `plugins/` 目录动态加载自定义 Python Skill 模块，支持热重载。

#### 3.3.12 消息限流与防刷

令牌桶算法实现三级限流：全局 / 每群 / 每用户，支持突发流量。LRU 自动淘汰过期桶。

#### 3.3.13 AI 对话质量监控

自动检测 6 类异常回复：

| 异常类型 | 说明 |
|---|---|
| `too_short` | 回复过短（<5 字符） |
| `too_long` | 回复过长（>2000 字符） |
| `repeated` | 近 N 条内与前回重复 |
| `garbled` | 包含乱码/控制字符 |
| `slow_response` | 响应超 30 秒 |
| `identity_leak` | AI 自曝身份（"作为AI语言模型"） |

#### 3.3.14 配置热迁移工具

升级版本时自动检查并补充缺失配置字段（保留已有值），迁移前自动备份。

#### 3.3.15 通道健康度评分

多维度加权评分（0-100 分）：

| 维度 | 权重 | 说明 |
|---|---|---|
| 状态 | 50% | active=100, maintenance=50, disabled=0 |
| 告警 | 20% | 24h 告警次数越多扣分 |
| 配置完整性 | 15% | 必填字段覆盖度 |
| 更新新鲜度 | 15% | 上次更新距今越久越低 |

#### 3.3.16 优先级消息队列

`asyncio.PriorityQueue` 实现，4 级优先级（CRITICAL > HIGH > NORMAL > LOW），GXP/配置命令为 HIGH，闲聊为 LOW。支持背压控制：队列利用率超 80% 时自动丢弃低优先级消息。

#### 3.3.17 Webhook 通知

配置变更时自动推送到外部系统，支持 JSON / Slack / WeChat / Telegram 格式。

### 3.4 AI 回复策略引擎（S1-S2 + 高级分析）

#### 3.4.1 多策略配置系统

5 种预设策略，每种独立控制 AI 调用参数：

| 策略 ID | 名称 | 特点 |
|---|---|---|
| `S1_instant_greet` | 闪电问候 | 跳过 AI，模板秒回，延迟 <1s |
| `S2_quick_answer` | 快速应答 | 低温度(0.3)，短输出(256 token)，精确稳定 |
| `S3_standard` | 标准对话 | 平衡模式，温度 0.7，512 token |
| `S4_deep_support` | 深度支持 | 高上下文(10轮)，长输出(1024 token)，用于投诉 |
| `S5_silent_observe` | 静默观察 | 仅 10% 概率回复，闲聊/水群用 |

**策略参数：** `temperature`、`max_tokens`、`context_rounds`、`reply_probability`、`skip_ai`、`model`（可覆盖全局模型）、`enabled`

**意图-策略映射：** 每个识别意图自动匹配对应策略（`intent_strategy_map`），可在线修改。

**持久化：** 策略存储在独立 `config/reply_strategies.yaml` 文件中，支持 mtime 热更新，修改后无需重启。原子写入（`.yaml.tmp` + rename）防止写入中断损坏。

#### 3.4.2 A/B 灰度测试

对特定意图分流多个策略变体，对比效果：

- **一致性哈希：** MD5(`"{intent}:{user_id}"`) % 100 确保同一用户始终进入同一分桶
- **权重配置：** 每个变体指定 weight 百分比（如 70/30 分流）
- **策略间对比：** 如 `S2_quick_answer` vs `S3_standard` 对比响应速度和追问率
- **模型间对比：** 创建两个策略仅改 `model` 字段（如 `claude-4.6-oups-high` vs `claude-4.6-oups-high-reasoner`），分流即为模型 A/B

配置示例：
```yaml
ab_tests:
  order_query:
    enabled: true
    variants:
      - strategy_id: "S2_quick_answer"
        weight: 70
      - strategy_id: "S3_standard"
        weight: 30
```

#### 3.4.3 策略效果追踪

`StrategyTracker` 记录每次 AI 回复的效果指标到 SQLite（`strategy_events.db`）：

| 字段 | 说明 |
|---|---|
| `strategy_id` | 使用的策略 ID |
| `intent` | 识别的用户意图 |
| `model_id` | 实际调用的 AI 模型名称 |
| `response_ms` | 响应耗时（毫秒） |
| `used_ai` | 是否调用了 AI API |
| `follow_up` | 用户是否追问（事后回填） |
| `follow_up_intent` | 追问的意图 |
| `session_id` | 所属会话 ID |

**事后回填机制：** 当用户在 5 分钟内再次发送消息，系统回溯标记前一条事件的追问状态，无需复杂定时器。

#### 3.4.4 会话级追踪

同一用户在同一对话中，30 分钟内的连续交互归为一个会话（`session_id`）。提供：

- 会话总数、平均轮次、整体解决率
- 按策略的会话统计（各策略解决率、平均轮次）
- 按首次意图的会话分布

#### 3.4.5 质量评分与智能诊断

**综合质量评分（0-100）：** 融合多维被动指标：

| 维度 | 权重 | 说明 |
|---|---|---|
| 响应速度 | 20% | <1s=满分，>5s=0 分 |
| 一次解决率 | 35% | 静默率越高越好（用户不追问=解决） |
| 同意图追问惩罚 | 30% | 同意图追问率越高扣分越多 |
| API 效率 | 15% | 模板命中率越高越好（节省 API 调用） |

**智能诊断（Advisor）：** 自动检测异常并给出建议：

| 告警级别 | 触发条件 | 建议示例 |
|---|---|---|
| critical | 同意图追问率 > 50% | "考虑增加 context_rounds 或换用更强模型" |
| warn | 平均响应 > 3000ms | "降低 max_tokens 或使用更快模型" |
| info | 追问率 > 40% | "检查回复完整度" |

**参数微调建议：** 根据指标异常自动生成具体参数调整建议（如 `context_rounds: 2 → 4`），支持一键应用。

#### 3.4.6 Auto-Pilot 自动策略降级

系统持续监测策略质量评分，当某策略评分持续低于阈值时自动将其负责的意图切换到更优策略：

- **触发方式：** 每 100 条消息自动检查一次（可配置）
- **安全阈值：** 评分 < 30 且样本 ≥ 10 且与最优策略差距 > 15 分
- **操作：** 仅重映射意图，不禁用策略；所有操作记录审计日志

#### 3.4.7 多模型 A/B 对比

策略支持 `model` 字段覆盖全局模型。在仪表盘提供：

- **模型效果卡片：** 每个模型的回复数、平均响应时间、追问率、静默率
- **模型×策略交叉表：** 同一模型在不同策略下的表现对比
- 所有已有分析（质量评分、Auto-Pilot、参数建议）自动包含模型维度

#### 3.4.8 用户分群分析

按分析窗口内的活跃度动态分群：

| 分群 | 条件 | 分析维度 |
|---|---|---|
| 高频用户 (heavy) | ≥10 条交互 | 用户数、事件数、加权追问率、偏好策略 Top 5 |
| 中频用户 (moderate) | 3-9 条交互 | 同上 |
| 低频用户 (light) | 1-2 条交互 | 同上 |

追问率采用事件数加权平均，避免低频用户异常值干扰统计。

#### 3.4.9 数据导出与清理

- **导出：** 策略追踪数据支持 CSV 和 JSON 格式导出
- **数据保留：** 可配置策略事件保留天数（默认 30 天）和通用事件保留天数（默认 90 天）
- **清理：** 一键清理过期数据并自动 VACUUM 回收磁盘空间

#### 3.4.10 全局术语注释系统

后台所有英文术语、参数名、指标名支持鼠标悬停显示中文注释和使用说明。目前收录 35+ 个术语。支持开关按钮（`?`）全局启用/关闭注释。

### 7.2 常用排查命令

```powershell

Select-String -Path logs\app.log -Pattern "ERROR"

# Linux — 查看最近日志
tail -50 logs/app.log

# Linux — 实时监控
tail -f logs/app.log

# Linux — 搜索错误
grep "ERROR" logs/app.log
```

### 7.3 关键日志标记

| 标记 | 含义 |
|---|---|
| `[群消息] 收到一条群消息 chat_id=xxx` | 群消息 handler 被调用（若此条都没有，说明未连接或不在群内） |
| `[群组监控] 收到消息` | 通过前置过滤后进入监控逻辑的群消息 |
| `四层触发决策: 回复` | 决定回复 |
| `四层触发决策: 不回复` | 决定不回复 |
| `已代发 gxp 命令:` | Bot 转发了 GXP 命令 |
| `已回复消息:` | Bot 发出了回复 |
| `跳过机器人发送的消息` | 正确跳过了 gxp_notify_bot 的消息 |
| `rate_limit blocked` | 消息被限流丢弃 |
| `alert_success_rate` | 成功率告警触发 |

### 7.4 Web 实时日志

启用 Web 管理面板后，可在浏览器中访问 `/logs` 页面查看实时日志流，无需登录服务器。

### 7.5 群消息不回复排查

若群里发消息但 AI 不回复，按下面顺序排查：

**1）确认是否收到群消息**

- 看 `logs/app.log`：有人发消息时是否出现 **`[群消息] 收到一条群消息 chat_id=xxx`**（最先打的一条）。
  - 若**连这条都没有**：说明 Telegram 客户端未真正连上（例如启动时报 `database is locked`、或旧进程占用 session）。处理：关掉所有占用端口的 Python 进程后单进程重启；确认启动日志里有「登录用户: xxx」「✅ Telegram客户端已启动」。
  - 若有 **`[群消息] 收到一条群消息`** 但没有 **`[群组监控] 收到消息`**：说明消息在「前置过滤」被跳过。此时日志会出现一条 **`[群消息] 跳过: xxx`**，常见原因：
    - **跳过: 自身发送的消息** — 当前登录账号（如 @wookfaith）自己发的消息不会处理，请用**另一个账号**在群里发消息测试。
    - **跳过: 无文本/图/语音** — 消息类型不支持（如纯表情、入群通知等），发**文字或带说明的图片**即可。
    - **跳过: 限流** / **多 Bot 路由** / **配置的不回复发送者** — 按提示检查配置或换发送者。
- 若有 **`[群组监控] 收到消息`** 但依然不回复，继续下面步骤。

**2）看触发决策日志**

- 在日志中搜 **`四层触发决策: 不回复`** 或 **`跳过未触发消息`**。
- 出现说明：消息已收到，但**触发逻辑**判定“不回复”。当前逻辑下，只有满足以下之一才会回复：
  - **回复我们**：用户回复的是 Bot 自己发的一条消息；
  - **@ 本账号**：消息里 @ 了当前登录账号（如 @wookfaith）；
  - **L1 触发**：消息含关键词（如 **你好、在吗、hello、hi** 等打招呼，或 订单、查、客服、通道、汇率）、或带图+文字、或符合订单号格式；
  - **L2 通过**：语义判定为与工作相关且置信度够高（如追问、会话窗口内）。

**3）如何让 Bot 更容易回复（便于测试/使用）**

- 在群里 **@ 你的 Bot**（例如 @wookfaith）再发问题，一般会触发回复。
- 或发带**业务关键词**的消息，如：`查订单`、`客服`、`通道状态`、`汇率`。
- 或**先让 Bot 发一条消息**，然后用户**回复那条消息**，后续在该会话窗口内也更容易触发。
- 若仅做连通性测试，可临时把 `config.yaml` 里 `telegram.group_reply.mode` 改为 **`always`**（每条群消息都尝试回复），测完改回 `mention_or_keyword` 等以免刷屏。

**4）其他可能原因**

- **策略静默跳过**：若日志出现 **`策略 S5_silent_observe 静默跳过 (概率 0.1)`**，说明该条消息被识别为闲聊（small_talk），且当前策略为「静默观察」仅 10% 概率回复。**被 @ 本账号时已改为一律回复**；未 @ 的「你好/在吗」等仍可能被概率跳过。若希望闲聊也常回复，可在 Web 策略页将 small_talk 映射改为 S3_standard，或将 S5 的回复概率调高。
- **限流**：`rate_limit.enabled: true` 时，同一用户/群过于频繁会被拒绝，日志中可见 `[限流] 拒绝`。
- **冷却**：同一用户或同一内容在冷却时间内不会重复回复，属正常防刷行为。

## 八、安全防护

### 8.1 消息安全

| 防护 | 说明 |
|---|---|
| 自身消息过滤 | Bot 不处理自己发送的消息（防自我循环） |
| Outgoing 过滤 | 跳过标记为 outgoing 的消息 |
| Bot 账号过滤 | 跳过 `is_bot=True` 的消息 |
| 用户名黑名单 | `no_reply_sender_usernames` 中的用户不回复 |
| 启动时间过滤 | 启动前的旧消息全部丢弃（防历史消息重放） |
| 机器人回复过滤 | "查询失败""查询成功"等特征消息不处理 |
| 私聊关闭 | `process_private: false`，不处理私聊 |
| 冷却机制 | 相同内容 30s / 全局 5s 冷却 |

### 8.2 限流防刷

- 三级令牌桶：全局（30 容量/2 QPS）、每群（20/1）、每用户（10/0.5）
- 超限消息静默丢弃，不回复不告警
- LRU 自动淘汰闲置桶，内存可控

### 8.3 权限控制

- RBAC 三级角色：super_admin > admin > operator
- 危险操作（批量修改、回滚、导入）仅 super_admin 可用
- Web API 支持 Bearer Token 认证
- Web 管理面板 Session 加密

### 8.4 数据安全

- 日志脱敏：消息体截断、连续数字替换为 `***`
- AI 身份泄露检测：自动告警 "作为AI语言模型" 等措辞
- 配置文件不含密码明文（API 密钥除外，需自行保护文件权限）
