# 当前已实现的逻辑与功能总结（供其他 AI 审阅与建议）

本文档汇总当前 Telegram 客服机器人（Camille）的**已实现逻辑与功能**，便于其他 AI 或同事快速理解并给出改进建议。不涉及未实现或纯规划内容。

---

## 一、项目角色与目标

- **身份**：Camille，客户订单查询与通道服务客服（Telegram 用户端，基于 Pyrogram）。
- **主业务**：客户订单查询（订单是否收到、是否匹配、是否处理成功）；兼做通道状态、额度、成功率、代付未到账等咨询。
- **目标**：在群内**尽量不漏回**客户消息，同时控制误回；回复内容符合客服 SOP，不编造订单信息，客户用什么语言就问什么语言回。

---

## 二、技术栈与入口

- **运行**：Python，入口 `main.py`，配置 `config/config.yaml`、`config/trigger_rules.yaml`。
- **消息处理**：群消息/私聊 → 入队 → `_process_message_async`（`src/client/telegram_client.py`）。
- **是否回复群消息**：由 `_should_reply_to_group_message(message)` 统一判定，返回 True 才进入 AI 生成回复并发送。

---

## 三、回复判定逻辑（核心：不漏回 + 可控误回）

**执行顺序**（任一步为 True 即回复，不再往后）：

| 顺序 | 规则 | 说明 |
|------|------|------|
| 1 | **回复链（Reply-to）** | 当前消息带有 `reply_to_message`，且被回复的那条消息的发送者 = 本机器人 → **一律回复**。用户明确「回复我们」即视为在跟我们说话。 |
| 2 | **四层触发**（若配置开启） | L1：关键词 / 图片+文字 / 订单号格式 / @提及 → 通过则回复；未通过则 L2 语义（与订单/通道/客服相关度）→ L3 上下文（连续消息、@、情绪等）。 |
| 3 | **追问上下文** | 四层未触发时：拉取该群**最近 K 条**消息（K 可配置，默认 10）；若其中**任一条**为我们所发，且当前消息**像追问**（含 `?` `吗` `呢` 或追问/查单多语言关键词，且长度 ≤ max_text_length）→ 判定为应回复。用于解决「中间有人插话」导致的漏回。 |
| 4 | **Legacy 关键词**（未开四层时） | 使用 `telegram.group_reply.keywords` 做关键词匹配，命中则回复。 |
| 5 | **追问上下文**（Legacy 路径） | 与上同理，在 Legacy 未命中后仍做「最近 K 条内有我们 + 当前像追问」判断。 |
| 6 | **会话窗口 + L2 兜底** | 前面都未通过时：若该 (chat_id, user_id) 在**我们最近回复该用户后的 T 分钟**内（默认 45 分钟），且消息长度在 [min, max] 内，则**仅跑 L2 语义**（不跑 L1/L3）；L2 判定应回复则回复。L2 调用带超时（如 6 秒），超时/异常不抛错，仅返回不回复。 |

**会话窗口记录方式**：每次我们**成功发送群回复**后，记录「被回复消息的发送者」的 (chat_id, user_id) 与当前时间戳；仅记录我们「针对」的那位用户，不在群内泛回复时把所有人记入窗口。存储为内存 dict，写入时顺带清理超过 2×T 的 key，避免无限增长。

**配置集中位置**：`config/config.yaml` → `telegram.reply_logic`（reply_chain、follow_up、session_window、l2_fallback 的开关与参数）。

## 四、多语言查单与同语言回复

- **触发**：L1 关键词（`trigger_rules.yaml` 的 `all_keywords`）已包含中/英/阿拉伯/土耳其/葡萄牙/西/法/德/意/俄等查单/订单/状态相关词（如 订单、查、回调、order、inquiry、pedido、consulta、طلب、sipariş、commande、Bestellung、заказ 等），便于多语言客户触发回复。
- **回复语言**：系统提示词（`ai.system_prompt`）中明确约定：**客户用什么语言发消息，就用同一语言回复**；查单、订单状态、通道说明等与客户语言一致。覆盖中东、巴西、欧美常用语言。

---

## 五、图像与订单凭证（识图作为查单依据）

- **图像理解**：支持「图片+文字」触发；图像内容通过 **Vision（智谱 GLM-4V）** 或 **OCR 兜底** 得到文本，作为上下文给 AI。
- **群内最近图**：当用户消息触发「查单/订单」相关且未带图时，会拉取该群**最近一张图**做识图，并将结果作为「最近聊天中的凭证」传给 AI，避免「图在上一句」导致的漏看。
- **机器人通知**：可拉取群内指定 bot（如 gxp_notify_bot）的最近消息，作为订单/回调相关上下文传给 AI。
- **查单依据**：系统提示词与 Vision 提示中约定：从图中识别**银行/账单类型**与**唯一识别依据**（交易号、订单号、参考号等），仅依据识图或机器人通知中的真实内容回复，**不编造**金额、单号、商品、地址。
- **话术**：若上下文中已有识图得到的交易号/订单号等，回复「已查询到/已根据凭证…请耐心等待系统回调」，**不再重复索要订单号**；仅当确实没有时才请用户提供订单号或截图。

## 六、客服人设与 SOP（系统提示词）

- **人设**：Camille，真人客服语气；被问是否 AI 时答「我是客服 Camille」；不用「作为一个 AI」等表述。
- **原则**：先确认问题类型再针对性回复；回复简洁、完整句，避免单独「稍等」「好的」；专业且带人情味。
- **订单状态类**：以「是否已有识图/凭证」区分——有则确认收到+处理中+不重复要单号；无则请提供订单号或截图。所有订单回复仅依据识图或机器人通知内容，不编造。

详见 `docs/SOP_PERSONA_AND_FLOW.md`、`docs/WORKFLOW_BANK_ORDER_LOOKUP.md`。

## 八、已实现的优化点（相对初版方案）

- 回复链在**未开四层**时也生效；Legacy 路径同样做**追问上下文**与 **L2 兜底**。
- 追问上下文从「上一条是否为我们」改为「**最近 K 条内任一条为我们**」，减少插话导致的漏回。
- 会话窗口**只记录被回复用户**，且记录时**惰性清理**过期 key。
- L2 在**无 AI 客户端**时仍用规则置信度（关键词）跑，便于 L2 兜底无额外依赖即可用；L2 规则关键词已扩展多语言与查单/订单/通道相关词。
- L2 兜底**超时与异常不抛错**，仅打 debug 日志并返回不回复。

## 九、希望其他 AI 重点给建议的方向

1. **回复判定**：当前顺序与各层阈值（K、T 分钟、L2 置信度、长度范围）是否合理？是否有明显漏回/误回场景可针对性优化？
2. **多语言**：触发词与「同语言回复」的约定是否足够？是否有推荐的多语言扩展（语种或关键词）？
3. **会话窗口与 L2 兜底**：仅对「我们近期回复过的用户」做 L2 兜底是否合适？是否需要「高优先级群」或按群配置放宽？
4. **识图与查单**：Vision + OCR + 最近图 + 机器人通知的配合是否合理？查单依据与话术是否有更清晰的 SOP 或数据结构建议？
5. **可观测与运维**：是否建议对「回复链/追问/会话窗口+L2」触发的回复打标签或指标，便于统计漏回率与各规则贡献？是否有更好的可观测方案？
6. **安全与边界**：是否有潜在滥用（如刷回复、跨群误判）、隐私或性能风险，以及建议的防护措施？
