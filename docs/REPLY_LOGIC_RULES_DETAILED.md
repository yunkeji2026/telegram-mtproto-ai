# 回复逻辑细化规则说明（先 1+3，再做 2+4）

本文档约定「是否回复群消息」的判定顺序与每条规则的输入、输出、边界情况，便于按阶段实现且不遗漏客户消息。

---

## 一、整体判定顺序（群消息入口）

每条群消息进入「是否回复」判断时，**按下列顺序依次检查**，任一环节判定为「应回复」则立即进入回复流程，不再往后判断；若全部未通过则「不回复」。

1. **回复链优先（方案 1）** — 最高优先级  
2. **四层触发** — L1 → L2 → L3（现有逻辑）  
3. **追问上下文（方案 3 扩展版）** — 四层未触发时的兜底  
4. **（阶段二）会话窗口 + L2 兜底（方案 2 + 4）** — 可选，见第四节  

下面分阶段写清每条规则的细则。

---

## 二、阶段一：先做 1 + 3

### 2.1 方案 1：回复链（Reply-to）优先

**目的**：用户点击「回复」我们某条消息时，视为明确在对我们说话，优先回复，避免漏回。

**输入**：

- 当前消息对象 `message`
- 需能读取：`message.reply_to_message`、`message.reply_to_message.from_user.id`（或被回复消息的发送者标识）
- 本机器人账号 ID（如 `self.user_info.id`）

**规则**：

1. 若当前消息**没有** `reply_to_message`（用户未点击「回复」），则本规则**不适用**，继续下一环节（四层触发）。
2. 若有 `reply_to_message`：
   - 取被回复消息的发送者 ID。
   - 若**被回复消息的发送者 = 本机器人 ID**，则**判定为应回复**，不再检查 L1/L2/L3 与追问。
3. 若被回复消息的发送者 ≠ 本机器人（用户是在回复别人），则本规则**不适用**，继续下一环节。

**边界与配置建议**：

- **纯表情/单字**：若希望减少噪音，可配置「回复我们时，若消息内容仅为 1 个表情或 1～2 个无意义字符则可不回复」；默认建议仍回复，避免漏掉「嗯？」「？」这类追问。
- **转发/回复链**：只判断「直接回复的那条」的发送者是否为本机器人，不追溯更上层。
- **顺序**：此规则必须在四层触发**之前**执行，作为最高优先级。

**输出**：布尔值——是否因「回复我们」而应回复。

---

### 2.2 方案 3：放宽追问上下文（最近 K 条内是否有我们）

**目的**：当前实现只看了「上一条是否为我们所发」，中间一旦有人插话就会漏。改为在「最近 K 条消息」内只要出现过我们，且当前消息像追问，就回复。

**输入**：

- 当前消息对象 `message`、当前消息文本 `current_text`（或 caption）
- 当前群最近若干条消息（由 `get_chat_history(chat_id, limit=K)` 得到，K 建议 8～12，可配置）
- 本机器人 ID
- 追问判定用：疑问符集合、追问/查单关键词集合（与现有多语言追问词一致或扩展）

**规则**：

1. **仅当**前面「回复链」与「四层触发」都**未**判定为应回复时，才执行本规则。
2. 按时间从新到旧遍历最近 K 条消息（注意：第一条一般为当前消息本身或紧邻最新）：
   - 对每条消息判断：发送者是否为本机器人。
   - 若在**前若干条内**（建议排除第 1 条后，在 2～K 条内）**任一条**为本机器人所发，则记「该会话最近曾出现我们」。
3. 若「该会话最近曾出现我们」为真，再判断**当前消息**是否像追问：
   - **疑问语气**：消息中含配置的疑问符（如 `?` `？` `吗` `呢` `啊` `呀` 等）。
   - **或**消息中含追问/查单关键词（与现有 `_should_reply_by_follow_up_context` 中的多语言关键词一致，可复用并略扩展）。
   - 可选：消息长度在合理范围（如 1～200 字符），避免超长转发/刷屏触发追问逻辑。
4. 若「最近 K 条内有我们」且「当前消息像追问」同时满足，则**判定为应回复**。
5. 若 K 条内没有我们，或当前消息不像追问，则本规则**不适用**，判定为不回复（或交给阶段二的会话窗口 + L2，若已实现）。

**边界与配置建议**：

- **K 的取值**：建议 8～12。过小（如 5）仍易被插话打断；过大（如 20）在热闹群里可能误把无关讨论当追问。可放在 `config` 中，如 `follow_up.lookback_count: 10`。
- **「当前消息」在历史中的位置**：若 `get_chat_history` 返回的第一条就是当前这条，则从第二条开始找「我们」；若接口不包含当前消息，则从第一条开始找。实现时需与当前 API 行为一致。
- **追问关键词**：与现有实现保持一致并略扩展即可（中英及中东/巴西/欧美多语种），无需在本规则内再列一遍，由配置或常量统一维护。

**输出**：布尔值——是否因「追问上下文」而应回复。

---

### 2.3 阶段一后的整体流程小结

- 群消息到达 →  
  **1）回复链**：回复我们？→ 是则回复，结束。  
  **2）四层触发**：L1/L2/L3 是否触发？→ 是则回复，结束。  
  **3）追问上下文（扩展）**：最近 K 条内有我们且当前像追问？→ 是则回复，结束。  
- 否则 → 不回复（或进入阶段二，若已实现）。

## 三、阶段二：再做 2 + 4

### 3.1 方案 2：会话窗口（同一用户 + 时间）

**目的**：对「我们近期回复过的用户」，在时间窗口内放宽判定，减少因其换说法、没打关键词而漏回。

**输入**：

- 当前消息：`chat_id`、`user_id`（发送者）、消息时间。
- 持久化或内存结构：**按 (chat_id, user_id) 记录**  
  - 我们最近一次**回复该用户**的时间（或该用户最近一次收到我们消息的时间）；  
  - 可选：该用户最近 N 条消息的时间列表（用于判断「连续多条未回」）。
- 配置：时间窗口 `T_reply`（如 30～60 分钟）、可选「连续消息条数」阈值。

**规则**：

1. **仅当**「回复链」「四层触发」「追问上下文」都**未**判定为应回复时，才考虑本规则。
2. 查询：该 (chat_id, user_id) 在**过去 T_reply 分钟内**是否收到过我们的回复（或我们是否在该窗口内回复过他）。
3. 若**是**（在会话窗口内）：
   - 可选 A：直接进入 **L2 兜底**（方案 4），即对本条消息单独跑一次 L2，L2 通过则回复。
   - 可选 B：放宽追问上下文——将「最近 K 条内有我们」改为「最近 K 条内有我们 **或** 该用户在 T_reply 内被我们回复过」，再结合追问关键词/疑问符判定。
4. 若**否**（该用户不在会话窗口内），本规则不改变判定，仍为不回复（或仅交给 L2 兜底的条件判断，见 3.2）。

**边界与配置建议**：

- **记录时机**：每次我们**成功向该群该用户所在会话发送回复**时，更新 (chat_id, user_id) 的「我们最后回复时间」。若回复是群内一条 @ 或非 @ 的群消息，可用该消息的 chat_id + 被回复者的 user_id 或「当前消息的 user_id」更新。
- **存储**：可用内存字典 + 定期清理过期 key；若需重启不丢，可落盘或 Redis，键为 `chat_id:user_id`，值为时间戳。
- **T_reply**：建议 30～60 分钟，可配置。

**输出**：本规则不直接输出「回复/不回复」，而是**决定是否进入 L2 兜底**或**是否放宽追问条件**（与方案 4 或 3 配合）。

---

### 3.2 方案 4：L2 语义兜底（未触发 L1 时单独跑 L2）

**目的**：部分消息未命中 L1，但内容明显是查单/订单/通道/投诉，用 L2 语义判断一次，减少漏回。

**输入**：

- 当前消息文本（及可选：当前用户是否在「会话窗口」内、消息长度等）。
- L2 现有能力：输入文本，输出是否与订单/通道/客服相关及置信度等。

**规则**：

1. **触发条件**（满足其一即可进入 L2 兜底，否则不跑 L2）：
   - **会话窗口内**：该 (chat_id, user_id) 在 T_reply 内被我们回复过（方案 2 判定为在窗口内）；或
   - **可选**：消息长度在 [min_len, max_len] 之间（如 2～300 字符），避免对纯表情、超长转发全文无意义调用 L2；或
   - **可选**：该群被配置为「高优先级群」，所有未触发 L1 的消息都走一次 L2 兜底。
2. 对满足条件的消息**仅调用 L2**（不再跑 L1），若 L2 判定「与订单/通道/客服相关且应回复」，则**判定为应回复**。
3. 超时与降级：L2 调用设超时（如 5～8 秒）；超时或失败时视为不回复，不打乱主流程。

**边界与配置建议**：

- **成本与延迟**：L2 为模型或语义接口，可仅对「未触发 L1 且满足上述条件」的少部分消息调用，并限制并发。
- **与四层关系**：正常四层流程是 L1 通过后才跑 L2；这里的「L2 兜底」是**绕过 L1**，只跑 L2，用于兜底漏判。

**输出**：布尔值——是否因 L2 兜底而应回复。

---

### 3.3 阶段二后的整体流程小结

在阶段一基础上，在「追问上下文仍未通过」之后增加：

- **会话窗口**：该用户是否在 T_reply 内被我们回复过？  
  - 若是，则**可选**放宽追问条件或**进入 L2 兜底**。  
- **L2 兜底**：在满足「会话窗口内 / 长度范围 / 高优先级群」等条件时，对当前消息跑一次 L2；L2 通过则回复。  
- 若仍不通过 → 不回复。

## 四、配置项建议汇总

便于在 `config.yaml` 或 `trigger_rules.yaml` 中统一管理：

| 配置项 | 说明 | 建议默认 |
|--------|------|----------|
| `reply_chain.reply_to_me_always_reply` | 用户「回复我们」时是否一律回复 | true |
| `reply_chain.skip_if_only_emoji_or_short` | 回复我们时，仅表情或极短内容是否跳过 | false（先不漏） |
| `follow_up.lookback_count` | 追问上下文：最近 K 条消息 | 10 |
| `follow_up.max_text_length` | 追问判定：消息最大长度（字符） | 200 |
| `session_window.enabled` | 是否启用会话窗口（方案 2） | true（阶段二） |
| `session_window.reply_within_minutes` | 我们回复该用户后的时间窗口 T_reply（分钟） | 45 |
| `l2_fallback.enabled` | 是否启用 L2 兜底 | true（阶段二） |
| `l2_fallback.only_in_session_window` | 是否仅对「会话窗口内」用户跑 L2 兜底 | true |
| `l2_fallback.text_min_len` / `text_max_len` | L2 兜底消息长度范围 | 2, 300 |
| `l2_fallback.timeout_seconds` | L2 兜底调用超时 | 6 |

## 五、与现有代码的对应关系（便于实现）

- **回复链（方案 1）**：在 `_should_reply_to_group_message` 最开头增加对 `message.reply_to_message` 的检查；若回复我们则直接返回 True。
- **追问上下文（方案 3）**：在 `_should_reply_by_follow_up_context` 中，将「上一条是否为我们」改为「最近 K 条内是否任一条为我们」，再结合现有追问关键词/疑问符逻辑。
- **会话窗口（方案 2）**：在回复成功发送时写入 (chat_id, user_id, timestamp)；在判定流程中，四层与追问都未通过时，查询该用户是否在 T_reply 内，若在则进入 L2 兜底或放宽追问。
- **L2 兜底（方案 4）**：在 `_should_reply_to_group_message` 中，当四层未触发且追问未触发（且可选：会话窗口内）时，调用一次 L2；L2 返回应回复则返回 True。

按「先 1+3，再做 2+4」的顺序实现并上线，可先交付减少漏回的主要收益，再通过 2+4 进一步收口。

## 六、阶段一实施结果与优化（已完成）

### 6.1 已实现内容

- **方案 1 回复链**：在 `_should_reply_to_group_message` 最前增加 `_should_reply_by_reply_chain(message)`；若 `message.reply_to_message` 存在且被回复消息的发送者为本机器人，则直接判定应回复。
- **方案 3 追问扩展**：`_should_reply_by_follow_up_context` 改为在**最近 K 条**（配置 `follow_up.lookback_count`，默认 10）内任一条为我们所发即记「曾出现我们」，再结合当前消息的疑问符/追问关键词判定；K 限制在 5～20 之间防止配置误设。
- **配置**：在 `config/config.yaml` 的 `telegram.reply_logic` 下增加 `reply_chain`（enabled、reply_to_me_always_reply、skip_if_only_emoji_or_short）与 `follow_up`（enabled、lookback_count、max_text_length）。

### 6.2 在现有方案基础上的优化

| 优化点 | 说明 |
|--------|------|
| **回复链在 legacy 下也生效** | 无论是否启用四层触发，只要用户「回复我们」都会先走回复链；legacy 模式下同样减少漏回。 |
| **Legacy 下也做追问兜底** | 未启用四层时，legacy 关键词未命中后仍会执行追问上下文判断，与四层模式一致，减少漏回。 |
| **reply_to_message.from_user 为空** | 被回复消息可能来自频道等无 `from_user` 的情况，仅当 `from_user` 存在且 `id == 本机` 时才判定为「回复我们」，避免误判。 |
| **skip_if_only_emoji_or_short 的细化** | 当开启「跳过极短」时，仅当内容长度 ≤2 **且** 不含疑问符（如 `?` `？` `吗` `呢`）才跳过，保证「？」「嗯？」仍会回复。 |
| **lookback_count 边界** | K 从配置读取后限制在 [5, 20]，避免配成 0 或过大导致行为异常或性能问题。 |
| **追问与配置解耦** | 追问是否启用、K、最大长度均从配置读取，便于按群或环境调参而无需改代码。 |

### 6.3 代码与配置位置

- 回复链：`src/client/telegram_client.py` — `_should_reply_by_reply_chain`
- 追问上下文：同文件 — `_should_reply_by_follow_up_context`（使用 `reply_logic.follow_up`）
- 入口顺序：同文件 — `_should_reply_to_group_message`：回复链 → 四层 → 追问
- 配置：`config/config.yaml` — `telegram.reply_logic`

## 七、阶段二实施与改进建议（2+4）

### 7.1 实施顺序建议

1. **方案 2 会话窗口**
   - 在**每次成功向群内发送回复**后，记录 (chat_id, user_id, 当前时间戳)。存储可用内存 dict，键 `f"{chat_id}:{user_id}"`，值时间戳；可选定期清理超过 T_reply 的 key。
   - 在 `_should_reply_to_group_message` 中，当「回复链 + 四层 + 追问」均未通过时，查询该 (chat_id, user_id) 在**过去 T_reply 分钟**内是否有记录；若有则标记「在会话窗口内」。
   - 配置：`telegram.reply_logic.session_window.enabled`、`reply_within_minutes`（建议 45）。

2. **方案 4 L2 兜底**
   - 仅当「在会话窗口内」时（或可选：消息长度在 [text_min_len, text_max_len]），对当前消息**单独调用 L2**（不跑 L1）；L2 判定与订单/通道/客服相关且应回复则返回 True。
   - 调用 L2 时设超时（如 6 秒），超时或失败则视为不回复。
   - 配置：`telegram.reply_logic.l2_fallback.enabled`、`only_in_session_window`、`text_min_len`/`text_max_len`、`timeout_seconds`。

### 7.2 建议的改进点（阶段二时一并考虑）

- **记录维度**：会话窗口除 (chat_id, user_id) 外，可增加「我们回复时是否针对该用户」（如 reply_to 该用户或 @ 该用户），仅在此情况下更新该用户的「最后被回复时间」，避免在群内泛回复也把所有在场用户都记入窗口。
- **L2 复用**：L2 兜底尽量复用现有四层里的 L2 接口，仅传入当前消息文本与必要上下文，避免重复实现。
- **降级**：L2 兜底失败或超时时只影响「是否回复」判定，不抛错、不阻塞主流程；可打 debug 日志便于排查。
- **可观测**：对「因回复链 / 追问 / 会话窗口+L2 兜底」触发的回复打不同 log 或标签，便于后续统计漏回率与各规则贡献度。

## 八、阶段二实施结果与优化（已完成）

### 8.1 已实现内容

- **方案 2 会话窗口**
  - 成功发送群回复后，在 `_send_reply` 中调用 `_record_session_reply(chat_id, user_id)`，仅记录**被回复消息的发送者**（即我们针对的那位用户），避免泛回复污染窗口。
  - 存储：内存 dict `_session_reply_ts`，键 `f"{chat_id}:{user_id}"`，值时间戳；在记录时顺带清理超过 2×T_reply 的 key，控制内存。
  - 判定：`_is_in_session_window(chat_id, user_id)` 查该 key 是否在 `reply_within_minutes` 内；超时则删除 key 并返回 False。
  - 配置：`telegram.reply_logic.session_window.enabled`、`reply_within_minutes`（默认 45）。

- **方案 4 L2 兜底**
  - 在「回复链 + 四层/legacy + 追问」均未通过后，若启用且（可选）仅在会话窗口内、消息长度在 [text_min_len, text_max_len]，则 `_should_reply_by_l2_fallback` 调用 `FourLayerTrigger.check_l2_only(text, chat_id, user_id)`，用 `asyncio.wait_for(..., timeout)` 限制耗时；L2 返回应回复则判定为回复。
  - 配置：`l2_fallback.enabled`、`only_in_session_window`、`text_min_len`/`text_max_len`、`timeout_seconds`（默认 6）。
  - L2 复用：`FourLayerTrigger` 新增公有方法 `check_l2_only(message_text, chat_id, user_id) -> (bool, str)`，内部只跑 `_check_l2_semantic_trigger`，不跑 L1/L3。

### 8.2 在现有方案基础上的优化

| 优化点 | 说明 |
|--------|------|
| **只记录被回复用户** | 仅在「我们成功发送回复」时记录，且记录的 user_id 为 `original_message.from_user.id`（被回复的那条消息的作者），符合「我们针对该用户的回复」才更新窗口，避免群内泛回复把所有人记入窗口。 |
| **记录时惰性清理** | 每次写入会话时间戳时，顺带删除超过 2×reply_within_minutes 的条目，避免 dict 无限增长，无需单独定时任务。 |
| **L2 无 ai_client 也可跑** | 去掉「无 ai_client 则 L2 直接返回不可用」的逻辑，无 AI 客户端时仍用规则置信度 `_get_ai_confidence`，使 L2 兜底在未配置 claude-4.6-oups-high 时也能用关键词命中。 |
| **L2 规则关键词扩展** | `_get_ai_confidence` 中业务关键词增加多语言与查单/订单/通道相关词（如 order、inquiry、payment、channel、pedido、consulta、sipariş、commande、statut、查、回调、到账等），提高兜底命中率。 |
| **超时与异常不抛错** | L2 兜底用 `asyncio.wait_for` 超时、捕获异常后仅打 debug 日志并返回 False，不抛错、不阻塞主流程。 |
| **Legacy 路径也走 L2 兜底** | 未启用四层时，legacy + 追问 未通过后同样尝试 L2 兜底（若存在 four_layer_trigger）；若未创建 trigger则跳过，行为明确。 |

### 8.3 代码与配置位置

- 会话窗口：`telegram_client.py` — `_record_session_reply`、`_is_in_session_window`；`_send_reply` 内成功发送后调用记录。
- L2 兜底：同文件 — `_should_reply_by_l2_fallback`；`_should_reply_to_group_message` 在追问未通过后调用。
- L2 仅跑接口：`four_layer_trigger.py` — `check_l2_only`；L2 内部去掉 ai_client 强依赖，保留规则置信度。
- 配置：`config/config.yaml` — `telegram.reply_logic.session_window`、`telegram.reply_logic.l2_fallback`。

## 九、后续可选优化（无强制下一阶段）

- **可观测**：对因「回复链 / 追问 / 会话窗口+L2 兜底」触发的回复打不同 log 前缀或指标，便于统计各规则贡献与漏回率。
- **高优先级群**：按群配置「高优先级」，在该群内对未触发 L1 的消息也跑 L2 兜底（可放宽 only_in_session_window），需群级配置与读取。
- **L2 真实语义**：若接入 claude-4.6-oups-high 等做「是否与订单/客服相关」的短文本分类，可将 `_get_ai_confidence` 改为调用该接口，提升 L2 与兜底准确率。
- **会话窗口持久化**：若需重启不丢，可将 `_session_reply_ts` 落盘或 Redis，键过期由存储负责。
