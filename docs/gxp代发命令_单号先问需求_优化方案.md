# gxp 代发命令：单号先问需求 — 逻辑与多场景优化方案

## 一、现状与目标

**现状**：用户必须「需求 + 单号」一起说（如「查代收 12345」）才会代发命令；若用户只发「12345678」，当前可能被订单查询等意图处理，不会进入「代发 gxp 命令」流程，也无法代发。

**目标**：
- 用户**只发订单号**时：先**确认需求**（查询代收 / 回调代收 / 查询提现 / 回调提现），再**用该单号组合命令**代发。
- 用户**先发单号、下条再说需求**时：用**上条缓存的单号** + 本条需求组合命令并代发。
- 保持「需求+单号同条」「无单号命令（汇率、余额等）」等现有行为不变。

---

## 二、整体流程概览

```
用户消息
    │
    ├─ 无单号类（汇率/余额/成功率/切换语言/添加成员/模拟回调+商户单号/utr）
    │       → 直接组命令并代发（与现有一致）
    │
    ├─ 有「需求关键词」且本条带单号
    │       → 直接组命令并代发（与现有一致）
    │
    ├─ 有「需求关键词」但本条无单号
    │       → 查上下文「待用单号」→ 有则用该单号组命令并代发，并清空待用单号
    │       → 无则回复「请先发单号，或说：查代收 单号」
    │
    └─ 仅单号（纯 6~24 位数字，或「单号：12345」且无需求词）
            → 将单号写入「待用单号」，回复询问需求
            → 用户下条说需求时，走「有需求无单号」分支，用待用单号组命令
```

## 三、多场景流程说明

### 场景 1：一句话带需求 + 单号（保持现状）

| 用户 | 系统 |
|------|------|
| 查代收 12345678 | 代发 `/cxds 12345678`，回复「已帮您发起请求，请查看上方机器人回复。」 |
| 回调代收 12345678 | 代发 `/htds 12345678`，同上 |
| 查提现 87654321 | 代发 `/cxdf 87654321`，同上 |

**逻辑**：本条既有需求关键词又有可提取单号 → 直接组命令并发送，不写、不读「待用单号」。

---

### 场景 2：仅发单号（先问需求）

| 用户 | 系统 |
|------|------|
| 12345678 | 1）识别为「仅单号」<br>2）将 `12345678` 存入该用户（及可选 chat）上下文：`gxp_pending_order_no`<br>3）回复：「收到单号 12345678。您需要我帮您：**查询代收** / **回调代收** / **查询提现** / **回调提现**？请直接说一项或发数字（如 1 查代收 2 回调代收 3 查提现 4 回调提现）。」 |

**逻辑**：消息为「仅单号」（见下节定义）→ 存单号 + 固定询问文案，不发命令。

---

### 场景 3：上条仅单号，本条只说需求（用缓存单号）

| 用户上条 | 用户本条 | 系统 |
|----------|----------|------|
| 12345678 | 查代收 / 1 / 查询代收 | 从上下文取 `gxp_pending_order_no` = 12345678，组 `/cxds 12345678` 并代发，回复「已帮您发起查询代收，请查看上方机器人回复。」，并清空 `gxp_pending_order_no` |
| 12345678 | 回调代收 / 2 / 回调 | 组 `/htds 12345678` 并代发，同上，清空待用单号 |
| 12345678 | 查提现 / 3 | 组 `/cxdf 12345678` 并代发，同上 |
| 12345678 | 回调提现 / 4 | 组 `/htdf 12345678` 并代发，同上 |

**逻辑**：本条有需求关键词（或数字 1~4 映射），但本条无单号 → 读 `gxp_pending_order_no`，有则用其组命令、发送、清空；无则提示「请先发单号」。

**数字简写**：1→查询代收，2→回调代收，3→查询提现，4→回调提现（可配置）。

---

### 场景 4：先说需求、本条无单号且上下文无待用单号

| 用户 | 系统 |
|------|------|
| 查代收（未带单号，且上下文无 gxp_pending_order_no） | 回复：「请先发单号，再说需求；或直接说：查代收 单号。」 |

**逻辑**：有需求无单号且无缓存 → 不代发，提示补单号。

---

### 场景 5：无单号类命令（不进入「单号流程」）

| 用户 | 系统 |
|------|------|
| 查汇率 / 余额 / 成功率 / 切换语言 / 添加成员 @xx 验证码 | 直接代发 /hl、/cxye、/cgl、/qhyy、/tjcy …，不读写待用单号 |

**逻辑**：这些命令不依赖单号，保持现有「识别即发」即可。

---

### 场景 6：仅单号但上下文已有待用单号（同号 / 异号）

| 情况 | 建议行为 |
|------|----------|
| 用户再发同一单号 12345678 | 可回复：「单号 12345678 已记录，您需要：查询代收 / 回调代收 / 查询提现 / 回调提现？」（不重复存，避免刷屏） |
| 用户发新单号 87654321 | 更新 `gxp_pending_order_no` = 87654321，再问：「收到单号 87654321，您需要：…」 |

**逻辑**：仅单号且已有 pending → 同号则只再问一次需求；异号则覆盖并再问。

---

### 场景 7：待用单号过期

| 建议 | 说明 |
|------|------|
| 为 `gxp_pending_order_no` 增加时间戳 `gxp_pending_time` | 若「当前时间 - gxp_pending_time」> 阈值（如 2 分钟或 5 分钟），视为过期 |
| 过期后用户说「查代收」且本条无单号 | 不读缓存，直接提示「请先发单号，或说：查代收 单号」 |
| 可选：回复中带过期提示 | 如「单号已过期，请重新发单号后再选需求。」 |

**逻辑**：避免长时间后误用旧单号，减少误操作。

## 四、「仅单号」的判定规则

满足以下**任一条**即视为「仅单号」消息（且本条**不包含**需求关键词）：

1. **纯数字**：整条消息去掉首尾空格后，匹配 `^\d{6,24}$`（6~24 位连续数字）。
2. **单号/订单号 + 数字**：如「单号 12345678」「订单号：12345678」，且句中**无**「查代收、回调代收、查提现、回调提现、查询、回调」等需求词。

**不视为仅单号**（保持原逻辑）：
- 消息中含需求关键词（如「查代收 12345」）→ 按「需求+单号」处理。
- 消息为纯数字但长度不在 6~24（如 123 或超长）→ 不进入「仅单号」分支，可由其他意图处理。
- 无单号类命令（汇率、余额等）→ 不进入单号流程。

## 五、上下文字段设计（user_context / context）

在现有 `user_context`（或 `context['data']`）中增加 gxp 相关状态，**按用户（及可选按 chat）隔离**：

| 字段 | 类型 | 含义 |
|------|------|------|
| `gxp_pending_order_no` | str | 待使用的单号（用户只发了单号、尚未选需求时写入） |
| `gxp_pending_time` | float (timestamp) | 写入待用单号的时间，用于过期判断 |
| 可选：`gxp_last_asked_order_no` | str | 上次询问需求时用的单号，用于「同号再问」时去重 |

**作用范围**：以 `user_id` 为键的上下文；若同一用户在多群使用，可用 `user_id + chat_id` 做键，避免跨群串单号（可选，按产品决定）。

**清理时机**：
- 成功用该单号代发命令后，清空 `gxp_pending_order_no` 与 `gxp_pending_time`。
- 过期后若用户再次「仅发单号」或「说需求但无单号」，可清空过期 pending。

## 六、需求关键词与数字简写

- **需求关键词**（与现有一致）：查代收、查询代收、回调代收、回调交易、查提现、提现查询、回调提现等；以及「1 / 2 / 3 / 4」可映射为：1→查询代收，2→回调代收，3→查询提现，4→回调提现。
- **仅单号** 分支不依赖需求词，只依赖「消息是否为仅单号」；**有需求无单号** 分支依赖需求词（含数字简写）+ 上下文有待用单号。

---

## 七、意图与技能协作（建议）

1. **意图层**
   - 增加对「仅单号」的识别并归入 gxp 流程：例如对**整条消息**匹配 pattern `^\s*\d{6,24}\s*$` 时，优先判为 `gxp_command`（或单独 `gxp_bare_order_no`），确保只发数字时进入 GxpCommandSkill 而非 order_query。
   - 或：在 GxpCommandSkill 内不依赖意图，由 skill_manager 在**识别意图前**先做一次「仅单号」检测，若命中则强制走 gxp_command 分支（需在 process_message 中增加一次预检）。

2. **GxpCommandSkill 内**
   - **入口分支顺序建议**：  
     (1) 无单号类命令（汇率/余额/成功率/语言/成员/模拟回调+商户单号/utr）→ 直接组命令发送。  
     (2) 本条为「仅单号」→ 存 pending + 回复询问需求。  
     (3) 本条有需求关键词（或 1~4）且（本条带单号 或 上下文有待用单号）→ 取单号（本条优先），组命令并发送，若有使用待用单号则清空。  
     (4) 本条有需求但无单号且无有效待用单号 → 提示「请先发单号」。  
     (5) 其他 → 不处理（返回 None）。

3. **与 order_query 的边界**
   - 纯 6~24 位数字消息：建议统一由 gxp 流程处理（先问需求），不再走 order_query 的「追问订单号」逻辑，避免同一句数字被理解为「订单号查询」和「gxp 待确认单号」两种含义。若希望 order_query 仍能处理「用户回复的订单号」，可在 order_query 中排除「当前上下文中已存在 gxp_pending_order_no 且本条为纯数字」的情况，将该条让给 gxp（或约定：纯数字在 gxp 开启时优先 gxp）。

## 八、询问与提示文案（可配置）

- **仅单号收到**：  
  「收到单号 {单号}。您需要我帮您：**查询代收** / **回调代收** / **查询提现** / **回调提现**？请直接说一项或发数字（1 查代收 2 回调代收 3 查提现 4 回调提现）。」
- **有需求无单号且无缓存**：  
  「请先发单号，再说需求；或直接说：查代收 单号。」
- **用缓存单号代发后**：  
  「已帮您发起请求，请查看上方机器人回复。」（可与现有一致）
- **待用单号过期**（可选）：  
  「单号已过期，请重新发单号后再选需求。」

以上文案可放入 `config.yaml` 或 `templates`，便于运营修改。

## 九、配置项建议

- `telegram.gxp_commands.enabled`：总开关（现有）。
- `telegram.gxp_commands.ask_when_bare_order_no`：是否开启「仅单号先问需求」（默认 true）。
- `telegram.gxp_commands.pending_order_expire_seconds`：待用单号过期时间（秒），如 120 或 300。
- `telegram.gxp_commands.number_shortcuts`：数字简写映射，如 `{"1": "查代收", "2": "回调代收", "3": "查提现", "4": "回调提现"}`（或对应命令 /cxds、/htds、/cxdf、/htdf）。
- 询问与提示文案：可放入 `templates.gxp_ask_intent`、`templates.gxp_need_order_no` 等。

## 十、实施顺序建议（无代码，仅步骤）

1. **上下文**：在 `user_context`（或等价处）增加 `gxp_pending_order_no`、`gxp_pending_time`（及可选 `gxp_last_asked_order_no`），按 user 或 user+chat 隔离。
2. **仅单号判定**：在意图或 GxpCommandSkill 入口实现「仅单号」识别（纯 6~24 位数字 或 「单号/订单号: 数字」且无需求词）。
3. **GxpCommandSkill 分支**：按第七章顺序实现 (1) 无单号类命令 (2) 仅单号→存+问 (3) 有需求→取单号（本条或 pending）组命令并发、清空 pending (4) 有需求无单号→提示。
4. **数字简写**：在「有需求无单号」分支中解析 1/2/3/4，映射到对应命令。
5. **过期**：在读取 pending 时判断 `gxp_pending_time`，超时则清空并视为无待用单号。
6. **同号/异号仅单号**：仅单号且已有 pending 时，同号只再问一次，异号覆盖并再问。
7. **意图优先**：确保「仅单号」消息进入 GxpCommandSkill（通过 pattern 或预检），与 order_query 边界清晰。
8. **配置与文案**：增加上述配置项与模板，便于关闭「先问需求」或调整过期时间与提示语。

按此方案可实现「用户只发单号 → 先问需求 → 再组合命令协助查询」的多场景流程，并与现有「需求+单号同条」「无单号命令」兼容。

## 十一、实施记录（已完成）

- **意图层**：在 `SkillManager` 中新增 `_is_bare_order_no(text)`，判定「仅单号」且无需求词；在 `_recognize_intent` 前若命中且 `gxp_commands.enabled` 且 `ask_when_bare_order_no` 为 true，则直接返回意图 `gxp_command`，保证纯单号进入 GxpCommandSkill。
- **上下文**：在 `user_context` 中使用 `gxp_pending_order_no`、`gxp_pending_time`、`gxp_pending_chat_id`（按用户+会话隔离）；读取 pending 时校验 chat_id 一致且未过期，过期则清空并返回过期提示。
- **GxpCommandSkill**：分支顺序为 (1) 无单号类命令 (2) 仅单号→存 pending + 询问 (3) 有需求→取单号（本条或 pending）组命令并发送，使用 pending 后清空 (4) 有需求无单号/过期→提示。数字简写 1→查代收、2→回调代收、3→查提现、4→回调提现（正则匹配整条为 "1"/"2"/"3"/"4"）。
- **配置**：`telegram.gxp_commands` 增加 `ask_when_bare_order_no`、`pending_order_expire_seconds`；`templates` 增加 `gxp_ask_intent`、`gxp_ask_same_no`、`gxp_need_order_no`、`gxp_expired`。
- **同号/异号**：仅单号且已有 pending 且与当前单号相同→回复 `gxp_ask_same_no`；不同则覆盖 pending 并回复 `gxp_ask_intent`。
