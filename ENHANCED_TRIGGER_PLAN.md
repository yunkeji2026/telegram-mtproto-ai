# 四层触发机制优化方案

## 目标
基于用户提出的四层触发机制，结合现有系统进行优化，实现：
1. **高精度触发**: 防误判率 < 3%
2. **快速响应**: L1规则触发瞬间判断
3. **智能过滤**: AI语义+上下文双重防闲聊
4. **高效冷却**: 90秒冷却避免过度回复

## 四层触发机制设计

### L1 规则触发 (70%激活，瞬间判断)
#### 触发条件（满足任意一条立即激活）：
1. **图片消息 + 任何文字**：用户发送图片并附带文字（哪怕只有"这个""收到吗"）
2. **高频关键词**：18个真实业务关键词
3. **订单号格式**：识别常见订单号格式
4. **@提及**：原有@ai_zkw或Camille

#### 技术实现：
- **优先级最高**：在L2之前判断，无需AI参与
- **即时响应**：直接触发回复流程，不等待语义分析
- **记录日志**：记录触发原因用于调试

### L2 语义触发 (25%激活，AI打分)
#### 触发条件：
1. **L1未触发**：只有L1不满足时才进入L2
2. **AI置信度 ≥ 0.88**：提高阈值确保准确性
3. **闲聊过滤**：Prompt强制执行闲聊检测

#### 技术实现：
- **AI模型增强**：claude-4.6-oups-high V3模型进行语义理解
- **置信度打分**：返回回复内容和置信度分数
- **阈值控制**：0.88高阈值，0.75-0.87人工兜底

### L3 上下文过滤 (防闲聊核心)
#### 过滤条件：
1. **同一用户持续闲聊**：最近10句内同一用户连续闲聊
2. **多人聊天跳过**：3人以上聊天且非求助消息
3. **冷却时间延长**：90秒冷却（原60秒）
4. **闲聊关键词检测**："哈哈""😂""好的""谢谢"等

#### 技术实现：
- **上下文分析增强**：在现有context_manager基础上增加闲聊检测
- **用户活跃度分析**：统计用户发言频率和内容类型
- **聊天场景判断**：区分业务咨询和社交闲聊

### L4 人工兜底 (安全机制)
#### 处理策略：
1. **0.75 ≤ 置信度 < 0.87**：只记录日志，不回复（等人工看）
2. **置信度 < 0.75**：彻底沉默（纯闲聊）
3. **异常情况记录**：所有未触发的情况都记录日志

#### 技术实现：
- **详细日志记录**：记录触发决策过程和各层结果
- **人工审核接口**：预留人工查看日志的机制
- **性能监控**：统计各层触发率和误判率

## 具体实施步骤

### 第一阶段：配置和基础框架 (30分钟)
1. **更新配置文件** (`config/config.yaml`)：
   - 添加四层触发配置参数
   - 定义高频关键词列表（18个）
   - 配置订单号识别正则表达式
   - 设置置信度阈值（0.88, 0.75）

2. **创建触发规则配置文件** (`config/trigger_rules.yaml`)：
   - 高频关键词分类（订单、支付、通道、状态等）
   - 订单号格式模式（0131..., 2102..., F2603..., T2603...）
   - 闲聊关键词列表
   - 图片触发规则

3. **修改系统架构**：
   - 在telegram_client.py中添加四层触发决策器
   - 在skill_manager.py中集成置信度处理
   - 在claude-4.6-oups-high_client.py中添加置信度返回

### 第二阶段：L1规则触发实现 (40分钟)
1. **图片消息检测增强**：
   - 修改telegram_client.py中的消息处理逻辑
   - 检测"图片+文字"组合，即使文字简短也触发
   - 记录触发原因（图片消息触发）

2. **高频关键词检测**：
   - 实现快速关键词匹配算法
   - 支持部分匹配和大小写不敏感
   - 18个关键词分类优化匹配效率

3. **订单号格式识别**：
   - 实现正则表达式匹配常见订单号格式
   - 支持多种格式：数字、字母数字混合
   - 长度验证和格式验证

4. **@提及检测保持**：
   - 保留现有@ai_zkw和Camille检测
   - 作为L1触发的一部分

### 第三阶段：L2语义触发实现 (60分钟)
1. **AI置信度打分**：
   - 修改claude-4.6-oups-high_client.py的generate_reply方法
   - 要求claude-4.6-oups-high API返回置信度分数
   - 实现置信度阈值过滤（≥0.88）

2. **闲聊过滤Prompt**：
   - 在系统提示词中添加闲聊检测要求
   - 强制AI识别闲聊内容并降低置信度
   - 示例："如果用户消息是闲聊，请返回低置信度"

3. **语义分析优化**：
   - 优化AI提示词以提高业务咨询识别准确率
   - 添加业务场景分类（订单、支付、通道等）
   - 提高对模糊查询的理解能力

### 第四阶段：L3上下文过滤实现 (45分钟)
1. **闲聊场景检测**：
   - 增强context_manager.py的上下文分析
   - 检测连续闲聊消息（"哈哈""😂""好的"等）
   - 分析对话主题是否偏离业务

2. **多人聊天判断**：
   - 统计最近10条消息中的不同用户数
   - 如果≥3人且非求助消息，跳过回复
   - 识别聊天场景（业务讨论 vs 社交闲聊）

3. **冷却机制优化**：
   - 修改冷却时间为90秒
   - 添加基于上下文的智能冷却
   - 连续业务咨询时可适当缩短冷却

4. **用户行为分析**：
   - 记录用户历史行为模式
   - 识别经常闲聊的用户
   - 针对不同用户调整触发策略

### 第五阶段：L4人工兜底实现 (25分钟)
1. **详细日志系统**：
   - 记录每层触发决策过程和结果
   - 记录置信度分数和过滤原因
   - 记录最终决策（回复/不回复）

2. **人工审核接口**：
   - 创建日志查看工具
   - 支持按时间、用户、聊天筛选
   - 提供决策统计和误判分析

3. **沉默机制**：
   - 置信度<0.75时彻底不回复
   - 0.75-0.87时记录日志等待人工
   - 确保不会误扰用户

4. **监控和统计**：
   - 实时统计各层触发率
   - 计算误判率和准确率
   - 监控系统性能指标

## 风险与缓解

### 技术风险：
1. **AI置信度不稳定** → 设置高阈值0.88，人工兜底
2. **规则匹配误判** → 定期更新关键词和规则
3. **上下文分析错误** → 多维度验证，保守过滤
4. **性能下降** → L1快速判断，L2异步处理

### 业务风险：
1. **漏掉重要消息** → L1覆盖主要场景，定期人工检查
2. **过度打扰用户** → 严格冷却和闲聊过滤
3. **用户感知不佳** → 保持回复质量和响应速度
4. **维护成本增加** → 自动化监控和告警

### 实施风险：
1. **兼容性问题** → 渐进式部署，可快速回滚
2. **配置错误** → 配置验证脚本，详细文档
3. **测试不充分** → 多场景测试，灰度发布
4. **资源不足** → 性能监控，容量规划
